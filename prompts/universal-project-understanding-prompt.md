# 通用项目分层理解文档生成提示词

你要帮助我接手一个陌生的软件项目。请把当前仓库作为主要事实来源，生成一套可持续维护的项目理解文档，而不是一篇试图解释全部内容的长文。

## 总体目标

建立以下文档结构：

```text
docs/project-understanding/
├── README.md                 # 主索引、阅读顺序、覆盖状态
├── 01-project-overview.md    # 项目定位、边界、成熟度
├── 02-architecture.md        # 总体架构、分层、模块关系
├── 03-runtime-flows.md       # 启动、主流程、关闭、异常流程
├── 04-data-models.md         # 数据模型、存储、消息和接口契约
├── 05-modules/               # 按模块拆分的专题文档
│   ├── README.md
│   └── <module>.md
├── 06-integrations.md        # 外部系统、第三方服务、依赖
├── 07-build-and-operations.md# 构建、运行、测试、部署和排障
├── 08-risks-and-gaps.md      # 风险、缺口、未知问题
└── 09-evidence-index.md      # 文件、符号、命令和结论证据索引
```

如果项目已有文档目录或既定文档规范，优先适配现有结构，不要强行创建重复目录。项目很大时继续拆分，按“一个文档解释一个稳定主题或责任边界”的原则控制篇幅。

## 分析规则

1. 先读项目规则、README、构建配置、依赖配置、入口文件和目录结构，再深入源码。
2. 不根据文件名或项目名称猜测。每个重要结论都必须能回溯到代码、配置、测试、脚本或文档。
3. 明确区分：
   - 已确认事实：附仓库相对路径、符号名，必要时附行号；
   - 设计意图：来自注释、文档或代码推断，标注“推断”；
   - 未知事项：无法确认的内容列入待确认清单。
4. 代码与文档冲突时，以实际代码行为为准，并在索引和专题文档中记录冲突。
5. 重点分析职责边界、输入输出、数据流、调用链、状态变化、生命周期、并发、事务、缓存、网络、副作用和错误处理。
6. 不修改业务代码，不执行破坏性命令。没有执行构建或测试时，不得声称构建或测试通过。

## 执行阶段

### 阶段一：勘察并创建文档地图

先输出“分析范围、项目入口、初步模块列表和文档拆分方案”，然后创建或规划主索引 `README.md`。主索引必须包含：

- 项目一句话定位；
- 当前成熟度结论；
- 推荐阅读顺序；
- 文档目录及每篇文档的职责；
- 每篇文档的状态：未开始 / 草稿 / 已核对 / 可能过时；
- 已覆盖模块和未覆盖区域；
- 事实源与代码冲突记录；
- 最后核对时间和核对范围。

### 阶段二：生成总体文档

至少完成以下专题：

- 项目概览：解决的问题、用户、边界、入口、成熟度、已实现和未实现能力；
- 总体架构：模块依赖、分层、公共边界、关键抽象和主要设计取舍；
- 运行流程：初始化、主业务流程、后台任务、关闭、重试、失败恢复和资源释放；
- 数据模型：实体、表、缓存、消息、API 请求响应、状态转换和数据生命周期；
- 构建运行：环境要求、依赖、配置、构建、测试、部署、日志和常见排障方式；
- 风险缺口：已确认风险、文档过时点、测试缺口、安全/性能/并发风险和未知问题。

### 阶段三：按模块生成专题文档

为每个重要模块单独建立文档。不要机械地为每个目录创建文件；应按实际责任边界拆分。每个模块文档固定包含：

1. 模块定位和业务职责；
2. 明确不负责的内容；
3. 入口文件、核心类/函数/接口；
4. 输入、输出和数据结构；
5. 依赖的上游和下游调用者；
6. 关键调用链和状态变化；
7. 生命周期、并发、事务、缓存、网络或文件副作用；
8. 错误处理、重试、幂等和权限规则；
9. 关键设计思路、取舍和修改影响范围；
10. 测试现状、验证命令和已知风险；
11. 相关文档链接和证据索引。

### 阶段四：建立交叉引用和证据索引

所有专题文档都必须从主索引可达。文档之间通过相对链接互相引用，不重复复制同一段事实。证据索引按“结论 → 文件 → 符号/配置 → 备注”记录关键依据，并标记证据是否已经直接阅读核对。

## 重点分析内容

追踪至少一条真实端到端链路，例如：请求/命令/UI 操作进入后，经过哪些模块、状态和数据转换，最终如何返回结果或产生副作用。对大型项目按子系统分别追踪，并在 `03-runtime-flows.md` 中拆成多个流程专题。

识别并说明：

- 公共 API、稳定边界和内部实现；
- 过宽的服务、仓储、控制器或聚合查询；
- 隐式全局状态、跨模块耦合和重复逻辑；
- 配置来源、默认值、敏感信息和外部数据出口；
- 数据库迁移、兼容性假设和版本约束；
- 线程、协程、任务队列、锁、事务和一致性边界；
- 超时、重试、降级、补偿、幂等和错误传播；
- 测试覆盖、固定样本、评估指标和可重复验证方式。

## 文档质量要求

- 主索引只负责导航和状态，不堆积所有实现细节。
- 每篇文档只解释一个稳定主题，过长时继续拆分，并在父文档保留索引。
- 结论先行，避免空泛形容词。
- 路径使用仓库相对路径，重要位置附符号名或行号。
- 用 Mermaid 绘制最有价值的模块图、时序图或数据流图；关系简单时使用文字。
- 同一事实只保留一个权威位置，其他文档使用链接引用。
- 发现文档与代码不一致时，在 `08-risks-and-gaps.md` 和主索引中登记。
- 对未阅读的目录、未执行的命令和未确认的推断明确标注，不得伪装成完成。

## 最终输出

最终交付应包括：

1. 文档树；
2. 主索引内容或主索引摘要；
3. 已生成专题文档及每篇覆盖范围；
4. 尚未分析的模块；
5. 仍需人工确认的问题；
6. 实际执行过的检查命令和结果；
7. 新维护者从“运行项目”到“安全修改核心功能”的推荐阅读路线。
