# 检验句转换指南

## 什么是检验句？

**检验句**是一种可验证的规则格式，Agent 可以在生成代码后真的去问自己这些问题，而不是抽象地"记住"某个原则。

## 声明式 vs 检验句

### ❌ 声明式规则

```markdown
- 保持代码简洁
- 遵循 DRY 原则
- 写清晰的注释
- 使用有意义的变量名
- 避免过度设计
```

**问题**：
- Agent 只能在生成前抽象地"记住"
- 生成后根本没有钩子去触发验证
- 无法量化"简洁"、"清晰"、"有意义"

### ✅ 检验句格式

```markdown
- 每一行改动都能追溯到用户的请求吗？
- 这个改动是否引入了用户没要求的抽象？
- 如果用户说"撤销最后一个功能"，是否能干净删除？
- 这段代码是第一次出现，还是复制粘贴的第三次？
- 删除这个文件会导致哪些测试失败？
```

**优势**：
- Agent 可以在生成后真的去问自己
- 提供明确的验证钩子
- 可量化、可操作

## 转换原则

### 1. 从"应该"到"是否"

❌ "应该保持简洁"
✅ "这个改动是否包含用户没要求的额外功能？"

❌ "应该遵循 DRY"
✅ "这段逻辑是否在三个地方重复？"

### 2. 从"避免"到"检查"

❌ "避免过度设计"
✅ "这个抽象是为一个用例设计的，还是为三个用例设计的？"

❌ "避免全局变量"
✅ "这个变量的作用域能否缩小到函数内部？"

### 3. 从"遵循"到"验证"

❌ "遵循命名约定"
✅ "这个函数名是否清楚说明了它的副作用？"

❌ "遵循测试覆盖率标准"
✅ "删除这个测试后，是否还能检测到相同的回归？"

## 常见规则的转换

### 代码质量

| 声明式 | 检验句 |
|-------|-------|
| 保持简洁 | 每一行改动都能追溯到用户的请求吗？ |
| 遵循 DRY | 这段逻辑是第几次出现？如果是第三次，能否抽象？ |
| 写清晰注释 | 删除这个注释后，代码是否仍然能被理解？ |
| 有意义的变量名 | 这个变量名是否说明了它存储什么类型的数据？ |
| 避免魔法数字 | 这个数字是否有业务含义？如果有，是否应该是常量？ |

### 架构

| 声明式 | 检验句 |
|-------|-------|
| 遵循单一职责 | 这个类/函数承担了几个职责？能否拆分？ |
| 依赖倒置 | 这个模块是依赖具体实现还是依赖接口？ |
| 开闭原则 | 添加新功能时是否需要修改现有代码？ |
| 低耦合 | 改动这个模块会影响几个其他模块？ |

### 测试

| 声明式 | 检验句 |
|-------|-------|
| 写充分的测试 | 删除这个测试后，是否还能检测到相同的回归？ |
| 测试独立性 | 这个测试是否依赖其他测试的执行顺序？ |
| 测试可读性 | 这个测试失败时，错误信息是否说明了失败原因？ |
| 避免测试实现细节 | 这个测试是在验证行为还是在验证实现？ |

### Git

| 声明式 | 检验句 |
|-------|-------|
| 提交信息要清晰 | 这个提交信息是否说明了为什么改，而不只是改了什么？ |
| 原子性提交 | 这个提交是否只做一件事？能否拆分？ |
| 及时提交 | 当前工作区有几个未提交的逻辑变更？ |

## 写检验句的技巧

### 技巧 1：使用具体的数字

❌ "避免过多的参数"
✅ "这个函数有几个参数？超过 3 个了吗？"

❌ "保持函数简短"
✅ "这个函数有几行？超过 50 行了吗？"

### 技巧 2：使用"如果...会..."

❌ "保持向后兼容"
✅ "如果用户使用旧 API，会发生什么？"

❌ "考虑边界情况"
✅ "如果输入是空数组，这段代码会崩溃吗？"

### 技巧 3：使用"是否能..."

❌ "代码应该可测试"
✅ "这段代码是否能在不启动数据库的情况下测试？"

❌ "保持可维护性"
✅ "6 个月后的你是否能理解这段代码在做什么？"

### 技巧 4：使用反向验证

❌ "遵循命名规范"
✅ "如果把这个变量名改成 `x`，代码是否还能被理解？"

❌ "写文档"
✅ "如果删除所有注释，新人是否还能理解这个模块？"

## Karpathy Skills 的检验句示例

来自 [forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)：

### Surgical Changes

✅ **原则**：每一行改动都应该能追溯到用户的请求

**检验句**：
- 每一行改动都能追溯到用户的请求吗？
- 这个改动是为了修复用户的问题，还是为了"顺便改善"？
- 如果用户说"撤销最后一个功能"，是否能干净删除？

### Avoid Premature Abstraction

✅ **原则**：等到有三个用例再抽象

**检验句**：
- 这个抽象是为几个用例设计的？
- 如果只有一个用例，为什么现在就要抽象？
- 这个接口是否比它要解决的问题更复杂？

### Avoid Premature Optimization

✅ **原则**：先让它工作，再让它快

**检验句**：
- 这个优化是为了解决已测量的性能瓶颈，还是猜测的？
- 优化前的代码是否已经不能满足性能要求？
- 优化后的代码是否更难理解？

### Use Examples

✅ **原则**：用真实例子代替抽象描述

**检验句**：
- 这个文档有具体例子吗？
- 这个例子是真实可运行的，还是伪代码？
- 如果删除所有文字描述，只看例子，是否仍然能理解？

## 实战示例

### 示例 1：过度设计

#### 用户请求
"修复登录按钮点击无响应的 bug"

#### ❌ Agent 做了什么（违反 Surgical Changes）
```diff
+ 修复登录按钮事件绑定 ✅
+ 添加 loading 状态显示 ❌
+ 添加按钮防抖 ❌
+ 添加错误重试逻辑 ❌
+ 添加日志记录 ❌
```

#### ✅ 检验句拦截
- "每一行改动都能追溯到用户的请求吗？" → **否**
- "loading、防抖、重试、日志是用户要求的吗？" → **否**
- "如果用户说'撤销最后一个功能'，是否能干净删除？" → **否**（因为混在一起了）

### 示例 2：过早抽象

#### 用户请求
"添加用户列表页面"

#### ❌ Agent 做了什么
```typescript
// 创建了通用的 DataTable 组件
// 创建了通用的 usePagination hook
// 创建了通用的 useFilter hook
// 只有一个用例（用户列表）
```

#### ✅ 检验句拦截
- "这个抽象是为几个用例设计的？" → **1 个**
- "如果只有一个用例，为什么现在就要抽象？" → **没有充分理由**
- "这个接口是否比它要解决的问题更复杂？" → **是**（通用接口很复杂）

## 在 AGENTS.md 中使用检验句

### 格式建议

```markdown
## 代码质量检验

完成代码后，自问以下问题：

### Surgical Changes（手术式改动）
- [ ] 每一行改动都能追溯到用户的请求吗？
- [ ] 这个改动是否引入了用户没要求的功能或优化？
- [ ] 如果用户说"撤销最后一个功能"，是否能干净删除？

### Avoid Premature Abstraction（避免过早抽象）
- [ ] 这个抽象是为几个用例设计的？（少于 3 个 = 过早）
- [ ] 如果只有一个用例，为什么现在就要抽象？
- [ ] 这个接口是否比它要解决的问题更复杂？

### Test Coverage（测试覆盖）
- [ ] 删除这个测试后，是否还能检测到相同的回归？
- [ ] 这个测试是在验证行为还是在验证实现？
- [ ] 这个测试失败时，错误信息是否说明了失败原因？
```

## 总结

### 核心区别

| 维度 | 声明式 | 检验句 |
|------|-------|-------|
| 时机 | 生成前"记住" | 生成后"检查" |
| 可验证性 | 抽象、主观 | 具体、可量化 |
| Agent 行为 | 靠记忆 | 靠验证钩子 |
| 效果 | 容易被忽略 | 提供明确检查点 |

### 转换公式

```
声明式规则 → 检验句
"应该 X" → "是否 Y？"
"遵循 X" → "能否通过 Y 验证？"
"避免 X" → "是否出现了 X？"
```

### 关键要素

1. **疑问句形式**："是否"、"能否"、"会不会"
2. **具体可量化**：数字、具体场景、反例
3. **可验证**：Agent 能真的去检查
4. **提供行动点**：明确下一步该怎么做
