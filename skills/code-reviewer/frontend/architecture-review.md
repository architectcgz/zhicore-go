# Frontend Architecture Review Prompt

## 用途

当需要对前端做架构级 code review，而不是只看语法、样式或单点 bug 时，使用本提示词。

目标是让 review 聚焦：

- 可维护性
- 可读性
- 健壮性
- 性能
- 测试性
- 工程规范
- 分层与 owner 收口

项目内如果保留了本地入口文件，先读本文件，再应用项目入口里的局部补充。

## 与主体严重度体系的关系

本文件是一段可以直接粘贴给模型使用的独立 prompt 模板，它自带 `P0 / P1 / P2` 分级和 10 分评分，只在“整段套用这个模板”时生效。

当 review 由 `code-reviewer` 主体驱动、需要统一输出口径时，按下面的映射回到主体的 `Blocker / Major / Minor / Nit` 与 gate verdict，不要在同一份 review 里混用两套词：

- `P0` → `Blocker`
- `P1` → `Major`
- `P2` → `Minor` 或 `Nit`
- “建议通过 / 修改后通过 / 不建议通过” → `pass / pass with minor issues / blocked`

## 提示词正文

```text
你是一名资深软件工程师，请帮我对这个前端做代码质量 review。

这不是语法检查。请从可维护性、可读性、健壮性、性能、测试性、安全性、类型边界和工程规范角度深入 review，并优先指出真实风险，不要把主观风格偏好包装成 blocker。

请先读代码、测试、路由、状态管理、API 包装和目录结构，再下结论。不要只看单个文件。

Review 时请重点关注：

1. 可读性
- 命名是否清晰
- 函数、组件、变量职责是否明确
- 代码是否容易理解
- 是否存在过长函数、过大文件、复杂条件判断

2. 可维护性
- 是否存在重复代码
- 是否有不合理的抽象
- 是否有过度封装或抽象不足
- 修改一个需求时影响范围是否可控

3. 代码结构
- 模块划分是否合理
- 函数/组件是否职责单一
- 是否存在业务逻辑和 UI、数据请求、状态管理混在一起的问题
- 是否存在隐式依赖或跨层调用
- route view、feature、widget、component、composable、store、api wrapper 的 owner 是否清晰
- 边界测试是在保护目标结构，还是只是在 allowlist 历史例外

4. 类型与边界
- TypeScript 类型是否准确
- 是否滥用 any、类型断言或非空断言
- 入参、返回值、接口数据是否有明确类型
- 是否正确处理 null、undefined、空数组、异常数据

5. 错误处理
- 异步请求是否有错误处理
- 用户可感知的错误是否有反馈
- 是否存在异常被吞掉的问题
- 是否区分业务错误和系统错误
- transport 层和 page/feature 层的错误 owner 是否清晰

6. 性能
- 是否存在不必要的重复计算
- 是否有不必要的重复渲染
- 是否存在大列表未优化、频繁创建对象/函数的问题
- 是否有明显的内存泄漏风险
- 是否把本应服务端 owner 的过滤、分页、聚合错误地放在前端

7. 测试性
- 代码是否方便写单元测试
- 业务逻辑是否和 UI 解耦
- 是否有关键分支缺少测试
- 是否有太多依赖外部环境导致难以测试
- 是否已有边界测试，但 allowlist 长期扩张导致测试失真

8. 安全性
- 是否存在 XSS、敏感信息暴露、权限绕过风险
- 是否信任了不可信输入
- 是否在前端硬编码 token、密钥、内部地址

9. 工程规范
- 是否符合 lint、format、团队编码规范
- import 顺序是否清晰
- 文件命名、目录命名是否一致
- 是否存在 dead code、console、TODO 长期遗留

输出要求：

## 总体评价
给出代码质量评分，满分 10 分，并说明主要原因。

## 关键问题
按 P0 / P1 / P2 分类：
- P0：可能导致线上 bug、安全问题、严重维护风险
- P1：会影响可维护性、扩展性、协作效率
- P2：代码风格、局部优化、低风险改进

每个问题请包含：
- 问题描述
- 具体位置
- 影响
- 修改建议
- 示例代码，如果有必要

## 优点
指出当前代码中做得好的地方。

## 重构建议
给出低风险、可渐进实施的改进路径。

## Review 结论
给出：建议通过 / 修改后通过 / 不建议通过。

附加要求：

- findings 优先，先说问题，再说优点
- 优先指出 correctness、结构耦合、owner 漂移、测试失真、错误边界不清这类真实风险
- 对每个判断尽量给文件和行号
- 明确区分“代码中已验证的事实”和“根据上下文推断的结论”
- 如果发现项目正在做架构迁移，重点判断迁移是否已经收口，还是长期双轨并存
- 如果边界测试依赖大量 allowlist，请把它当成结构性风险，而不是当成已有 guardrail 就放过
- 如果一个过宽 feature / service / composable 已经成为持续堆积点，把它列为 blocker，不要只写成 follow-up
```
