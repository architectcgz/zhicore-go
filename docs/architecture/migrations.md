# 数据库 Migration 规则

本文件定义 `zhicore-go` 的数据库 schema 演进规则。服务内分层和 repository 落点见 `docs/architecture/go-service-design.md`，运行时启动规则见 `docs/architecture/runtime-operations.md`。

## 工具选择

正式数据库 migration 使用 `golang-migrate`。

规则：

- 每个服务的 schema 事实源是 `services/<service>/migrations/` 下的 SQL migration 文件。
- `golang-migrate` 只作为显式 migration 执行工具，不嵌入普通服务启动路径。
- 服务启动、`runtime.Build`、HTTP handler、worker、consumer、定时任务和容器 entrypoint 不得自动执行 schema migration。
- GORM 可以用于 repository / infrastructure 数据访问实现，但 GORM model、tag 或 `AutoMigrate` 不是正式 schema 事实源。
- GORM `AutoMigrate` 只允许出现在测试 fixture、临时本地实验或一次性验证代码中，不得进入正式运行路径。

## 文件放置

每个服务只迁移自己拥有的数据表：

```text
services/<service>/migrations/
```

文件必须成对出现：

```text
<version>_<description>.up.sql
<version>_<description>.down.sql
```

命名规则：

- `<version>` 使用 UTC 时间戳，格式 `YYYYMMDDHHMMSS`。
- `<description>` 使用小写 snake_case，描述本次 schema 变化，例如 `create_users`、`add_posts_public_id`。
- 一个 migration pair 只处理一个可审查的逻辑变化；不要把多个服务、多个无关表族或大量重命名混在一个 pair 中。
- 文件名一旦进入共享分支，不再重命名；需要修正时新增 migration。

示例：

```text
services/zhicore-user/migrations/
├── 20260622103000_create_users.up.sql
└── 20260622103000_create_users.down.sql
```

## 创建命令

推荐使用 `golang-migrate` CLI 创建文件：

```bash
migrate create -ext sql -dir services/<service>/migrations -format 20060102150405 <description>
```

如果本机 `migrate create` 不支持 `-format`，可以使用默认 timestamp 版本，但同一服务内必须保持版本单调递增。

## 执行命令

每个服务使用自己的数据库 URL：

```bash
migrate -path services/<service>/migrations -database "$<SERVICE>_DATABASE_URL" up
migrate -path services/<service>/migrations -database "$<SERVICE>_DATABASE_URL" down 1
```

示例：

```bash
migrate -path services/zhicore-user/migrations -database "$ZHICORE_USER_DATABASE_URL" up
```

规则：

- 不要用一个服务的 migration 操作另一个服务的数据库。
- 不要把数据库密码、token 或完整生产连接串提交到仓库。
- 执行生产 migration 前必须先在本地或测试数据库运行 `up`，并至少验证一次最近一条 `down 1`，除非该 migration 明确不可逆。
- 处理 `dirty` 版本时，不得直接盲目 `force`。必须先确认数据库实际 schema、失败语句和目标版本，再记录修复步骤。

## SQL 编写规则

默认使用 PostgreSQL SQL。

规则：

- 内部主键默认使用 `BIGINT GENERATED BY DEFAULT AS IDENTITY`。
- 数据库对象命名使用 snake_case。
- 对外公开 ID、业务编号和唯一约束按 `docs/architecture/id-strategy.md`。
- 表、索引、约束和枚举必须属于当前服务的数据归属；跨服务引用只保存 opaque reference，例如 `author_id`、`post_id`、`file_id`。
- 不创建跨服务外键；服务边界由 contract、事件、read model 或 facade 维护。
- 不把 Java 全量初始化 SQL 原样复制到 Go 服务；必须按服务归属拆分。
- 不在 migration 中依赖运行时业务代码、GORM model 或 Go struct tag。

## 事务规则

- 对 PostgreSQL 可事务化 DDL，migration 文件应显式使用 `BEGIN;` / `COMMIT;` 包裹。
- `CREATE INDEX CONCURRENTLY`、长时间回填、分批数据修复等不能放在普通事务里的操作，必须拆成独立 migration，并在文件顶部写注释说明原因、风险和验证方式。
- 一个 migration 中如果包含 schema 和大量数据修复，优先拆分；无法拆分时必须说明执行时间、锁影响和回滚策略。

示例：

```sql
BEGIN;

CREATE TABLE users (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  public_id VARCHAR(32) NOT NULL UNIQUE,
  username VARCHAR(64) NOT NULL UNIQUE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);

COMMIT;
```

## Down Migration

每个 `.up.sql` 必须有对应 `.down.sql`。

规则：

- `down` 必须撤销本次 `up` 引入的 schema 变化。
- 删除列、删除表、数据转换等可能丢数据的 `down` 必须在文件顶部写明数据丢失风险。
- 如果某个 migration 真实不可逆，`down` 文件必须保留，并只包含说明性注释和显式失败语句，不能留空。

不可逆示例：

```sql
-- Irreversible migration.
-- This migration squashes historical rows and cannot restore original values.
SELECT 1 / 0;
```

## Seed 和参考数据

Migration 默认只管理 schema。

允许放入 migration 的数据：

- 必须随 schema 存在的枚举/参考数据，例如系统角色、固定状态映射。
- 为约束、默认值或兼容查询必须存在的最小数据。

规则：

- seed / reference data 必须幂等，优先使用 `INSERT ... ON CONFLICT ...`。
- 大量业务数据导入、历史补偿、缓存重建和一次性修复不放普通 schema migration；应作为受控 repair / backfill 任务处理，并在 `docs/todos/debt/` 或对应运维记录中登记。

## Review 清单

提交 migration 前至少检查：

- 表归属是否符合 `docs/architecture/service-boundaries.md`。
- 文件名版本是否单调递增，`up` / `down` 是否成对。
- 是否存在跨服务外键、跨服务写模型或共享表。
- 是否有唯一约束、必要索引和外部公开 ID 约束。
- 是否显式处理事务、锁风险和不可逆风险。
- 是否没有运行时自动迁移或 GORM `AutoMigrate`。
- 是否补充了 repository / service 行为测试或 runtime 验证计划。

## 与 GORM 的边界

如果某个服务使用 GORM：

- GORM 只放在 infrastructure repository 实现中。
- GORM model / PO 不进入 `domain`、`api/http` 或 `libs/contracts`。
- GORM tag 可以帮助 repository 映射字段，但不能替代 migration 审查。
- domain model 与 GORM model 之间必须有显式 mapper。
- 所有数据库操作必须传递 `context.Context`，使用 `db.WithContext(ctx)`。
- 事务边界由 application 或 repository 明确拥有，不通过隐藏的全局 `*gorm.DB` 隐式传播。
