# Comment 数据和事件设计

PostgreSQL 是评论和统计的真相源。Redis 只保存可重建缓存。Comment 通过 producer outbox 发布跨服务集成事件。

## 数据归属

Comment 拥有：

- `comments`
- `comment_stats`
- `comment_likes`
- `comment_post_counters`
- `comment_counter_deltas`
- `comment_hot_rank`
- Comment 服务自己的 `outbox_events`

评论图片只保存 Upload / File Service 返回的图片文件 ID；语音只保存语音文件 ID。可展示图片 URL 和可播放语音 URL 由查询时解析或缓存派生，不作为 Comment 持久化事实。文件元数据、对象存储路径、签名 URL、CDN 规则和文件删除事实归 Upload / File Service。

`comments.post_id` 和相关表的 `post_id` 保存 Content 对外 `postId` 字符串，例如 `p1K8x9Q2`，不保存 Content 私有数据库主键。Comment 只把 `postId` 当作 opaque reference、查询分区和楼层计数范围。

`outbox_events` 是 Comment 服务私有表。仓库允许不同服务数据库中存在同名 outbox 表；表归所在服务数据库拥有，不使用跨服务共享 outbox 表。

## `comments` schema 草案

```sql
CREATE TABLE comments (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  post_id VARCHAR(32) NOT NULL,
  floor BIGINT NOT NULL,
  author_id BIGINT NOT NULL,
  root_id BIGINT NULL,
  parent_id BIGINT NULL,
  content TEXT NULL,
  image_file_ids TEXT[] NULL,
  voice_file_id TEXT NULL,
  voice_duration INTEGER NULL,
  status VARCHAR(32) NOT NULL,
  deleted_by BIGINT NULL,
  delete_reason VARCHAR(500) NULL,
  deleted_at TIMESTAMPTZ NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT uq_comments_post_floor UNIQUE (post_id, floor),
  CONSTRAINT ck_comments_floor_positive CHECK (floor > 0),
  CONSTRAINT ck_comments_status CHECK (status IN ('NORMAL', 'DELETED')),
  CONSTRAINT ck_comments_root_parent_pair CHECK (
    (root_id IS NULL AND parent_id IS NULL)
    OR (root_id IS NOT NULL AND parent_id IS NOT NULL)
  ),
  CONSTRAINT ck_comments_no_self_parent CHECK (
    (root_id IS NULL OR root_id <> id)
    AND (parent_id IS NULL OR parent_id <> id)
  ),
  CONSTRAINT ck_comments_body_present CHECK (
    (content IS NOT NULL AND btrim(content) <> '')
    OR COALESCE(cardinality(image_file_ids), 0) > 0
    OR (voice_file_id IS NOT NULL AND btrim(voice_file_id) <> '')
  ),
  CONSTRAINT ck_comments_image_limit CHECK (
    COALESCE(cardinality(image_file_ids), 0) <= 9
  ),
  CONSTRAINT ck_comments_voice_image_exclusive CHECK (
    voice_file_id IS NULL OR COALESCE(cardinality(image_file_ids), 0) = 0
  ),
  CONSTRAINT ck_comments_voice_duration_required CHECK (
    (voice_file_id IS NULL AND voice_duration IS NULL)
    OR (voice_file_id IS NOT NULL AND voice_duration > 0)
  )
);
```

推荐索引：

```sql
CREATE INDEX ix_comments_post_top_time
  ON comments (post_id, floor DESC)
  WHERE root_id IS NULL AND parent_id IS NULL AND status = 'NORMAL';

CREATE INDEX ix_comments_root_replies_time
  ON comments (root_id, floor ASC)
  WHERE status = 'NORMAL';
```

`root_id` 和 `parent_id` 可以加服务内自引用外键，也可以先只用索引和 repository 校验保护，取决于 migration 阶段对软删除、导入顺序和批量修复的要求。无论是否加外键，application / repository 都必须在创建回复的事务内校验根评论和直接父评论属于同一 `post_id`、父评论未删除、根评论确实是顶级评论，并且父评论属于该根评论树。

## `comment_post_counters` schema 草案

```sql
CREATE TABLE comment_post_counters (
  post_id VARCHAR(32) PRIMARY KEY,
  next_floor BIGINT NOT NULL,
  CONSTRAINT ck_comment_post_counters_next_floor_positive CHECK (next_floor > 0)
);
```

推荐分配语义：

```sql
UPDATE comment_post_counters
SET next_floor = next_floor + 1
WHERE post_id = ?
RETURNING next_floor - 1 AS floor;
```

首次评论可用 UPSERT 初始化 counter。`floor` 一旦分配，即使评论删除也不复用。

## 统计、点赞和计数 delta 表

`comment_stats`：

- `comment_id` 唯一，引用 `comments.id`。
- `like_count`、`reply_count` 必须非负。
- `reply_count` 在创建 / 删除回复事务内同步增减；`like_count` 由 `comment_counter_deltas` 后台批量聚合更新。
- `RebuildFromQuery` 可以从 `comment_likes` 和 `comments` 重建统计。

```sql
CREATE TABLE comment_stats (
  comment_id BIGINT PRIMARY KEY,
  like_count BIGINT NOT NULL DEFAULT 0,
  reply_count BIGINT NOT NULL DEFAULT 0,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT ck_comment_stats_like_count_non_negative CHECK (like_count >= 0),
  CONSTRAINT ck_comment_stats_reply_count_non_negative CHECK (reply_count >= 0)
);
```

`comment_likes`：

- `(comment_id, user_id)` 联合唯一。
- 点赞插入和取消删除必须与 `comment_counter_deltas` 和对应 outbox 事件在同一事务内完成。
- 重复点赞或取消未点赞按幂等成功处理，不写 delta、不写事件。

```sql
CREATE TABLE comment_likes (
  comment_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (comment_id, user_id)
);
```

`comment_counter_deltas` 第一阶段只记录点赞计数变化：

```sql
CREATE TABLE comment_counter_deltas (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  comment_id BIGINT NOT NULL,
  post_id VARCHAR(32) NOT NULL,
  floor BIGINT NOT NULL,
  counter_type VARCHAR(32) NOT NULL,
  delta_value INTEGER NOT NULL,
  source_event_id VARCHAR(64) NULL,
  status VARCHAR(32) NOT NULL DEFAULT 'PENDING',
  claimed_by VARCHAR(128) NULL,
  claimed_at TIMESTAMPTZ NULL,
  processed_at TIMESTAMPTZ NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT ck_comment_counter_deltas_type CHECK (counter_type IN ('LIKE')),
  CONSTRAINT ck_comment_counter_deltas_value CHECK (delta_value IN (-1, 1)),
  CONSTRAINT ck_comment_counter_deltas_status CHECK (status IN ('PENDING', 'PROCESSING', 'DONE', 'FAILED'))
);

CREATE INDEX ix_comment_counter_deltas_pending
  ON comment_counter_deltas (status, id)
  WHERE status IN ('PENDING', 'FAILED');
```

worker 按批次 claim `PENDING` delta，按 `comment_id` 聚合后更新 `comment_stats.like_count`。如果目标评论是顶级评论，同步更新 `comment_hot_rank.like_count`。delta 表是可重试台账，不是长期审计表；统计漂移时以 `comment_likes` 作为事实源重建。

如果未来回复创建 QPS 也成为瓶颈，可以把 `reply_count` 改为同一 delta 机制；第一阶段不提前冗余，回复数仍在创建 / 删除回复事务内同步维护。

## HOT 排序读模型

`comment_hot_rank` 只保存顶级评论的 HOT 排序锚点，避免列表查询对 `comments + comment_stats` 做大范围排序 join：

```sql
CREATE TABLE comment_hot_rank (
  comment_id BIGINT PRIMARY KEY,
  post_id VARCHAR(32) NOT NULL,
  floor BIGINT NOT NULL,
  like_count BIGINT NOT NULL DEFAULT 0,
  visible BOOLEAN NOT NULL DEFAULT TRUE,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT uq_comment_hot_rank_post_floor UNIQUE (post_id, floor),
  CONSTRAINT ck_comment_hot_rank_like_count_non_negative CHECK (like_count >= 0)
);

CREATE INDEX ix_comment_hot_rank_post_hot
  ON comment_hot_rank (post_id, like_count DESC, floor ASC)
  WHERE visible = TRUE;
```

根评论创建时在同一事务内插入 `comment_hot_rank`，初始 `like_count = 0`。根评论删除时标记 `visible = FALSE` 或删除 rank 行；推荐先标记不可见，方便排障和重建。回复不进入 `comment_hot_rank`。

HOT 查询流程：

```text
comment_hot_rank 按 post_id + like_count DESC + floor ASC 取一页 comment_id
-> comments 批量加载评论正文、状态和媒体引用
-> comment_stats 批量加载 reply_count / like_count
-> User 批量补作者摘要
```

这样排序发生在窄读模型索引上，后续补数据只对一页结果做批量读取。

## Outbox 表

推荐 schema 至少包含：

- `event_id`
- `event_type`
- `payload_version`
- `producer`
- `aggregate_type`
- `aggregate_id`
- `occurred_at`
- `payload`
- `status`
- `retry_count`
- `next_retry_at`
- `claimed_by`
- `claimed_at`
- `created_at`
- `sent_at`
- `error_message`

`event_id` 同时就是 RabbitMQ envelope 的 `eventId`，由 application / outbox writer 在业务事务内生成并落库，dispatcher 不生成新的业务事件 ID。

发布到 RabbitMQ 时有两种等价实现：

1. outbox 表直接保存完整 envelope JSON，dispatcher 原样发布。
2. outbox 表分列保存 envelope metadata 和 payload，dispatcher 在发布前组装统一 envelope。

对外事件 envelope 的语义由 `libs/contracts/events/comment` 拥有；私有 outbox 表只是 Comment 的可靠投递台账。

## 缓存 key

这里的 `commentId` 是 Comment 内部 `comments.id`，只用于服务内缓存定位，不作为对外 HTTP 资源 ID。

| 缓存 | Key 模式 |
| --- | --- |
| 评论详情 | `comment:{commentId}:detail:v2` |
| 评论点赞数 | `comment:{commentId}:stats:likes` |
| 评论回复数 | `comment:{commentId}:stats:replies` |
| 用户是否点赞评论 | `comment:{commentId}:liked:{userId}` |
| 文章评论数 | `comment:post:{postId}:count` |
| 文章顶级评论时间排序 | `comment:post:{postId}:top:time` |
| 文章顶级评论热度排序 | `comment:post:{postId}:top:hot` |
| 根评论回复列表 | `comment:{rootId}:replies` |
| 首页评论快照 | `comment:post:{postId}:homepage:{sort}:{size}:{replyLimit}` |
| Ranking 热门候选 | `comment:ranking:posts:hot:candidates` |
| Ranking 热门候选元信息 | `comment:ranking:posts:hot:candidates:meta` |

## 热门候选同步

- `SyncRankingHotCandidates` 由 Comment 后台 job 定时调用 Ranking contract 获取热门文章候选。
- 多实例部署时使用分布式锁避免重复拉取和重复写 Redis。
- 成功拉取后用 `replaceCandidates` 原子替换热门候选和元信息缓存。
- 拉取失败时保留旧值并记录告警，不清空候选集。
- Ranking 仍拥有榜单分数和候选生成规则；Comment 只拥有本地缓存副本和首页评论查询时的使用策略。

## 集成事件

集成事件是跨服务 contract，必须放到 `libs/contracts/events/comment` 后才能被其他服务稳定依赖。本文只固定语义方向，不替代字段级 contract。

集成事件中的 `commentId` 是 Comment 内部 `comments.id` 的跨服务 opaque reference，用于 consumer 幂等、通知、统计和热度 ledger；对外 HTTP 仍以 `(postId, floor)` 定位评论。

| 事件 | 触发用例 | 主要 payload | 当前 / 目标 consumer | outbox 要求 |
| --- | --- | --- | --- | --- |
| `comment.created` | `CreateComment` / `CreateReply` | `commentId`、`postId`、`postAuthorId`、`floor`、`authorId`、`rootId/rootFloor/rootAuthorId`、`parentId/parentFloor/parentAuthorId`、`createdAt` | Content、Notification、Ranking | 关键事件，使用 producer outbox |
| `comment.deleted` | `DeleteComment`、`AdminDeleteComment` | `commentId`、`postId`、`floor`、`rootId`、`rootFloor`、`authorId`、`deletedBy`、`deleteReason`、`deletedAt`、`isRoot`、`affectedCount` | Content、Ranking、Notification | 关键事件，使用 producer outbox |
| `comment.liked` | `LikeComment` | `commentId`、`postId`、`floor`、`commentAuthorId`、`likedBy`、`occurredAt` | Notification、Ranking | 关键事件，使用 producer outbox |
| `comment.unliked` | `UnlikeComment` | `commentId`、`postId`、`floor`、`commentAuthorId`、`unlikedBy`、`occurredAt` | Ranking | 关键事件，使用 producer outbox |

`comment.created` payload 草案：

```json
{
  "eventId": "evt_01HY...",
  "eventType": "comment.created",
  "payloadVersion": 1,
  "producer": "zhicore-comment",
  "occurredAt": "2026-06-23T10:30:00Z",
  "aggregateType": "comment",
  "aggregateId": "12345",
  "payload": {
    "commentId": 12345,
    "postId": "p1K8x9Q2",
    "postAuthorId": 9001,
    "floor": 26,
    "rootId": null,
    "rootFloor": null,
    "rootAuthorId": null,
    "parentId": null,
    "parentFloor": null,
    "parentAuthorId": null,
    "authorId": 1001,
    "hasImages": false,
    "hasVoice": false
  }
}
```

`comment.created` 首版 payload 必须带足 Notification 创建通知所需的稳定事实：文章作者 `postAuthorId`，以及回复场景下的 `rootFloor/rootAuthorId` 和 `parentFloor/parentAuthorId`。Notification 不应为了构造一条评论通知再同步回查 Comment 或 Content；回查只能作为修复或降级路径。

## Consumer 归属

- Content：消费 `comment.created` / `comment.deleted` 更新 `post_stats.comment_count`。
- Notification：消费 `comment.created` / `comment.liked` 创建站内通知，是否消费 `comment.deleted` 隐藏通知按产品规则决定。
- Ranking：消费 `comment.created` / `comment.deleted` / `comment.liked` / `comment.unliked` 更新热度 ledger。
- Search：默认不消费 Comment 事件；如果未来搜索评论，再单独定义 Search read model。
