# Comment 模块决策日志

本文件记录 `zhicore-comment` 模块设计压测过程中已经确认的设计问题、结论、原因和后续依赖。


| # | 决策项 | Question | Decision | Rationale | Follow-up |
| --- | --- | --- | --- | --- | --- |
| 1 | User 外部公开标识 | User 服务是否应该拥有面向外部 HTTP / 前端的公开用户标识，而不是只暴露内部 `UserID`？ | User 应该拥有独立的外部 `publicId`；前端可见的用户标识默认使用 `publicId`，内部 `UserID` 不直接作为外部资源 ID 暴露。 | `publicId` 可以把外部 contract 和数据库内部主键解耦，避免暴露自增序列和后续主键策略变化；同时仍保留内部 `UserID` 作为 User 服务内的稳定聚合标识。 | 需要继续决定 Comment 持久化、Gateway 身份 header、User typed client 和 Comment 事件分别使用内部 `UserID`、`publicId`，还是同时携带两者。 |
| 2 | Comment 用户引用持久化 | Comment 持久化层应该保存 User 内部 `UserID`，还是保存 User 外部 `publicId`？ | Comment 内部持久化使用 User 内部 `UserID`，例如 `comments.author_id BIGINT` 和 `comment_likes.user_id BIGINT`；对外 HTTP 响应使用 User `publicId`，由 User 查询 contract 返回作者摘要时补齐。 | 作者权限、点赞唯一键、关系判断、事件幂等和内部查询都需要稳定且高效的服务间用户引用；`publicId` 主要服务前端可见 contract 和 URL，不应替代内部关系表的归属键。 | 需要继续决定 Gateway 身份 header 是否注入内部 `UserID`，以及 Comment 事件是否同时携带 `userPublicId` 快照。 |
| 3 | Gateway 用户身份 Header | Gateway 注入给 Comment 的 `X-User-Id` 应该是内部 `UserID` 还是外部 `publicId`？ | `X-User-Id` 使用 User 内部 `UserID`；Comment handler 将其解析为 application `Actor.UserID`。如未来前端或审计需要公开标识，可另增 `X-User-Public-Id`，但 Comment 写路径不依赖它。 | 登录态写操作、作者权限、点赞幂等和关系判断都需要内部主体标识。若只传 `publicId`，Comment 每次写请求都必须反查 User，增加延迟、失败点和权限链路复杂度。 | 需要继续决定 Comment 集成事件是否只携带内部 `UserID`，还是同时携带 `userPublicId` 展示快照。 |
| 4 | Comment 事件中的用户标识 | Comment 集成事件是否需要携带 User 的 `publicId`？ | Comment 集成事件默认只携带 User 内部 `UserID` 作为 opaque reference，例如 `authorId`、`commentAuthorId`、`likedBy`、`deletedBy`；不携带 User `publicId`。未来只有出现明确 consumer 需要事件内直接获得公开用户标识时，才以可选字段追加 `authorPublicId` 等字段。 | User `publicId` 是外部展示标识，不是 Comment 事件幂等、权限或内部关联所需字段。把它放进写路径事件会让 Comment 在创建、点赞等 mutation 中额外依赖 User 摘要查询，扩大失败面。需要展示资料的查询路径或 consumer 读模型可以通过 User contract 批量获取。 | 需要同步收紧 Comment 事件文档中的用户字段语义，明确 `authorId` 等字段是 User 内部 opaque reference，不是前端公开 ID。 |
| 5 | Comment HTTP 作者标识字段 | Comment HTTP 响应中的作者摘要字段应该继续叫 `userId`，还是改成明确的公开标识字段？ | Comment HTTP 响应中的作者摘要使用 `publicId` 表示 User 外部公开标识，不再用 `userId` 表示 public ID。 | 服务内 `userId` 已明确指 User 内部 `UserID`。如果 HTTP `AuthorSummary.userId` 实际表示 `publicId`，会造成 contract、前端路由和服务间调用长期歧义。 | 需要同步更新 `services/zhicore-comment/api/http/README.md`、endpoint 示例和 User typed client contract 草案。 |
| 6 | User Simple Contract 标识字段 | User typed client 给 Comment 的用户摘要 DTO 是否应该同时返回内部 `userId` 和外部 `publicId`？ | `BatchGetUserSimple(userIds)` 入参使用内部 `UserID` 列表；返回 DTO 同时包含内部 `userId` 和外部 `publicId`，并携带 `displayName`、`avatarFileId`、`avatarUrl`、`profileVersion` 等摘要字段。 | Comment 查询按内部 `author_id` 批量查 User，返回结果需要内部 `userId` 稳定映射回评论作者；HTTP response 需要 `publicId` 作为前端可见标识。两者职责不同，不能只保留其中一个。 | 需要在 `libs/contracts/clients/user/` 草案中固定该 DTO，并让 Comment HTTP mapper 只把 `publicId` 写入对外作者摘要。 |
| 7 | Comment 外部定位 | Comment 是否需要自己的 `commentPublicId`？ | 不需要。Comment 对外继续用 `(postId, floor)` 定位评论；内部 `commentId` 只用于服务内表关联、缓存和事件 opaque reference。 | `floor` 在同一文章内有产品语义，且已经承担外部定位。新增 `commentPublicId` 会形成两套对外定位方式，增加前端、管理端、事件和缓存的一致性成本。 | 如果未来出现跨文章全局评论链接、独立评论聚合页、外部 webhook 隐藏 `postId`、评论迁移后仍需稳定原链接等真实需求，再作为独立 API 演进重新评估。 |
| 8 | 文章评论总数语义 | `post_stats.comment_count` 应该统计全部评论含回复，还是只统计顶级评论？ | `post_stats.comment_count` 统计文章下全部未删除评论数，包含顶级评论和回复；回复也算评论。 | 旧实现创建根评论和回复都会增加文章评论数；Go 设计中的 `comment.deleted.affectedCount` 和 Ranking 账本也要求根评论批量删除时按实际影响的评论总数回滚。 | 需要同步明确 `comment.deleted.affectedCount = 根评论 + 本次实际被软删除的未删除回复数`；Content / Ranking 消费删除事件时按 `-affectedCount` 更新统计。 |
| 9 | 删除根评论的回复处理 | 根评论被删除时，回复应该一起软删除，还是保留为孤儿回复 / 历史回复？ | 删除根评论时，软删除根评论及其全部回复。 | 回复列表入口依赖根评论 `{floor}/replies`。如果根评论删除但回复保留，会产生没有稳定上下文的回复树，并增加 `replyCount`、文章评论总数、通知跳转和管理端状态的复杂度。整棵评论子树一起软删除可以让普通读取、统计回滚和 Ranking 账本语义保持一致。 | 删除事件的 `affectedCount` 必须等于本次实际从未删除状态变为删除状态的评论数量，避免重复删除导致计数重复回滚。 |
| 10 | 根评论删除事件粒度 | 删除根评论并级联软删除回复时，是否要为每条被删除的回复分别发 `comment.deleted` 事件？ | 不逐条发送。根评论删除只发送一条 `comment.deleted`，携带 `isRoot=true` 和 `affectedCount`。 | Content 和 Ranking 只需要统计 delta，`affectedCount` 足够表达本次删除影响；逐条事件会放大 outbox 和 consumer 幂等成本。Notification 可以根据根评论删除事件处理跳转入口失效，不需要每条回复一条事件。 | 如果未来 Search 或外部索引需要逐条 tombstone，再新增批量删除明细事件或查询修复机制，不扩展当前主删除事件为逐条流。 |
| 11 | 删除任意评论节点的子树语义 | 删除回复或任意中间评论节点时，是否保留其子回复？ | 删除任意评论节点时，都软删除该节点及其整棵子树。 | `parentFloor` 允许指向任意回复，评论树可能超过两层。如果只删除中间节点但保留子回复，会产生断裂对话，并增加前端展示、通知跳转、`replyCount` 和 `affectedCount` 语义复杂度。统一为“删除目标节点及 descendants”更容易维护一致性。 | `comment.deleted.affectedCount` 对所有删除场景都表示本次实际从未删除状态变为删除状态的节点数。 |
| 12 | 根评论回复数语义 | 根评论的 `replyCount` 应该统计直接回复数，还是整棵子树的回复总数？ | `replyCount` 统计根评论下所有未删除后代回复总数，不只统计直接子回复。 | 前端展示的“回复数”通常表示根评论下所有回复总量。由于 `parentFloor` 允许指向任意回复，楼中楼回复也应该计入根评论回复数；这也与文章 `comment_count` 统计全部评论和回复的语义保持一致。 | 创建任意回复时所属根评论 `replyCount +1`；删除回复子树时所属根评论 `replyCount -= affectedCount`；删除根评论时根评论不再展示，`replyCount` 不需要对外维护。 |
| 13 | 回复列表展示形态 | 回复列表接口 `{floor}/replies` 应该返回整棵回复树的平铺列表，还是只返回直接子回复？ | 返回根评论下整棵回复子树的平铺列表，按 `floor ASC` 排序；每条回复通过 `parentFloor` 表达直接父回复关系。 | `replyCount` 已定义为根评论下所有后代回复总数。若列表只返回直接子回复，会造成数量和列表范围不一致。平铺列表分页稳定，前端可根据 `parentFloor` 做缩进、引用或高亮父回复，不需要服务端做多层树分页。 | 回复列表 contract 需要明确 `{floor}` 必须是根评论楼层；列表项必须返回 `rootFloor` 和 `parentFloor`。 |
| 14 | 回复列表默认排序 | 回复展开列表默认按时间顺序，还是按点赞数排序？ | 回复列表仍返回根评论下整棵回复子树的平铺列表，但默认排序使用 `like_count DESC, floor ASC`。 | 展开回复时优先展示更被认可的回复；同点赞数下更早回复优先，保留对话上下文的稳定性。`parentFloor` 继续表达直接父回复关系，前端可据此展示引用或层级提示。 | 需要更新回复列表 contract；如果保留 `TIME` 回复排序选项，再单独登记 `sort=TIME` 使用 `floor ASC`。 |
| 15 | 回复排序一致性 | 回复列表按 `like_count DESC, floor ASC` 排序时，能否接受 `like_count` 最终一致导致排序短暂滞后？ | 可以接受。回复排序依赖最终一致的 `like_count`，点赞后排序允许短暂不立即变化。 | 点赞状态 `viewer.liked` 已强一致，前端可本地点赞数乐观更新；排序刷新由 `comment_counter_deltas` worker 更新统计后自然收敛。强制回复排序实时变化会重新引入点赞统计热点。 | 回复列表 contract 需要说明排序基于异步统计点赞数。 |
| 16 | 回复排序读模型 | 回复是否需要单独的 rank 读模型，还是查询时 join `comment_stats` 排序？ | 首版不建回复 rank 表；回复列表在单个根评论范围内查询时 join `comment_stats`，按 `like_count DESC, floor ASC` 排序。 | 回复列表只在展开某个根评论时查询，范围限定为单个根评论子树，不像文章顶级评论 HOT / RECOMMENDED 需要大范围排序。先保留简单查询，只有真实出现超大回复树和高频展开时再新增 `comment_reply_rank`。 | 需要为 `comments(root_id, status)` 和 `comment_stats(comment_id, like_count)` 查询路径设计索引，并在性能测试中观察大回复树场景。 |
| 17 | 回复列表排序选项 | 回复列表默认按点赞排序后，`floor ASC` 的 TIME 排序还要不要保留为可选参数？ | 保留 `sort=TIME` 作为可选，默认 `sort=HOT`。 | 默认展示高赞回复更符合内容平台体验；但有些用户需要按对话发生顺序阅读完整讨论。`TIME` 作为可选排序成本较低，也为前端后续切换能力预留空间。 | 回复列表 contract 支持 `sort=HOT` -> `like_count DESC, floor ASC`，`sort=TIME` -> `floor ASC`；首版前端可以不暴露切换。 |
| 18 | 顶级评论排序选项 | 顶级评论列表是否保留 `sort=TIME` / `sort=HOT`，同时默认 `sort=RECOMMENDED`？ | 顶级评论列表支持三种排序：`RECOMMENDED` 默认，按 `recommended_score DESC, floor DESC`；`HOT` 严格热门，按 `like_count DESC, floor ASC`；`TIME` 最新，按 `floor DESC`。 | 默认评论流需要自然兼顾热门和新评论曝光；用户明确想看最新或纯热门时仍应有稳定排序选项。三种排序语义清楚，不互相混用。 | 顶级评论 list/page/cursor contract 默认 `sort=RECOMMENDED`，并分别定义 cursor 锚点。 |
| 19 | 顶级评论列表 Total 语义 | `RECOMMENDED` 的 `total` 应该表示什么？ | 顶级评论列表的 `total` 表示该文章未删除顶级评论总数，不表示推荐候选数。 | 排序方式不应改变列表总量语义。分页 UI 和评论区总量需要稳定数字；删除或不可见评论不计入顶级评论 `total`。回复数量通过根评论 `replyCount` 表达。 | `RECOMMENDED`、`HOT`、`TIME` 三种顶级评论排序共享同一个顶级评论 total 语义。 |
| 20 | 顶级评论总数字段命名 | 文章总评论数和顶级评论列表总数不一致时，HTTP response 应该如何命名避免混淆？ | 顶级评论列表分页总数字段使用 `totalTopLevelComments`；文章维度全部评论数继续使用明确的 `totalComments` / `commentCount` 语义。 | 文章总评论数包含根评论和回复；顶级评论列表总数只统计根评论。如果都叫 `total` 或都叫 `commentCount`，前端会误解为什么文章显示的评论数和列表分页总数不同。 | 顶级评论列表 contract 需要把传统 `Page.total` 调整为或补充 `totalTopLevelComments`，并在文档中说明它不包含回复。 |
| 21 | 顶级评论列表返回文章总评论数 | 顶级评论列表 response 是否还要带文章的 `totalComments`？ | 顶级评论列表 response 同时返回 `totalTopLevelComments` 和 `totalComments`。 | 评论区标题通常展示文章总评论数，加载更多顶级评论需要顶级评论总数。两个数字语义不同，放在同一 response 中可以减少前端额外请求和状态不同步。 | `totalComments` 表示文章下根评论和回复的全部未删除评论数；`totalTopLevelComments` 只表示未删除根评论数。 |
| 22 | 文章级评论统计表 | `totalComments` / `totalTopLevelComments` 普通查询时如何获得，是否每次 `COUNT(*) WHERE post_id = ?`？ | Comment 服务维护文章级评论统计表 `comment_post_stats`，普通查询直接读取该表，不在列表请求中实时 `COUNT(*)`。 | `COUNT(*) WHERE post_id = ?` 会随评论量增长变慢，不适合作为普通列表路径。`comment_stats` 已用于单条评论维度统计（`comment_id`、`like_count`、`reply_count`），文章级统计粒度不同，应单独建表。 | `comment_post_stats` 至少包含 `post_id`、`total_comments`、`total_top_level_comments`、`updated_at`；创建 / 删除评论必须在同一事务内维护；全表 COUNT 只作为 `RebuildFromQuery` 修复工具。 |
| 23 | Comment 与 Content 评论计数一致性 | `comment_post_stats.total_comments` 和 Content 的 `post_stats.comment_count` 如果短暂不一致，以谁为准？ | Comment 为评论计数事实源；Comment API 返回 `totalComments` 使用 `comment_post_stats`。Content 的 `post_stats.comment_count` 是消费 Comment 事件后的文章摘要读模型，允许短暂滞后。 | 评论事实 owner 是 Comment，Content 只为了文章卡片 / 详情展示维护评论数副本。副本不能反向成为源数据。 | 必须设计对账机制，定期或按需比对 Comment `comment_post_stats.total_comments` 与 Content `post_stats.comment_count`，并支持事件重放或修复任务同步 Content 副本。 |
| 24 | 评论计数对账归属 | Comment 与 Content 评论计数对账机制归属放在哪里？ | Ops 拥有跨服务对账编排；Comment 和 Content 各自提供受控的只读对账接口 / 修复接口，不跨库读写。 | Comment 是评论计数事实源，但不能直接写 Content 数据库；Content 维护文章摘要副本，但不能把 Comment 统计拉进本地当源数据。Ops 更适合做跨服务巡检、批量对账、差异报告和修复任务编排。 | Comment 提供按 `postId` 查询 / 批量导出 `comment_post_stats`；Content 提供按 `postId` 查询 / 修正 `post_stats.comment_count`；Ops 负责比对、触发修复和记录审计。 |
| 25 | 文章级评论统计初始化 | `comment_post_stats` 在什么时候创建？ | 第一次创建评论时通过 UPSERT 惰性创建；文章无评论时可以没有 `comment_post_stats` 行，查询时返回零值。 | Comment 不拥有文章生命周期，不能要求 Content 创建文章时同步初始化 Comment 统计。惰性创建符合服务边界，并避免为无评论文章预写统计行。 | 查询统计缺失时返回 `totalComments=0`、`totalTopLevelComments=0`；创建根评论和回复时必须在事务内 UPSERT 并递增对应计数。 |
| 26 | 文章级评论统计归零处理 | 删除评论后，如果 `comment_post_stats.total_comments` 减到 0，要删除统计行还是保留零值行？ | 保留零值行。 | 保留行可以表达该 post 曾经初始化过评论统计，减少反复创建 / 删除行；对账时也能明确看到当前统计为 0。查询语义上无行和零值行都返回 0，但写路径保留行更简单。 | 删除逻辑只更新计数到非负值，不物理删除 `comment_post_stats` 行。 |
