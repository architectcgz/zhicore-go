# Comment Domain 设计

Comment 是独立限界上下文。统一语言以“评论、根评论、回复、评论树、评论状态、评论内容、评论媒体、评论点赞、评论统计、游标分页、增量补拉、管理删除、评论事件”为核心。

## `Comment` 聚合

`Comment` 是评论树的聚合根，负责维护评论本身的强一致生命周期。

- 内部标识：`CommentID` 使用 Comment 服务数据库生成的 `BIGINT GENERATED BY DEFAULT AS IDENTITY` 或 sequence，只用于 Comment 内部关联、事务、点赞、统计、缓存和事件。
- 外部定位：`CommentFloor` 是文章内单调递增楼层号，对外用 `(postId, floor)` 定位评论。
- 归属引用：`PostID` 保存 Content 公开 `postId` 字符串；`AuthorID` 引用 User 内部 `UserID`。
- 树结构：`RootID` 指向所属根评论；根评论 `root_id = NULL`。`ParentID` 指向直接被回复的评论；根评论 `parent_id = NULL`。
- 内容字段：文本、图片文件引用集合、语音文件引用、语音时长。
- 状态字段：`Normal`、`Deleted`。
- 行为：`Edit`、`Delete`、`MarkDeletedByAdmin`、`RecordDeletionMetadata`、`IsTopLevel`、`IsReply`、`AssertEditableBy`、`AssertDeletableBy`。
- 领域事件：`CommentCreated`、`CommentDeleted`。更新评论只失效 Comment 本服务缓存，不发布跨服务 `comment.updated`。

`Comment` 聚合只放评论树和评论内容的强一致规则。点赞数、回复数属于独立 `CommentStats` 聚合，避免高频点赞或回复写入锁住评论本体。

## `Comment` 不变量

- 评论必须归属于一个有效 `PostID`，但文章是否存在、是否可评论和是否可见由 application 通过 Content contract 校验。
- 作者是否存在、是否允许互动、是否被拉黑由 application 通过 User contract 校验。
- 评论必须至少包含文本、图片或语音中的一项。
- 文本长度当前固定为最多 2000 字。
- 图片最多 9 张。
- 语音评论不能同时包含图片；语音时长上限按 Java 上传和 DTO 约束迁移，字段级 contract 提取时必须核对最终配置。
- 已删除评论不能编辑、点赞或作为新回复目标。
- 普通用户只能编辑和删除自己的评论。
- 删除任意评论节点时，都软删除该节点及其整棵子树。application 编排批量软删除和统计扣减，repository 只执行持久化批量更新，domain 不访问 repository。
- 管理员删除需要携带 `deletedBy` 和非空 `deleteReason`；作者删除不要求原因。完整审计仍归 Admin，Comment 只保存执行删除所需的最小删除元数据。
- 成功编辑评论时设置 `editedAt`；删除评论保留 `editedAt`，不保存完整编辑历史。

## `CommentStats` 聚合

`CommentStats` 是独立统计读模型聚合根，不放在 `comments` 行内：

- 标识：`CommentID`，与评论一对一。
- 统计字段：`LikeCount`、`ReplyCount`。
- 行为：`ApplyLikeDelta`、`IncrementReplies`、`DecrementReplies`、`RebuildFromQuery`。
- 不变量：所有计数不能为负数。

`ReplyCount` 表示根评论下所有未删除后代回复总数，在创建 / 删除回复子树事务内同步维护。`LikeCount` 由 `comment_counter_deltas` 后台批量聚合后更新，避免高 QPS 点赞请求争抢同一统计行。`RebuildFromQuery` 用于从 `comment_likes` 和 `comments` 表重建 `like_count` / `reply_count`，修正手工 SQL、迁移、脚本绕过 application 或未来异步化导致的计数漂移。

## `CommentLike`

`CommentLike` 是以 `(CommentID, UserID)` 为自然唯一键的互动关系实体，不作为独立聚合根暴露生命周期。

点赞和取消点赞由 application 在同一 PostgreSQL 事务内维护关系事实、delta 和事件：

```text
comment_likes 插入/删除
+ comment_counter_deltas 追加 LIKE +/-1
+ outbox_events(comment.liked / comment.unliked)
```

重复点赞或取消未点赞是幂等成功，不写 delta，不写事件。`viewer.liked` 以 `comment_likes` 为强一致事实；`likeCount`、HOT / RECOMMENDED 排序和缓存计数允许短暂最终一致。

## `CommentMedia`

评论媒体是 `Comment` 聚合内的值对象或子对象：

- `ImageFileIDs` 保存 Upload / File Service 返回的图片文件 ID。
- `VoiceFileID` 保存 Upload / File Service 返回的语音文件 ID。
- `VoiceDuration` 保存语音时长。

Comment 只拥有“评论引用了哪些媒体文件”和评论内容规则，不拥有文件元数据、对象存储路径、URL 生成 / 解析规则、签名 URL、CDN 规则或文件删除事实。查询 DTO 可以带展示 / 播放 URL，但这些 URL 是派生值。

## 非聚合对象

以下对象不建成领域聚合：

- `outbox_events`：跨服务可靠消息机制，不是 Comment 领域对象。
- outbox summary、dead retry、metrics：管理和运维视图，不进入 domain。
- 首页评论缓存、详情缓存、点赞缓存：Redis 读模型，不是真相源。
- Ranking hot candidate：Ranking 拥有榜单和分数，Comment 只读取热门候选作为缓存或物化输入。
- Sentinel / 限流配置：运行时保护机制，属于 infrastructure 或 middleware。

## 值对象

| 值对象 | 含义 | 约束 |
| --- | --- | --- |
| `CommentID` | 评论内部标识 | 不进入对外 HTTP path。 |
| `CommentFloor` | 文章内楼层号 | 同一 `post_id` 下唯一；删除后不复用、不重排。 |
| `PostID` | Content 公开 `postId` 字符串引用 | application 校验文章事实；Comment 不保存 Content 私有主键。 |
| `UserID` / `AuthorID` | User 内部用户引用 | application 校验用户事实；对外 HTTP 作者摘要使用 User `publicId`。 |
| `RootCommentID` | 根评论标识 | 根评论为空。 |
| `ParentCommentID` | 直接被回复评论标识 | 根评论为空。 |
| `CommentContent` | 评论文本 | 评论整体必须非空；文本最多 2000 字。 |
| `ImageFileIDs` | 评论图片文件引用集合 | 最多 9 张；必须是 Upload / File Service 返回的 opaque file ID。 |
| `VoiceFileID` / `VoiceDuration` | 评论语音文件引用和时长 | `VoiceFileID` 必须是 Upload / File Service 返回的 opaque file ID；语音不能与图片同时存在；时长必须为正数。 |
| `CommentStatus` | 评论生命周期状态 | `NORMAL`、`DELETED`。 |
| `DeletionMetadata` | 删除操作者、原因和时间 | 完整审计归 Admin。 |
| `CommentStatsValue` | 非负计数值 | 点赞数和回复数不能为负。 |
| `CommentSortType` | 排序方式 | 顶级评论支持 `RECOMMENDED`、`TIME`、`HOT`；回复支持 `HOT`、`TIME`。 |
| `CursorToken` | 不透明游标 | 不能暴露内部 SQL 排序列语义。 |
| `AuthorSnapshot` | 评论展示作者摘要 | 查询 DTO 或快照，不是 User 源数据。 |

## 游标值

首版采用简单不透明 token：

- `TIME` 游标使用 URL-safe Base64 无 padding 编码，原文为 `floor`。
- `HOT` 游标使用 URL-safe Base64 无 padding 编码，原文为 `likeCount + "_" + floor`。
- `RECOMMENDED` 游标使用 URL-safe Base64 无 padding 编码，原文为 `recommendedScore + "_" + floor`。
- 游标不加密、不签名，只隐藏内部实现细节，不提供安全防篡改能力；解码失败按参数错误处理。
- 如果后续要增加签名或版本字段，必须作为 API 演进处理，旧游标在一个过渡期内仍应可解析。
- 如果未来排序加入 `createdAt`、更多分数或其他排序列，cursor 必须包含所有排序锚点。

## 领域服务

| 领域服务 | 职责 |
| --- | --- |
| `CommentContentPolicy` | 校验文本、图片、语音组合规则和长度限制。 |
| `CommentTreePolicy` | 判断根评论、回复、被回复评论之间的树结构是否合法。 |
| `CommentPermissionPolicy` | 判断普通作者或管理员是否可编辑、删除评论；只基于已加载聚合和操作者上下文。 |
| `CommentInteractionPolicy` | 判断评论状态是否允许点赞、取消点赞或回复。 |
| `CommentSortPolicy` | 解释 `RECOMMENDED` / `TIME` / `HOT` 排序语义，但不生成 SQL。 |

## 工厂

`CommentFactory` 封装复杂聚合创建逻辑：

- `CreateTopLevel(postID, floor, authorID, content, media)`：创建根评论，`RootID` 和 `ParentID` 均为空。
- `CreateReply(postID, floor, authorID, rootComment, parentComment, content, media)`：创建回复，验证根评论和直接父评论存在且未删除，根评论确实是顶级评论，父评论属于同一根评论树，三者属于同一文章，并设置 `RootID` 和 `ParentID`。

Go 当前不强制让工厂生成 ID。内部主键由数据库生成时，application 可以先创建无 ID 的领域对象，再由 repository 保存后返回 `CommentID`，并在同一事务中写统计和 outbox。
