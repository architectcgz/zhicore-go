# Comment ID 设计

本文是 Comment 评论标识和排序锚点的事实源。`service.md` 只引用本文，不重复发号逻辑。

## 结论

- 评论内部主键使用 `comments.id BIGINT GENERATED BY DEFAULT AS IDENTITY`，由 PostgreSQL identity 在插入时生成。
- 对外评论定位使用 `(postId, commentId)`；`commentId` 是由内部 `comments.id` 派生的短字符串，不直接暴露自增数字。
- `floor` 不再作为评论资源 ID、排序锚点或 cursor 锚点；第一版不保留独立楼层号。
- 不使用 Redis、segment、每文章 counter 或 `zhicore-id-generator` 为评论发号。
- `TIME`、`HOT` tie-breaker、`RECOMMENDED` tie-breaker 和 cursor 内部锚点统一使用内部 `comments.id`。

## 为什么不用文章内 `floor`

文章内 `floor` 如果用 `comment_post_counters` 行锁分配，会把同一篇文章的并发创建串行化。改成 DB segment 虽然能减少取号次数，但会让不同实例提前持有不连续号段：

```text
实例 A 先取到 1..50
实例 B 后取到 51..100
B 的评论先提交，A 的评论后提交
```

此时如果用 `floor DESC` 表示最新，后发或后提交评论会被错误排序。Segment 能保证不重复，但不能保证“号段顺序 = 评论提交顺序”。

评论创建本来必须写 PostgreSQL。直接使用 `comments.id` 作为插入时生成的主键，不需要额外取号请求，也不会引入 per-post counter 热点。

## ID 生成

`comments` 表使用数据库 identity：

```sql
CREATE TABLE comments (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  post_id VARCHAR(32) NOT NULL,
  content_internal_id BIGINT NOT NULL,
  author_id BIGINT NOT NULL,
  ...
);
```

创建评论时由 repository 插入并返回内部 ID：

```sql
INSERT INTO comments (...)
VALUES (...)
RETURNING id;
```

约束：

- 多个 Comment 服务实例共享同一个 PostgreSQL 主库时，identity 由数据库统一发号，不会重复。
- identity 背后的数据库发号器不保证连续；事务回滚、插入失败或数据库缓存都可能产生 gap。
- identity 表达取号顺序，不是严格提交顺序。Comment 本地创建事务应保持短事务；当前排序接受这个边界。
- 如果未来出现多主写入、离线写入后合并或单库 identity 发号成为明确瓶颈，再重新评估 Snowflake / segment / 独立发号集群。

## 对外 `commentId`

HTTP 不直接暴露内部 `comments.id` 数字。Comment 对外返回字符串 `commentId`：

```text
内部 id -> 64-bit 可逆置换 -> Base58/Base62 编码 -> prefix + version + code
```

示例：

```text
comments.id = 123456
commentId = c1K8x9Q2
```

要求：

- `commentId` 必须能被 Comment 服务解析回内部 `comments.id`，或通过本地唯一映射查询到内部 ID。
- `prefix` 建议使用 `c`，`version` 用于后续算法演进。
- 可逆置换必须带服务级密钥或盐，避免简单 Base62 直接暴露连续增长。
- 解码失败、版本未知或映射不存在时，对外按 `Comment not found` 或 `Invalid request` 的 endpoint 语义处理。

## 排序和 Cursor 锚点

对外 contract 仍保持 cursor 不透明；内部锚点使用 `comments.id`：

| 排序 | 内部排序 | 说明 |
| --- | --- | --- |
| 顶级评论 `TIME` | `id DESC` | 新评论优先。 |
| 顶级评论 `HOT` | `like_count DESC, id ASC` | 同点赞数下更早评论优先。 |
| 顶级评论 `RECOMMENDED` | `recommended_score DESC, id DESC` | 同分时新评论优先。 |
| 回复 `TIME` | `id ASC` | 同一根评论下按创建顺序阅读。 |
| 回复 `HOT` | `like_count DESC, id ASC` | 同点赞数下更早回复优先。 |

Cursor 内部必须包含对应排序的全部稳定锚点：

- `TIME`：`id`
- `HOT`：`likeCount + id`
- `RECOMMENDED`：`recommendedScore + id`

## Migration 影响

- 不创建 `comment_post_counters`。
- 不创建 `comment_post_floor_segments`。
- 不创建 `comments(post_id, floor)` 唯一索引。
- `comment_hot_rank` 和 `comment_recommended_rank` 使用 `comment_id` 作为 tie-breaker 字段；需要按查询路径建立复合索引。
- Redis 的 `comment:floor:{postId}` key 和任何楼层号段缓存不进入 Go 设计。
