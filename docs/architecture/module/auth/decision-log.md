# Auth 决策日志

本文件记录 Auth 模块设计压测过程中已经确认的设计问题、结论、原因和后续依赖。

| # | 决策项 | Question | Decision | Rationale | Follow-up |
| --- | --- | --- | --- | --- | --- |
| 1 | Auth/User 身份映射 | Auth 是否在账号表保存 User 返回的内部 `userId`，并把 `accountId` 和 `userId` 都放进 access token claims？ | Auth 在 `auth_accounts.user_id` 保存 User 返回的内部用户 ID；access token 携带 `accountId`、`userId`、`sessionVersion`、`principalVersion`、`jti`、`iat`、`exp`、`type=access`；Gateway 注入 `X-Account-Id` 和 `X-User-Id`。 | User 已确认服务间写路径和 Gateway 身份上下文使用内部 `userId`；Auth 持有映射可避免 Gateway 认证热路径同步查 User，并让注销、封禁和 token 失效闭合。 | 注册链路必须由 User `CreateProfileForAccount` 返回 `userId`，Auth 再完成账号可用状态。 |
| 2 | 注册一致性和 pending 重试 | 注册时 Auth 本地账号和 User profile 怎么形成一致结果？User 下游不可用后前端重试是否从 pending 状态继续？ | 首批采用同步编排；Auth 先创建 `PENDING_PROFILE` 账号并暂不签发 token；User `CreateProfileForAccount` 成功返回 `userId` 后 Auth 写入 `auth_accounts.user_id` 并切 `ACTIVE`；同一注册请求重试时，如 `email_normalized` 命中 `PENDING_PROFILE`，允许用本次请求覆盖 pending 的 `nickname/password` 后继续 User 初始化；User 按 `accountId` 幂等返回同一个 `userId`。 | 跨服务不能共享事务；`PENDING_PROFILE` 显式隔离“账号已创建但 profile 未闭合”的裂缝，避免前端拿到可登录但无 User 资料的账号；邮箱所有权证明由后续 #76-#81 的验证码/token 体系闭合。 | 注册链路、验证码表和 pending 释放策略以 `data-model.md` 和后续 HTTP schema 为准。 |
| 3 | 账号状态集 | Auth 需要哪些账号主状态表达正常、注册未完成、主动注销、管理员封禁和 pending 过期废弃？ | 首批账号主状态固定 `PENDING_PROFILE`、`ACTIVE`、`DISABLED`、`BANNED`、`EXPIRED`；暂不做 `PENDING_VERIFICATION`；临时风控锁定不作为主状态，使用 `locked_until/lock_reason` 派生表达。 | 当前没有邮箱/手机验证需求；主状态需要覆盖注册恢复、用户注销、管理员封禁和 pending 过期废弃，并避免把 Auth 封禁复制到 User；`EXPIRED` 比 `DELETED` 更准确表达未激活账号的超时废弃；锁定是临时风控属性，不应覆盖账号主状态。 | 需要固定 `PENDING_PROFILE` 和长期未使用账号对登录标识唯一性的影响。 |
| 4 | 注册标识和释放策略 | 注册失败或账号注销/删除后，`email` 和 `username` 是否释放？ | Auth 注册不再使用 `username`，只接收 `nickname`、`email`、`password`；`email` 是 Auth 登录标识，`nickname` 是 User 资料字段；注册失败允许释放 `email` 后用同一 `email` 换 `nickname` 重试；注销/删除后首批不释放 `email`/`nickname`；长期未使用后的释放策略后续单独设计。 | `username` 和 `nickname` 同时存在会制造重复展示名口径；注册失败不应永久占用邮箱；已激活后的注销/删除涉及账号接管和历史归属，首批安全优先不释放。 | 需要固定 Auth 是否持久化 pending 阶段的 `nickname`，以及长期释放策略的触发条件。 |
| 5 | Pending nickname/password 语义 | `PENDING_PROFILE` 重试时是否允许更换 `nickname/password`？Auth 是否长期持有 `nickname`？ | Auth 不把 `nickname` 当长期账号字段；pending 阶段可在 `registration_nickname` 或注册状态中临时保存；`email_normalized` 命中 pending 后允许更换 `nickname` 和 `password`；User 返回 `USER_NICKNAME_TAKEN` 时保留 pending 可重试状态，用户可换 nickname 再提交；成功后 Auth 不维护 nickname 权威含义。 | 支持注册失败换 nickname/密码，同时避免 Auth 成为 nickname 第二 owner；账号 ACTIVE 前还没有 User profile，不存在资料一致性风险。 | 需要在字段级 HTTP schema 中把注册请求固定为 `nickname`、`email`、`password`。 |
| 6 | Token 失效语义 | 账号封禁、注销、改密码、角色变更和单设备登出后，Gateway 怎么判断旧 access token 是否仍可用？ | Access token 带 `jti`、`exp`、`accountId`、`userId`、`sessionVersion`、`principalVersion`；Auth 维护 `auth_accounts.session_version` 和 `auth_accounts.principal_version`；单设备登出吊销当前 refresh token 并把当前 access token `jti` 写入 Redis 黑名单至过期；全设备登出、改密码、注销、封禁等增加 `sessionVersion` 并吊销账号全部 refresh token；Gateway 校验 `jti` 黑名单和当前账号 `sessionVersion/principalVersion` 缓存。 | 单设备登出只吊销 refresh token 会违背“登出后不能继续访问”的语义；`jti` 黑名单闭合单 token 失效，`sessionVersion` 闭合账号级强制失效，同时避免 Gateway 每次同步打 Auth。 | 需要固定 refresh rotation 是否黑名单旧 access token；当前结论是不因 refresh 自动黑名单旧 access token。 |
| 7 | `auth/me` 返回边界 | `GET /api/v1/auth/me` 是否返回昵称、头像、bio、avatarUrl 等 User profile 字段？ | `auth/me` 只返回认证主体，例如 `accountId`、`userId`、`email`、`roles`、`accountStatus`、`sessionVersion`、`principalVersion` 和必要认证上下文；User profile 字段由 `GET /api/v1/users/me` 返回。 | Auth 只回答“你是谁、账号是否有效、有什么角色”；返回 profile 会复制 User DTO，并把 avatarUrl 解析职责混进 Auth。 | 需要在 Auth HTTP schema 和 User HTTP schema 中分别固定两个 `me` 的字段边界。 |
| 8 | 主动注销编排 | 用户主动注销时 Auth 和 User 的执行顺序是什么？User 调用失败时前端看到什么？ | 注销入口归 Auth，要求当前密码确认；Auth 先将账号置为 `DISABLED`、递增 `sessionVersion`、吊销全部 refresh token 并黑名单当前 access token；再调用 User `DeactivateUserProfile`；User 成功返回 `200`；User 失败则账号保持 `DISABLED`，写补偿任务和告警，返回 `202 Accepted`/`AUTH_DEACTIVATION_PROCESSING`，前端清理本地登录态并显示处理中。 | 注销后登录态失效优先级高于资料停用；不应在 User 失败时回滚账号可登录状态，也不应谎称资料已完成注销。 | 需要固定补偿任务存储和管理端排查入口。 |
| 9 | 管理员封禁边界 | 管理员封禁/解封是否调用 User，状态如何恢复？ | 封禁命令归 Auth，可由 Admin facade 暴露；封禁只改 Auth 状态为 `BANNED`、递增 `sessionVersion`、吊销全部 refresh token、清理 Gateway principal 缓存并发布 `auth.account.banned`；不调用 User；解封只允许 `BANNED -> ACTIVE`，不恢复 `DISABLED` 主动注销账号；解封后用户需重新登录；如需资料隐藏，Admin facade 单独调用 User `MarkUserDeleted`。 | 封禁是认证和访问控制事实，不是 User 资料生命周期；保持单 owner，避免 Auth/User 双写 banned 状态。 | 需要固定 Admin facade 调 Auth 的 typed client command。 |
| 10 | 角色变化和 principal 刷新 | 角色从普通升级到 VIP 或降级时，旧 token 如何处理，Gateway 刷新 principal 后是否回写前端？ | 区分 `sessionVersion` 和 `principalVersion`；改密码、注销、封禁、全设备登出递增 `sessionVersion` 并使旧 access token 401；角色/VIP 变化递增 `principalVersion`，Gateway 发现过期后调 Auth typed client 获取最新 principal，更新缓存并用最新 roles/status 注入下游，不强制 401、不吊销 refresh token、不在业务响应中回写新 token；前端展示通过关键页面/route guard/主动 `auth/me` 刷新全局 store。 | 角色变化不等于认证失效；升级 VIP 返回 401 体验和语义都不对；后端权限必须即时以 Auth 当前 principal 为准，前端 UI 刷新不应和 Gateway 普通业务响应耦合。 | 需要在 Gateway 设计中登记 principal cache 和版本刷新规则。 |
| 11 | Email 登录标识规范 | `email` 登录标识大小写是否视为同一个账号？ | Auth 保存展示用 `email` 和唯一查询用 `email_normalized`；`email_normalized=trim + lower-case`；注册唯一性和登录查询都用 `email_normalized`；不做 Gmail 点号/加号等 provider-specific 归一化；User 不保存 email。 | 邮箱登录标识大小写敏感会制造重复账号和登录/找回歧义；通用 lower-case 足够保守。 | 需要在 Auth schema 固定 `email_normalized` 唯一约束和释放策略。 |
| 12 | 密码策略 | 密码长度校验是加密前还是加密后？首批允许多长？ | 校验原始 `password` 字段；长度 `8..32` 个 Unicode 字符，同时 UTF-8 字节数 `<=128 bytes`；至少包含字母和数字；不强制特殊字符；后端不 trim 密码内容，按原始字符串 hash；`password_hash` 预留 `VARCHAR(255)`；`PasswordHasher` 首批推荐 Argon2id，bcrypt 也不会触发 72 bytes 截断风险。 | 密码长度是用户输入约束，不是 hash 长度；`8..32` 更符合常规体验，字节上限防止异常多字节输入和算法边界问题。 | 需要在 Auth HTTP schema 和错误码中固定 `AUTH_PASSWORD_INVALID`。 |
| 13 | Refresh token transport | Web 首批 refresh token 用 body token 还是 HttpOnly cookie？rotation 是什么语义？ | Web 首批采用 HttpOnly cookie refresh token：`login/register/refresh` 通过 `Set-Cookie` 写入 `refresh_token`，不在响应 body 返回；access token 仍在 body 返回并由前端以 Bearer 调业务 API；refresh/logout/deactivate 等 session 变更接口读取 cookie；rotation 每次 refresh 都签发新 refresh token 并覆盖 cookie，旧 `tokenId` 失效；服务端只在 PostgreSQL refresh session 保存 `sessionId/currentTokenId/currentTokenHash`，不保存 refresh token 明文；多设备用 `sessionId` 建模；变更 session 的 endpoint 需要 CSRF 保护。 | HttpOnly cookie 由浏览器 Cookie Jar 保存，JS 读不到 refresh token 字符串，降低 XSS 直接窃取 refresh token 后异地长期复用的风险；rotation 降低 refresh token 泄露后的长期共用风险；cookie 自动携带带来 CSRF 风险，必须配套防护。 | 需要固定 SameSite 策略、CSRF token 方案和 refresh 重放处置范围。 |
| 14 | SameSite 和 CSRF | Cookie refresh token 的 SameSite 策略和 CSRF 防护怎么做？ | 首批使用 `SameSite=Lax`；`refresh_token` cookie 设置 `HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth; Max-Age=...`；同时设置非 HttpOnly 的 `csrf_token` cookie；`refresh`、`logout`、`account/deactivate` 等会改变 session 的 endpoint 必须提交 `X-CSRF-Token`，Auth 校验 header 与 cookie 中的 `csrf_token` 一致；`auth/me` 不改变 session，可不要求 CSRF。 | `SameSite=Lax` 能阻止大多数跨站 POST 自动携带 cookie，同时保留外部顶层导航的可用性；不能只依赖 SameSite，double-submit CSRF token 能补齐 cookie 自动携带带来的 CSRF 风险。 | 需要在 Auth HTTP schema 中登记 cookie/header 名称和 CSRF 错误码。 |
| 15 | Refresh 重放处置 | refresh token 为什么会被重复使用？rotation 发现旧 token 重放后吊销当前 session 还是全账号？ | refresh token 重复使用可能来自多标签页并发、网络重试、代理重发、前端未做 refresh singleflight，也可能来自泄露攻击；首批发现已失效 `tokenId` 被再次使用时，吊销当前 `sessionId` 并记录安全事件，不立即全账号下线；同账号短时间多次、跨 IP/UA 异常或多个 session 重放时，再升级为全账号吊销或设置 `locked_until` 临时锁定；前端必须实现 refresh 请求 singleflight，避免并发触发误判。 | rotation 下旧 token 再出现是重放信号，但并不一定是攻击；先按 session 处置可降低误伤，singleflight 能减少正常并发导致的重复 refresh。 | 需要在前端接入说明或 Auth HTTP schema 中登记 refresh singleflight 要求。 |
| 16 | Auth 集成事件范围 | Auth 哪些变化需要发集成事件？ | 首批发布 `auth.account.registered`、`auth.account.disabled`、`auth.account.banned`、`auth.account.unbanned`、`auth.role.changed`、`auth.password.changed`；`auth.account.registered` 只在账号 `ACTIVE` 且 User profile 初始化成功后发布，payload 含 `accountId`、`userId`、`email`、`occurredAt`，不含 password、token、nickname；session 级撤销首批只做 Auth 内部安全审计，不作为跨服务事件。 | 业务服务通常只需要账号状态/角色变化来刷新权限或读模型；session 级事件太细，容易制造噪声，且主要属于安全审计。 | 需要更新 `libs/contracts/events/auth/README.md` 和 Auth data-events 文档。 |
| 17 | Auth typed client 能力 | Auth provider-owned typed client 首批提供哪些同步能力？ | 首批提供 `ValidateAccessState(claims)`、`GetAccountPrincipal(accountId)`、`BanAccount(accountId or userId, reason, operatorId)`、`UnbanAccount(...)`、`AssignRole(...)`、`RemoveRole(...)`、`RevokeAccountSessions(accountId or userId, scope)`；`ValidateAccessState` 供 Gateway 在 Redis 不可用、L1 miss、principal 版本需刷新或 refresh 降级后的新 AT 校验时回源 Auth，不接收 raw access token，只接收 Gateway 已验签解析出的 `accountId/userId/sessionId/jti/sessionVersion/principalVersion/iat/exp` claims；Auth 以 PostgreSQL account/session 真相源判断 `ALLOW/DENY`，并在 principalVersion 落后但 session 有效时返回最新 principal；`BatchGetAccountPrincipal` 先登记为后续可选；命令入参可用 `accountId` 或 `userId` 二选一，Auth 内部解析为 `accountId`；返回始终同时带 `accountId` 和 `userId`；不提供 profile 字段或按 email 查用户资料能力。 | Gateway 需要刷新 principal，也需要在 Redis 故障时确认 session revoked、sessionVersion、principalVersion 和 account 状态；单纯 `GetAccountPrincipal` 容易漏掉 session 维度。Auth contract 不能扩散 User profile DTO，也不能让 Gateway 传 raw token 给 Auth 二次解析。 | 需要更新 `libs/contracts/clients/auth/README.md`、Gateway fallback 设计、Auth application query 和错误码映射。 |
| 18 | Auth 首批 HTTP 范围 | Auth 首批前端/API endpoint 固定哪些？管理端 endpoint 是否一起提取？ | 首批固定 `POST /api/v1/auth/register`、`POST /api/v1/auth/login`、`POST /api/v1/auth/refresh`、`POST /api/v1/auth/logout`、`GET /api/v1/auth/me`、`GET /api/v1/auth/csrf`、`GET /api/v1/auth/sessions`、`DELETE /api/v1/auth/sessions/current`、`DELETE /api/v1/auth/sessions/{sessionId}`、`GET /api/v1/auth/security-operations/{operationId}`；密码修改、密码重置、账号禁用/启用、角色变更和 Admin 管理端 HTTP 后续补；管理端封禁、解封、角色调整和强制下线首批只固定 typed client 和 Admin facade 边界，不直接提取管理端 HTTP schema。 | 首批 Auth 聚焦用户登录态、refresh/session 管理和安全 operation 可见性；Admin API 的 path、审计、权限和批量操作应在 Admin 模块统一设计。 | 已同步 `services/zhicore-auth/api/http/README.md` endpoint 索引；后续补齐单 endpoint 字段级 schema。 |
| 19 | Pending 注册清理而非自动补偿 | User 下游不可用导致 `PENDING_PROFILE` 后，是否由后台 worker 自动创建 User profile？ | 注册 pending 不做自动补偿成功；前端重试是唯一继续注册的主路径；Auth 可有后台过期清理任务释放长期 pending 的 `email_normalized`，但不在用户已经看到注册失败后悄悄激活账号；注销场景不同，`DEACTIVATE_USER_PROFILE` 可以后台补偿，因为用户已经明确提交注销且 Auth 已禁用登录态。 | 后台自动完成注册会造成“用户看到失败但系统后来莫名成功”的体验和安全问题；注册成功应由用户当前请求触发，注销补偿则是在用户明确意图后的收尾。 | 需要固定 pending 保留期和过期释放规则。 |
| 20 | Pending 保留期 | `PENDING_PROFILE` 保留多久，过期后如何处理？ | 首批 `PENDING_PROFILE` 保留 30 分钟；过期清理后释放 `email_normalized`；pending 不是有效账号资产，不签发 token，不进入 User profile；后续接入邮箱验证码后可重新评估保留期。 | 30 分钟足够覆盖用户前端重试和短暂下游故障，又避免注册失败长期占用邮箱。 | 需要在 Auth data/schema 文档中登记清理任务。 |
| 21 | 注册公开错误语义 | 注册时 pending、email 占用、nickname 占用和输入非法分别返回什么？ | 首批使用 `AUTH_EMAIL_EXISTS`/409 表示 email 被占用主状态账号占用，即 `PENDING_PROFILE`、`ACTIVE`、`DISABLED`、`BANNED`；`AUTH_REGISTER_PENDING_RETRYABLE`/503 或 409 表示 email 处于 `PENDING_PROFILE` 且本次 User 下游仍不可用，前端可稍后重试；`USER_NICKNAME_TAKEN`/409 表示 nickname 已占用，pending 保留且可换 nickname 重试；`AUTH_PASSWORD_INVALID`/400、`AUTH_EMAIL_INVALID`/400、`AUTH_CSRF_INVALID`/403 分别表示密码、邮箱和 CSRF 错误；pending 命中且本次初始化成功时直接返回注册成功。 | 前端需要区分“换 nickname 重试”、“稍后重试”和“邮箱已不可用”；pending 是内部恢复状态，成功时不暴露给前端；临时锁定不影响 email 唯一占用。 | 需要在 `docs/contracts/error-codes.md` 和 Auth HTTP schema 中登记。 |
