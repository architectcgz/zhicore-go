# Auth 提问记录

本文件记录 Auth 设计压测过程中用户提出的澄清问题、术语问题、风险追问、例子请求和后续回看线索；它不是决策日志。

## 问题索引

| 主题 | 覆盖问题 | 关键结论 |
| --- | --- | --- |
| Outbox 与 RabbitMQ | #1-#2 | Auth 业务事务写 outbox；DB commit 后可 best-effort 发 RabbitMQ；可靠投递由后台 publisher 补偿完成。 |
| Auth MQ 事件与用户通知 | #3 | 永久封禁发 `auth.account.banned.v1` 系统事件；站内信/邮件/短信等用户可见通知由对应通知/消息服务消费事件后处理。 |
| 邮箱验证 token | #4 | `emailVerificationToken` 是验证码已通过后的短期一次性凭证，绑定 `email_normalized` 和 `purpose=register`。 |
| 验证码发送服务边界 | #5 | Auth 负责验证码安全生命周期；验证码邮件由 Mail/Message Delivery 服务发送，不归站内 Notification 语义。 |
| 验证码尝试次数 | #6 | 单条验证码最多校验尝试 5 次，达到上限后该验证码失效，即使之后输入正确 code 也需要重新发送。 |
| Argon2id 参数与小内存部署 | #7-#12 | `memory` 是每次 hash/verify 的内存成本，`t` 是迭代轮数，`p` 是并行度；线上限流保护登录接口，memory-hard 主要防 DB 泄露后的离线爆破；小内存默认倾向 `32MiB,t=3,p=1` 并加 hash 并发上限，极小内存可 fallback bcrypt。 |
| 密码凭证存储、salt 与 pepper | #13-#15 | 密码表存 PHC 格式 `password_hash`，其中包含算法、参数、salt 和最终 hash；salt 可随 hash 存 DB；明文 password 和 pepper 不进 DB；pepper 是服务端 secret，只在“DB 泄露但 secret 未泄露”时提供额外防线。 |
| Admin 角色撤销规则 | #16 | 首期多个 admin 也不能通过普通 Admin role API 撤销任何账号的 `ROLE_ADMIN`；admin 授权可通过 bootstrap/Admin 授予，撤销需走更高阶运维或后续专门流程。 |
| Access token `jti` 与 session | #18 | `jti` 是单个 access token 的唯一编号，用于黑名单、审计、重放排查和精确吊销；`sessionId` 标识登录设备/会话，二者粒度不同。 |
| Gateway access 状态缓存 | #19-#24 | Gateway 可用进程内 L1 短 TTL 缓存优化 Redis 查询，但 Redis 仍是跨实例共享撤销/版本事实源；Redis 不可用时，L1 只能使用尚未过期的已知状态，不能证明 token/session 没被新撤销；高风险路径由 Gateway route policy 的高风险规则集显式标记，不能只依赖本地缓存；公开资料显示授权缓存、外部授权 fail-open/fail-closed 开关和 token introspection/revocation 都是常见实践，JWT blocklist 场景尤其强调 Redis/存储持久化和高可用，否则 revoked token 会重新有效。 |
| 时间容忍窗口 `clockSkew` | #25 | `clockSkew` 是服务间时钟不完全一致时的容忍窗口，不是业务有效期；Auth、Gateway、Redis、DB 的机器时间可能差几秒，例如 Auth 认为 `10:00:00`，Gateway 认为 `09:59:58`；JWT `iat/exp/nbf` 校验可允许默认 60 秒误差；黑名单 TTL 用 `access token 剩余有效期 + clockSkew`，避免某台 Gateway 仍认为 token 未过期时 Redis 黑名单已删除。 |
| Redis 写失败与 logout/security revoke | #26-#28 | `logout current` 中 DB 是会话事实源，成功后 `auth_refresh_sessions` 会标记当前 session revoked，例如写 `revoked_at/revoked_reason` 并使 refresh cookie 对应 session 不可再 refresh；Redis 是 Gateway 快速感知 access token/session 撤销的投影。如果 Redis 写 `jti blacklist`/`session revoked` 失败，旧 AT 仍可能继续访问；最终决策是不返回完全成功，而是返回 `202 PROCESSING` 或失败，同时清 cookie、写 audit/告警并交给补偿任务写 Redis。若用户目标是阻止被盗 AT，必须等待撤销投影写入成功或进入全局 fail-closed/降级，不能向用户承诺“已安全退出”。Redis 成功应定义为写命令成功返回且必要时读回/WAIT 确认，不只是把任务扔进内存队列。 |
| 安全操作异步补偿与页面关闭 | #29-#30 | `202 PROCESSING` 的 operation 是后端持久状态，不依赖前端页面存活；用户关页后 worker 继续补偿，operation 最终变为 `SUCCEEDED/FAILED`；正常 Redis 短抖动下应在数秒内完成，默认前端轮询 1-2 秒起步、总等待 10-30 秒后转后台处理中；下次登录或打开安全中心可查询最近未完成/失败的安全操作。 |
| Redis 不可用时 Auth 降级 | #31 | Redis 不可用时不能全站 fail-open，也不应全站 fail-closed；按能力分级：安全撤销/权限收敛类 fail closed 或 degraded 202，普通登录/注册可短时用本机限流降级，refresh 依赖 DB rotation 但若无法更新 Gateway 可见状态则返回降级错误，普通已登录读请求由 Gateway L1 短时兜底。 |
| 验证码状态与 Redis 依赖 | #32-#33 | 按已定设计，验证码状态、尝试次数、消费状态在 DB 中闭合，Redis 只做发送/接口限流加速或分布式计数；verify 不应因为 Redis 不可用而无法判断验证码正确性。refresh session 等最终事实也以 DB 为准，但 Gateway 不每次查 DB，Redis/Gateway 投影决定新 AT 是否能被立即顺畅使用。 |
| Refresh 降级与 Gateway 回源 | #34 | refresh 降级已经在 decision #107、`rate-limiting.md`、Gateway README 和 `route-risk-policy.md` 中定过：Redis 正常时正常 rotation；Redis 短时不可用且 Gateway 能通过 `ValidateAccessState(claims)` 回源 Auth 校验 access state 时，允许短时降级成功；Redis 不可用且 Gateway 不能回源 Auth 时 refresh 返回 `503`；存在未完成安全 operation 或高风险撤销投影未完成时 refresh 失败或返回安全同步中。 |
