# User 数据和事件设计

本文记录 User 模块目标数据归属、schema 草案、缓存 key、事件 payload 和跨服务一致性规则。完整 migration SQL 后续按 `docs/architecture/migrations.md` 落到 `services/zhicore-user/migrations/`。

## 数据归属

User 拥有：

- `users`
- `user_follows`
- `user_follow_stats`
- `user_blocks`
- User 本地 `outbox_events`

Check-in 属于 User 完整边界，但不进入首批 schema 细化；后续再补 `user_check_ins`、`user_check_in_stats` 和 Redis bitmap。

User 不拥有：

- Auth `accounts`、password、role、token、封禁状态。
- Upload / File Service 文件元数据和 URL。
- Content 文章、Comment 评论、Message 私信、Notification 通知。

本地表之间不建数据库 FK，跨服务也不建 FK。关系完整性由 application/repository 事务、唯一约束、计数非负保护和测试维护。

## `users` schema 草案

```sql
CREATE TABLE users (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  public_id VARCHAR(64) NOT NULL UNIQUE,
  account_id BIGINT NOT NULL UNIQUE,
  nickname VARCHAR(15) NOT NULL UNIQUE,
  avatar_file_id VARCHAR(128),
  bio VARCHAR(100) NOT NULL DEFAULT '',
  stranger_message_allowed BOOLEAN NOT NULL DEFAULT TRUE,
  user_status VARCHAR(32) NOT NULL DEFAULT 'ACTIVE',
  profile_version BIGINT NOT NULL DEFAULT 0,
  deleted_reason VARCHAR(255),
  deleted_by BIGINT,
  deleted_at TIMESTAMPTZ,
  restored_reason VARCHAR(255),
  restored_by BIGINT,
  restored_at TIMESTAMPTZ,
  deactivated_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL,
  CHECK (profile_version >= 0),
  CHECK (user_status IN ('ACTIVE', 'DEACTIVATED', 'DELETED'))
);
```

规则：

- `nickname` 唯一大小写敏感。
- 写入前 trim 首尾空白；`Alice` 和 `alice` 可共存，`Alice ` 和 `Alice` 不可绕过。
- 非 ACTIVE 用户继续占用 `nickname`。
- `public_id` 永不复用。
- `account_id` 是 Auth 引用，不是 User 主键。
- `avatar_file_id` 是文件引用事实，`avatarUrl` 不入库。

索引：

```sql
CREATE UNIQUE INDEX ux_users_public_id ON users(public_id);
CREATE UNIQUE INDEX ux_users_account_id ON users(account_id);
CREATE UNIQUE INDEX ux_users_nickname ON users(nickname);
CREATE INDEX ix_users_status ON users(user_status);
```

## `user_follows` schema 草案

```sql
CREATE TABLE user_follows (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  follower_id BIGINT NOT NULL,
  following_id BIGINT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL,
  UNIQUE (follower_id, following_id),
  CHECK (follower_id <> following_id)
);
```

索引：

```sql
CREATE INDEX ix_user_follows_follower_id_desc ON user_follows(follower_id, id DESC);
CREATE INDEX ix_user_follows_following_id_desc ON user_follows(following_id, id DESC);
```

规则：

- 不建 FK 到 `users`。
- `id` 只作为 cursor 锚点，不进入 HTTP。
- 关注列表按 `id DESC`。
- 注销 / 删除用户不清理 `user_follows`。

## `user_follow_stats` schema 草案

```sql
CREATE TABLE user_follow_stats (
  user_id BIGINT PRIMARY KEY,
  followers_count BIGINT NOT NULL DEFAULT 0,
  following_count BIGINT NOT NULL DEFAULT 0,
  updated_at TIMESTAMPTZ NOT NULL,
  CHECK (followers_count >= 0),
  CHECK (following_count >= 0)
);
```

规则：

- `user_follow_stats` 是可从 `user_follows` 重建的读模型。
- follow/unfollow/block 在事务内同步更新统计。
- 查询缺失统计返回零值或惰性创建。
- 删除关注时只在真实删除行后扣减，避免负数。
- 后续保留 `RebuildFollowStats(userId)` / Ops 对账入口。

## `user_blocks` schema 草案

```sql
CREATE TABLE user_blocks (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  blocker_id BIGINT NOT NULL,
  blocked_id BIGINT NOT NULL,
  reason VARCHAR(255),
  created_at TIMESTAMPTZ NOT NULL,
  UNIQUE (blocker_id, blocked_id),
  CHECK (blocker_id <> blocked_id)
);
```

索引：

```sql
CREATE INDEX ix_user_blocks_blocker_id_desc ON user_blocks(blocker_id, id DESC);
CREATE INDEX ix_user_blocks_pair ON user_blocks(blocker_id, blocked_id);
```

规则：

- 不建 FK 到 `users`。
- 拉黑列表按 `id DESC`。
- 注销 / 删除用户不清理 `user_blocks`。
- 新增拉黑会同事务解除双方关注；解除拉黑不恢复关注。

## `outbox_events`

User 本地 outbox 按项目通用 outbox 规则建表。最低字段：

```text
event_id
event_type
aggregate_type
aggregate_id
payload_version
payload_json
status
attempt_count
next_retry_at
created_at
updated_at
published_at
```

关键事件必须和业务表在同一 PostgreSQL 事务内写入 outbox。

## 缓存 key

缓存不是首批正确性依赖。

| 缓存 | Key 模式 | 失效 |
| --- | --- | --- |
| UserSimple | `user:{userId}:simple` | profile 更新、状态变更、资料删除/恢复 |
| Profile | `user:{userId}:profile` | profile 更新、状态变更 |
| publicId 解析 | `user:public:{publicId}:id` | publicId 不变，通常无需失效 |
| Availability | `user:{userId}:availability` | 状态变更 |
| Follow stats | `user:{userId}:follow_stats` | follow/unfollow/block |
| Followers list | `user:{userId}:followers` | 被关注、取消关注、block 自动解除关注 |
| Following list | `user:{userId}:following` | 关注、取消关注、block 自动解除关注 |
| Block pair | `user:block:{blockerId}:{blockedId}` | block/unblock |
| Blocked list | `user:{userId}:blocked` | block/unblock |

`avatarUrl` 不缓存进 User profile 缓存。

## 集成事件

事件 payload 使用内部 `userId`，默认不携带 `publicId`，除非 consumer 有明确展示需求。User `publicId` 和 `nickname` 展示可通过 User typed client 查询。

| 事件 | 触发用例 | 主要 payload | Consumer | Outbox |
| --- | --- | --- | --- | --- |
| `user.profile.created` | 首次 `CreateProfileForAccount` | `userId`、`accountId`、`nickname`、`avatarFileId`、`profileVersion`、`occurredAt` | Content/Search/运营读模型按需 | 必须 |
| `user.profile.updated` | 公开资料变化 | `userId`、`accountId`、`nickname`、`avatarFileId`、`bio`、`profileVersion`、`occurredAt` | Content 必需；Search/Notification 可选 | 必须 |
| `user.deactivated` | 主动注销 profile | `userId`、`accountId`、`occurredAt` | Auth 编排观测、Content/Notification 按需 | 必须 |
| `user.deleted` | 管理员资料逻辑删除 | `userId`、`operatorId`、`reason`、`occurredAt` | Content/Notification/Search 按需隐藏资料 | 必须 |
| `user.restored` | 管理员恢复资料 | `userId`、`operatorId`、`reason`、`occurredAt` | Content/Notification/Search 按需恢复展示 | 必须 |
| `user.followed` | 首次关注 | `followerId`、`followingId`、`occurredAt` | Notification | 必须 |
| `user.unfollowed` | 主动取关或 block 自动解除关注 | `followerId`、`followingId`、`reason`、`occurredAt` | 关系读模型、Notification/运营按需 | 必须 |
| `user.blocked` | 首次拉黑 | `blockerId`、`blockedId`、`reason`、`occurredAt` | Message/运营按需 | 必须 |
| `user.unblocked` | 解除拉黑 | `blockerId`、`blockedId`、`occurredAt` | Message/运营按需 | 必须 |

`user.profile.updated` 只在公开展示资料变化时发布。`strangerMessageAllowed` 更新不发布该事件。

`user.unfollowed.reason` 枚举：

- `USER_REQUEST`
- `BLOCKED`

## 跨服务一致性

### Auth

- Auth 注册同步调用 User `CreateProfileForAccount`。
- Auth 主动注销编排调用 User `DeactivateUserProfile`，再失效账号和 token。
- Auth 管理员封禁使用 `BANNED`，User 不复制该状态。

### Content

- Content 消费 `user.profile.updated` 刷新作者快照。
- 作者快照字段使用 `nickname` 口径，例如 `authorNickname` / `ownerNickname`。
- Content 不能直接读取 User 数据库。

### Comment

- Comment 写路径组合 Content 可评论校验、User availability 和 block 检查。
- Comment 查询路径通过 `BatchGetUserSimple` 获取作者摘要。
- Comment 作者摘要使用 `publicId` + `nickname` + `avatarFileId`；如需 URL，由面向前端的服务解析。

### Message

- Message 查询 `GetStrangerMessageSetting` 和 block/follow contract。
- 用户不存在或设置缺失时，陌生人私信默认返回 `false`。

### Upload / File Service

- 写头像前同步校验 `avatarFileId`。
- 前端 HTTP 读 profile 时解析 `avatarUrl`。
- URL 解析失败时不让 profile 查询整体失败，返回 `avatarFileId` 并省略或置空 `avatarUrl`。

## 数据修复和对账

首批文档预留但不要求实现：

- `RebuildFollowStats(userId)`：从 `user_follows` 重建单用户统计。
- 全量 follow stats 对账：扫描 `user_follows` 聚合 counts，与 `user_follow_stats` 比较修复。
- nickname 唯一冲突修复：只允许管理端显式资料修正，不因状态变更自动释放 nickname。
- publicId 修复：`publicId` 不应重生成；如发生冲突视为数据事故。
