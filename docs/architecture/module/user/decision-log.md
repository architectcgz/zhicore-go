# User 模块决策日志

本文件记录 `user` 模块设计压测过程中已经确认的设计问题、结论、原因和后续依赖。

| # | 决策项 | Question | Decision | Rationale | Follow-up |
| --- | --- | --- | --- | --- | --- |
| 1 | 首批模块范围 | User 模块文档是只做 Profile，还是覆盖完整上下文但首批详细切片聚焦阻塞链路？ | 覆盖完整 User 上下文边界；首批详细切片聚焦 Profile、UserSimple 和 BlockCheck。 | 这样能一次说清 Auth、Content、Comment 的依赖边界，同时避免把关注、签到、Admin 全部拉进首批实现。 | 后续文档需区分模块完整边界和首批实现切片。 |
| 2 | User 与 Auth 标识分工 | 是否直接用 Auth `accountId` 作为业务用户主键，还是保留 User 自己的 `userId`？ | 保留 User 自己的 `userId`；`accountId` 只作为 Auth 拥有的账号引用，存入 `users.account_id` 并保持唯一。 | Auth account 管认证生命周期，User ID 管业务用户生命周期；直接复用 `accountId` 会把 Auth 的禁用、登录、账号绑定和注销语义泄漏到 Content、Comment、Message 等业务表。 | User schema 和 contract 需明确 `users.id`、`users.account_id`、`users.public_id` 三者语义。 |
| 3 | User 公开标识 | `publicId` 是否由 User 生成，并只用于前端可见 contract？ | `publicId` 由 User 在 profile 初始化时生成并持久化；HTTP 面向前端默认使用 `publicId`；服务间写路径、权限、关系和持久化仍使用内部 `userId`。 | `publicId` 是展示和 URL contract，不能替代内部关系键；User typed client 可用内部 `userId` 入参并返回 `userId` + `publicId` 以支持 mapper。 | 需在 User contract、Gateway header 和 Comment 作者摘要中保持命名一致。 |
| 4 | Auth 注册初始化协议 | Auth 注册后同步初始化 User profile，还是只发布事件由 User 异步创建？ | 第一阶段 Auth 注册流程同步调用 User `CreateProfileForAccount`；User 初始化失败时不向客户端承诺完整注册成功，并登记补偿任务或 outbox。 | 同步初始化能避免注册/登录成功后 `users/me` 404；早期系统更需要用户资料立刻可用的确定性。 | 需固定 User 初始化接口、Auth 重试/补偿语义和 User 幂等规则。 |
| 5 | Profile 初始化幂等 | `CreateProfileForAccount` 遇到同一 `accountId` 已有 profile 时返回冲突还是幂等成功？ | 同一 `accountId` 重复初始化按幂等成功返回已有 profile；不覆盖已有资料，不重新生成 `publicId`，不重置 `profileVersion`。 | Auth 注册链路会因超时或补偿重试；按幂等成功可避免重复 profile 和不必要的用户可见失败。 | 需决定重复初始化是否重放 `user.profile.created` 事件。 |
| 6 | Profile 创建事件幂等 | 重复 `CreateProfileForAccount` 是否重放 `user.profile.created`？ | 不重放；只有首次真实创建 profile 时写一条 `user.profile.created` outbox。 | `user.profile.created` 表达业务事实首次发生，不是调用成功回执；重复事件会让下游读模型和审计承压。 | Consumer 仍需按 `eventId` 做消费幂等。 |
| 7 | Profile 更新事件 consumer | `user.profile.updated` 的首批下游是谁，是否让 Comment / Ranking 默认消费？ | 首批以 Content 为必需 consumer，用于刷新作者快照；Search、Notification 仅作为可选登记；Comment 默认通过 User typed client 查询作者摘要，Ranking 不拥有用户事实。 | Content 作者快照需要版本化事件抗乱序；Comment 列表查询更适合批量同步查询最新摘要；Ranking 不应复制 User 事实。 | 需继续固定 `profileVersion` 递增和事件触发字段。 |
| 8 | Profile 版本递增 | `profileVersion` 是否只绑定公开展示资料变化？ | `profileVersion` 只在公开展示资料变化时由数据库原子递增，并发布 `user.profile.updated`；无变化提交不递增；隐私设置、关注、拉黑和签到不触发该事件。 | Content 用 `profileVersion` 抵御作者快照乱序，版本含义必须保持为展示资料版本，不能被无关设置污染。 | 如非展示字段需要事件，后续定义独立事件。 |
| 9 | UserSimple 头像字段 | `BatchGetUserSimple` 是否返回 `avatarUrl`，还是只返回 User 拥有的 `avatarFileId`？ | `BatchGetUserSimple` 默认只返回 `avatarFileId`，不返回 `avatarUrl`；需要展示 URL 的调用方批量走 File service 解析。 | `avatarFileId` 是 User 保存的文件引用事实；`avatarUrl` 是 File Service 派生的临时展示值，签名、有效期和权限不归 User。 | User 自己的前端 HTTP 查询是否内联 URL 后续单独决定。 |
| 10 | UserSimple 缺失和状态语义 | `BatchGetUserSimple` 是否因部分缺失或 Auth 状态不可用整体失败？ | 缺失 User profile 时批量接口不整体失败，结果省略并返回 `missingUserIds`；UserSimple 不实时调用 Auth 判断账号状态；写路径资格校验使用专门 guard。 | 批量展示查询必须稳定，不能因个别脏数据或 Auth 短暂不可用打崩整页；写路径需要严格校验互动资格，不能复用展示摘要。 | 需设计写路径用户状态和关系 guard。 |
| 11 | 拉黑事实源和首批范围 | User 是否只提供拉黑查询 contract，还是首批同时实现拉黑管理 API？ | User 拥有 `user_blocks` 作为拉黑事实源；首批包含 `BlockUser`、`UnblockUser`、`ListBlockedUsers` 和 `BatchCheckBlocked`，Comment/Message 只调用 User contract，不直接写表。 | 拉黑 API 本身复杂度不高，且 Comment 写路径依赖拉黑事实；首批纳入可以避免只有查询 contract、没有事实维护入口的半成品边界。 | 需决定拉黑是否自动解除关注关系，以及幂等语义。 |
| 12 | 拉黑与关注关系 | 拉黑时是否自动解除双方关注关系？ | 拉黑时在同一事务内写入 `user_blocks`，并删除 `blocker -> blocked`、`blocked -> blocker` 两个方向的关注关系及修正统计；解除拉黑不恢复关注。 | 已拉黑但仍互相关注会让产品和权限语义冲突；把解除关注放在 User 本地事务内能保持关系事实一致。 | 需固定关注/取关事件和统计修正语义。 |
| 13 | 关注 API 首批设计 | 关注/取关 API 是否纳入首批 User 模块设计？ | 纳入首批设计和 contract，但实现优先级排在 Profile + UserSimple、Block + BatchCheckBlocked 之后。 | Block 已经需要修改 follow 事实；如果不定义 follow owner 和统计规则，block 的事务副作用会悬空。 | 需固定关注/取关幂等、事件和统计语义。 |
| 14 | 关系命令幂等 | 关注/取关、拉黑/解除拉黑重复调用返回冲突还是幂等成功？ | 关系命令重复调用按幂等成功处理；统计和事件只在真实状态变化时更新或发布；自操作返回 400，任一方向拉黑时关注返回 403。 | 关注和拉黑按钮容易重复点击或因网络重试重复提交；幂等成功可减少用户可见失败，同时避免重复事件和统计漂移。 | HTTP contract 需明确 status、body code 和事件触发条件。 |
| 15 | Check-in 首批范围 | 签到 Check-in 是否进入首批实现和 HTTP contract 细化？ | Check-in 属于 User 完整边界，但不进入首批实现或 contract 细化；首批只标注子域和后续 use case。 | Check-in 不阻塞 Auth、Content、Comment 链路，且会引入业务日期、时区、连续天数、Redis bitmap 和月历查询，容易扩大本轮范围。 | 后续专门做 Check-in 时再压事务、时区和缓存边界。 |
| 16 | 管理端归属 | User 管理端资料查询/修正和 Auth 账号状态管理如何分工，是否进入首批？ | User 拥有资料相关管理查询/修正能力；Auth 拥有账号禁用、启用、角色调整和 token 失效能力；二者本轮只登记边界，不进入 User 首批实现细化。 | 管理端会引入权限、审计、账号状态和资料修正；它不阻塞 Auth、Content、Comment 核心链路，且账号状态不应回流到 User。 | 后续 Admin facade 设计需分别委托 User 和 Auth contract。 |
| 17 | User API 分层 | User 首批 API 是否区分前端公开 API、服务间 typed client/internal API 和 Admin API？ | 区分三类：前端公开 API 使用 `publicId`；服务间/internal API 使用内部 `userId` 和 `accountId`；Admin API 本轮只登记边界不提取字段级 endpoint。 | 前端不应看到内部 `userId`，服务间关系判断不应使用 `publicId` 做权限和关系键，Admin 能力不应污染首批核心链路。 | 需在 HTTP contract 和 typed client contract 中分别固定路径、字段和 ID 语义。 |
| 18 | PublicID 解析边界 | 前端 `publicId` 路径下的目标用户解析由 Gateway 还是 User 负责？ | Gateway 不解析业务目标 `publicId`；User 在 application/repository 内解析 `publicId -> userId` 并执行业务规则；关系表只保存内部 `userId`。 | Gateway 应保持薄入口，只负责认证、清理 header、注入当前操作者和路由；目标资源解析、not-found 和关系规则属于 User 业务逻辑。 | User HTTP contract 需明确 path 用 `publicId`，服务间 contract 默认用内部 `userId`。 |
| 19 | Avatar URL 响应边界 | User 面向前端的 HTTP 响应是否返回 `avatarUrl`，以及它是否进入持久化、事件和 typed client？ | User 前端 HTTP 响应应由后端内部解析并返回 `avatarUrl`；`avatarUrl` 只作为派生展示字段，不落库、不进事件、不进默认 typed client。 | 前端需要可展示 URL；但 URL 的签名、有效期和权限归 File service，不能污染 User 事实、事件和内部 contract。 | HTTP contract 需说明 URL 解析失败时返回 `avatarFileId` 并省略或置空 `avatarUrl`。 |
| 20 | 写路径互动 Guard | User 是否提供统一互动资格 contract，还是让调用方复用 UserSimple 判断？ | 首批提供窄 guard：`BatchCheckBlocked` 和 `BatchGetUserAvailability`；不让 `BatchGetUserSimple` 承担写路径资格判断，也不复制 Auth 账号状态。 | User 能回答资料存在性和关系事实，但 Auth 禁用/锁定仍归 Auth/Gateway；展示摘要不能作为写路径权限依据。 | Comment 写路径需组合 Content 可评论校验、User availability 和 block 检查。 |
| 21 | 关系事件 outbox | User 关系事件如何发布，block 自动解除关注时是否发布 `user.unfollowed`？ | 真实状态变化都写 User outbox；block 自动解除关注时发布对应 `user.unfollowed`，并在 payload 中标明 `reason=BLOCKED`。 | 下游关系读模型只消费 follow/unfollow 事件也能收敛，不需要理解 block 内部副作用；事件和本地关系/统计必须事务一致。 | 需在事件 contract 中固定 reason 枚举和幂等字段。 |
| 22 | User 缓存边界 | UserSimple、publicId 解析、block 检查和 follow stats 是否作为首批正确性依赖？ | 文档定义缓存 key 和失效规则，但首批实现可以先 cache-aside 可选；关系权限类检查优先直接查 PostgreSQL，不把缓存作为正确性依赖。 | block/follow 权限缓存失效不严谨会影响 Comment/Message 权限；首批以 PostgreSQL 为真相源更稳。 | 后续实现缓存时需补失效测试和权限延迟说明。 |
| 23 | Profile 初始化默认值 | `CreateProfileForAccount` 初始化 profile 时默认资料从哪里来，陌生人私信默认是否允许？ | Auth 调用 User 时传入 `accountId`、`username` 和可选 `email`；User 用 `username` 初始化 `displayName/nickname`，`avatarFileId` 为空，`bio` 为空字符串；`strangerMessageAllowed` 默认 `true`；User 不保存 email。 | User 初始化只需要生成展示资料，不应复制 Auth 登录标识和隐私字段；陌生人私信按产品默认允许。 | HTTP schema 需固定空字符串和默认布尔值语义。 |
| 24 | Profile 更新字段 | 用户资料更新允许改哪些字段，是否允许普通用户改 `publicId`？ | 首批只允许更新 `displayName/nickname`、`avatarFileId`、`bio`、`strangerMessageAllowed`；不允许修改 `publicId`、`accountId`、`userId`、`profileVersion`；`bio` 最大 100。 | `publicId` 进入 URL 和外部 contract 后不应由普通资料编辑接口改变；公开资料变化和设置变化需要分开处理事件。 | 需固定头像文件校验和 `strangerMessageAllowed` 事件语义。 |
| 25 | 头像文件校验 | 更新 `avatarFileId` 时是否同步校验 File service，旧头像是否自动删除？ | 非空 `avatarFileId` 写入前必须同步校验文件存在、类型为图片且状态可引用；校验失败不写 profile、不递增版本、不发事件；旧头像首批不自动删除。 | 头像是公开展示资料，脏 fileId 会污染作者快照和前端展示；自动删除旧头像可能误删仍被其他资源引用的文件。 | 后续如需头像清理，单独设计引用关系和补偿策略。 |
| 26 | 关系列表分页锚点 | 关注、粉丝和拉黑列表使用 cursor 分页时是否为关系表增加内部递增 `id`？ | 关系列表使用 cursor 分页；`user_follows` 和 `user_blocks` 增加不对外暴露的单调递增内部 `id` 作为 cursor 锚点，列表按 `id DESC` 稳定排序。 | 关系创建时间排序需要稳定唯一锚点；内部行 ID 比 `(created_at, user_id)` 组合 cursor 更简单，且不暴露给 HTTP。 | schema 和索引需覆盖 followers、following、blocked 列表查询。 |
| 27 | User 错误语义 | User 服务首批公开错误码如何固定，是否用 HTTP status 作为 body code？ | 先固定稳定 symbolic code 和 HTTP status，不用 HTTP status 充当 body code；包括 `USER_NOT_FOUND`、`USER_DISPLAY_NAME_INVALID`、`USER_BIO_TOO_LONG`、`USER_AVATAR_INVALID`、`USER_CANNOT_FOLLOW_SELF`、`USER_CANNOT_BLOCK_SELF`、`USER_INTERACTION_BLOCKED`、`USER_CURSOR_INVALID`。 | 错误语义会进入前端和 contract test，必须稳定；数字码可后续登记到项目错误码表，但 body code 不能退化成 HTTP status。 | 提取 User HTTP schema 时同步登记错误码数字或项目范围。 |
| 28 | User 核心 schema | User 首批核心表和约束如何定义，是否使用数据库 FK？ | 首批核心表为 `users`、`user_follows`、`user_follow_stats`、`user_blocks` 和 User 本地 `outbox_events`；`users.public_id`、`users.account_id`、关系自然键使用唯一约束；不建跨服务 FK，也不建 User 本地表之间的 FK，关系完整性由业务事务维护。 | 服务边界不能依赖跨库 FK；本地也不建 FK 可降低迁移和删除/修复耦合，但必须由 application/repository、唯一约束、计数非负保护和测试承担一致性责任。 | data-events 文档需列出业务维护的不变量、唯一索引和统计修正规则。 |
| 29 | Follow stats 事实源 | `user_follow_stats` 是否作为强事实，是否需要对账/重建设计？ | `user_follow_stats` 是可从 `user_follows` 重建的读模型；关系命令事务内同步更新统计，首批文档预留重建/对账设计，查询缺失统计返回零值或惰性创建。 | 关注统计是展示字段，可能因并发、补偿或历史数据漂移；提前声明事实源可避免后续把 stats 当强事实。 | Ops 对账入口和重建实现可后置，但 repository 更新必须防负数。 |
| 30 | 陌生人私信设置 | `strangerMessageAllowed` 的更新、事件和 Message 查询缺失语义如何定义？ | 字段归 User profile/settings；更新不触发 `user.profile.updated`；Message 通过 User typed client 查询；正常新用户默认 `true`，用户不存在或设置缺失时给 Message 返回 `false`。 | 产品默认允许和下游缺失容错要分开；查不到用户时默认放行会放大脏数据和权限风险。 | 如 Message 需要本地缓存，后续再定义 `user.message_settings.updated`。 |
| 31 | 用户注销/删除首批范围 | 用户注销和删除是否进入首批，采用物理删除还是逻辑状态？ | 用户注销和删除进入首批，采用逻辑状态变更，不物理删除 User profile 和关系事实。 | 注销/删除是用户生命周期核心能力；逻辑状态能保留历史内容引用、审计和跨服务稳定标识，避免破坏 Content/Comment 作者引用。 | 需继续固定状态枚举、对外展示、关系处理、事件和 Auth 分工。 |
| 32 | 用户状态模型 | User 逻辑注销/删除使用哪些状态，非 ACTIVE 如何展示？ | `users.user_status` 使用 `ACTIVE`、`DEACTIVATED`、`DELETED`；非 ACTIVE 禁止资料更新、关注、拉黑、签到等写操作；UserSimple 可返回最小摘要和状态，前端主页对 `DELETED` 建议 404。 | 写路径需要严格禁用非 ACTIVE 用户，读路径仍要让历史内容和评论稳定渲染。 | 需决定注销/删除时是否清理关系，以及是否发布 profile/status 事件。 |
| 33 | 注销/删除后的关系保留 | 用户注销/删除时是否清理 `user_follows` 和 `user_blocks`？ | 不清理 `user_follows` 和 `user_blocks`；注销/删除只改变 `users.user_status`，关系列表和 UserSimple 返回用户状态或占位展示。 | 保留关系事实更利于恢复、审计和关系历史；历史关注列表显示“用户已注销/已删除”即可，不需要破坏关系表。 | 查询层需过滤或展示非 ACTIVE 用户状态，写路径需禁止与非 ACTIVE 用户建立新关系。 |
| 34 | 非 ACTIVE 关系写操作 | 非 ACTIVE 用户作为 actor 或 target 时，关注/拉黑/解除关系如何处理？ | actor 非 ACTIVE 禁止所有 User 写操作；新增关注/拉黑要求目标用户 ACTIVE；取关和解除拉黑允许清理历史关系，即使目标已注销或删除。 | 新增关系必须要求双方可用，避免对不可用用户产生新互动；解除关系是用户清理历史状态，应保持可用。 | HTTP schema 需区分新增关系失败和解除关系幂等成功。 |
