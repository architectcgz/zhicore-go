# Ops 页面设计

本文记录 Ops 相关内部运维页面的前端草稿、页面分区、加载状态和降级规则。Ops 不是普通业务服务；页面只服务迁移、对账、补偿、事件重放和修复任务。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- Ops 页面默认内部可见，所有写操作必须有权限、原因、预览、审计和可恢复状态。
- Ops 不拥有业务事实，修复业务数据时优先通过归属服务 contract；只有受控 repair 任务才可执行低层修复。
- 灰度发布接口当前不迁移，页面不设计 Java/Go 并存灰度操作。
- 对账和修复任务必须展示输入范围、影响预估、dry-run 结果、执行状态、失败项和重试策略。
- 任务页面优先表格、状态机、日志摘要和 trace 链接，不做装饰性仪表盘。

## 页面范围

本文覆盖：

- Ops 总览。
- 对账任务。
- 修复 / 补偿任务。
- 事件回放任务。
- Worker / queue 运行状态。
- 任务详情和审计。

本文不覆盖：

- Admin 审核处理页面。
- Gateway 路由 / 认证诊断页面。
- Ranking rebuild 专属页面，除非以 Ops 总览链接方式进入。

## Ops 总览

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Ops workspace                              │
├────────────────────────────────────────────┤
│ Task summary                               │
│ running · failed · pending · recent fixed  │
├────────────────────────────────────────────┤
│ Dependency health                          │
│ db · redis · mq · downstream services      │
├────────────────────────────────────────────┤
│ Recent operations                          │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 进入页面确认运维权限。
2. 加载任务摘要、依赖健康和近期操作。
3. 任一面板失败时局部降级。
4. 没有写权限的用户只读展示，隐藏启动 / 重试按钮。

## 对账任务

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Reconciliation                             │
├────────────────────────────────────────────┤
│ Form: service · object range · dry-run     │
│ [Preview] [Start task]                     │
├────────────────────────────────────────────┤
│ Result preview                             │
│ mismatch · missing · extra · suggested fix │
├────────────────────────────────────────────┤
│ Task status · progress · failed items      │
└────────────────────────────────────────────┘
```

规则：

- `Preview` 只读，展示影响范围和样例差异。
- `Start task` 必须要求原因和确认。
- 任务运行中展示进度、失败项和可重试状态。
- 对账发现不等于自动修复；修复需要单独动作或明确勾选。

## 修复 / 补偿任务

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Repair task                                │
├────────────────────────────────────────────┤
│ Scope · reason · risk note · dry-run diff  │
├────────────────────────────────────────────┤
│ Execution                                  │
│ pending · running · succeeded · failed     │
├────────────────────────────────────────────┤
│ Failed items · retry · export              │
└────────────────────────────────────────────┘
```

页面必须展示：

- 修复目标归属服务。
- 是否通过归属服务 contract。
- 是否存在直接数据修复。
- 幂等键或任务 ID。
- 回滚或补偿说明。

直接数据修复如果不可避免，页面必须标出高风险，并要求更高权限和审计。

## 事件回放

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Event replay                               │
├────────────────────────────────────────────┤
│ Source: outbox · DLQ · event id range      │
│ target consumer · partition · dry-run      │
├────────────────────────────────────────────┤
│ Replay status                              │
│ produced · skipped · failed · duplicated   │
└────────────────────────────────────────────┘
```

规则：

- 回放前必须展示目标 consumer、事件范围和幂等策略。
- 回放不绕过 consumer 幂等。
- DLQ 重放失败项需要保留原始 `eventId` 和 request trace，但页面不展示敏感 payload。
- 有序事件族必须遵守事件 contract 的分区规则，不能在 UI 中按 worker 数重路由。

## Worker / Queue 状态

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Workers / queues                           │
├────────────────────────────────────────────┤
│ service · queue · lag · active consumer    │
│ partition count · retry · dlq              │
├────────────────────────────────────────────┤
│ Detail · ack mode · last error             │
└────────────────────────────────────────────┘
```

规则：

- 显示逻辑分片数、active consumer、lag、retry 和 DLQ。
- 对有序事件族，强调 `partitionCount` 与 worker 实例数不同。
- 不提供“改 worker 数后重新分配 key”的 UI 操作。

## 跨服务页面约定

- Admin 页面可链接到 Ops 任务详情，但 Ops 任务状态仍归 Ops。
- Ops 修复 Content / Comment / User / Ranking 等数据时，页面必须显示归属服务和修复边界。
- Gateway 诊断面板可提供入口到 Ops 依赖健康，但 Gateway 不拥有 Ops 任务。
