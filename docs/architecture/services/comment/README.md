# Comment 服务设计

## 事实来源

- Java `zhicore-comment` controller：`CommentCommandController`、`CommentQueryController`、`CommentLikeCommandController`、`CommentLikeQueryController`、`CommentMediaController`、`AdminComment*Controller`、`CommentOutboxAdminController`。
- Java `zhicore-comment` domain model：`Comment`、`CommentStats`、`CommentLike`、`CommentStatus`、`OutboxEvent`。
- Java `zhicore-comment` application service：评论命令、评论查询、点赞、媒体上传、首页缓存、热门候选同步和 outbox 管理。
- Java `database/init-all-databases.sql` 和 `zhicore-comment/src/main/resources/db/schema.sql` 中 comment 表。
- Ranking、Content、Notification 对评论事件的消费设计。

## 职责边界

`zhicore-comment` 拥有评论、回复、评论树、评论状态、评论统计、评论点赞、评论媒体引用和 Comment 自己的 outbox 事件。

Comment 不拥有文章、用户资料、文件存储、通知收件箱或榜单分数。

## DDD 目标设计

Comment 是独立限界上下文。统一语言以“评论、根评论、回复、评论树、评论状态、评论内容、评论媒体、评论点赞、评论统计、游标分页、增量补拉、管理删除、评论事件”为核心，不把 Content、User、Upload、Ranking 或 Notification 的模型引入 Comment 领域层。

DDD 设计用于指导 Go 目标实现，不表示当前 Go 代码已经完成。Java 侧已有领域模型、查询服务、缓存、outbox 和游标编解码是事实来源之一，但 Go 实现按本仓库 `api/http -> application -> domain/ports -> infrastructure` 的依赖方向重新落点。

### 限界上下文与子域

Comment 上下文内按职责拆成以下子域：

| 子域 | 职责 | 主要存储 |
| --- | --- | --- |
| Comment Tree | 创建根评论、创建回复、维护 `rootId` / `parentId`、删除评论和评论状态 | `comments` |
| Comment Content / Media | 评论文本、图片文件引用、语音文件引用、语音时长和内容校验 | `comments.image_ids`、`comments.voice_id`、`comments.voice_duration` |
| Comment Interaction | 点赞、取消点赞和用户点赞状态 | `comment_likes` |
| Comment Stats | 评论点赞数、回复数和统计修正 | `comment_stats` |
| Comment Query | 详情、文章评论分页、回复分页、游标分页、增量补拉和管理端查询 | `comments`、`comment_stats`、Redis cache |
| Homepage / Hot Candidate | 首页评论缓存和热门文章候选输入；第一阶段由 Comment 定时拉取 Ranking 候选并替换本地缓存 | Redis cache、Ranking contract |
| Integration | 评论事件、outbox 派发、dead retry、缓存失效和幂等 | Comment `outbox_events` |

### 聚合

#### `Comment` 聚合

`Comment` 是评论树的聚合根，负责维护评论本身的强一致生命周期。

Comment 是本轮明确重做的服务，不要求兼容 Java 的外部 `commentId` API。外部评论定位使用 `(postId, floor)`，内部仍使用 `comments.id`。

- **内部标识**：`CommentID` 使用 Comment 服务数据库生成的 `BIGINT GENERATED BY DEFAULT AS IDENTITY` 或 sequence。它只用于 Comment 内部关联、事务、点赞、统计、缓存和事件，不作为对外资源 ID。
- **外部定位**：`CommentFloor` 是文章内单调递增楼层号，对外用 `(postId, floor)` 定位评论，例如 `/posts/{postId}/comments/{floor}`。路径中的 `postId` 是 Content 的对外文章 ID；Comment 内部保存解析后的 `post_id BIGINT` 引用。根评论和回复共享同一篇文章内的 `floor` 序列；`floor` 不是根评论下的回复序号，不重排、不复用，不为 Comment 额外设计 `public_id`。
- **归属引用**：`PostID` 引用 Content 拥有的文章；`AuthorID` 引用 User 拥有的用户。
- **树结构**：`RootID` 指向所属根评论；根评论 `root_id = NULL`。`ParentID` 指向直接被回复的评论；根评论 `parent_id = NULL`。被回复用户通过 `parent_id -> parent.author_id` 查询，不额外保存 `reply_to_user_id`。
- **内容字段**：文本、图片文件 ID 集合、语音文件 ID、语音时长。
- **状态字段**：`Normal`、`Deleted`。
- **行为**：`Edit`、`Delete`、`MarkDeletedByAdmin`、`RecordDeletionMetadata`、`IsTopLevel`、`IsReply`、`AssertEditableBy`、`AssertDeletableBy`。
- **领域事件**：`CommentCreated`、`CommentUpdated`、`CommentDeleted`。

`Comment` 聚合只放评论树和评论内容的强一致规则。点赞数、回复数等高频计数属于独立的 `CommentStats` 聚合，避免点赞或回复高频写入锁住评论聚合本体。

`Comment` 的核心不变量：

- 评论必须归属于一个有效 `PostID`，但文章是否存在、是否可评论和是否可见由 application 通过 Content contract 校验。
- 作者是否存在、是否允许互动、是否被拉黑由 application 通过 User contract 校验。
- 评论必须至少包含文本、图片或语音中的一项。
- 文本长度第一阶段保留 Java 约束，最多 2000 字。
- 图片最多 9 张。
- 语音评论不能同时包含图片；语音时长上限按 Java 上传和 DTO 约束迁移，当前文档记录为 60 秒候选值，字段级 contract 提取时必须核对最终配置。
- 已删除评论不能编辑、点赞或作为新回复目标。
- 普通用户只能编辑和删除自己的评论；管理员删除通过 Admin facade 或 Comment admin API 委托到 Comment mutation。
- 删除顶级评论时，回复处理策略固定为“软删除根评论及其全部回复”。实现位置固定为 `DeleteComment` / `AdminDeleteComment` use case 编排：application 先加载目标评论并调用 domain 行为完成权限和状态校验，再在同一事务内调用 repository 的批量软删除方法标记回复。repository 只执行持久化批量更新，不自行决定业务级联规则；domain 不访问 repository。
- 管理员删除需要携带 `deletedBy` 和可选 `deleteReason`。Admin 服务仍拥有完整审核审计日志；Comment 只保存执行删除所需的最小删除元数据，并在 `comment.deleted` 事件中携带这些字段，方便 Content、Notification、Ranking 和审计读模型理解删除来源。

#### `CommentStats` 聚合

`CommentStats` 是独立统计聚合根：

- **标识**：`CommentID`，与评论一对一。
- **统计字段**：`LikeCount`、`ReplyCount`。
- **行为**：`IncrementLikes`、`DecrementLikes`、`IncrementReplies`、`DecrementReplies`、`RebuildFromQuery`。
- **不变量**：所有计数不能为负数。`RebuildFromQuery` 用于从 `comment_likes` 和 `comments` 表重建 `like_count` / `reply_count`，修正重复事件、补偿失败、历史迁移或人工修复导致的计数漂移。

为什么 `CommentStats` 独立于 `Comment`：

- 点赞和回复是高频写入，如果每次都修改 `Comment` 聚合，会造成评论本体锁冲突。
- 统计计数可以通过事件或回源重建修正，不要求和评论内容同一聚合强一致。
- 点赞事务只需要修改 `comment_likes`、`comment_stats` 和 outbox，不应该递增 `Comment` 聚合版本。

#### `CommentLike`

`CommentLike` 是以 `(CommentID, UserID)` 为自然唯一键的互动关系实体，不作为独立聚合根暴露生命周期。

点赞和取消点赞由 application 在同一 PostgreSQL 事务内完成：

```text
comment_likes 插入/删除
+ comment_stats.like_count 原子增减
+ outbox_events(comment.liked / comment.unliked)
```

Redis 点赞状态和计数缓存只在事务提交后 best-effort 更新或删除，失败不回滚业务事务。

#### `CommentMedia`

评论媒体是 `Comment` 聚合内的值对象或子对象：

- `ImageFileIDs` 保存 Upload / File Service 拥有的图片文件引用。
- `VoiceFileID` 保存 Upload / File Service 拥有的语音文件引用。
- `VoiceDuration` 保存语音时长。

Comment 只拥有媒体引用和评论内容规则，不拥有文件元数据、对象存储路径、文件 URL 或文件删除事实。评论媒体上传入口如果保留，只作为业务便捷 facade，内部转发到 Upload/File Service，文件归属不转移。新前端或新 API 可以直接调用 Upload 服务；是否保留 Comment 媒体 facade 由目标 HTTP contract 单独决定。

#### 基础设施对象和非聚合对象

以下对象不建成领域聚合：

- `outbox_events`：跨服务可靠消息机制，不是 Comment 领域对象。
- outbox summary、dead retry、metrics：管理和运维视图，不进入 domain。
- 首页评论缓存、详情缓存、点赞缓存：Redis 读模型，不是真相源。
- Ranking hot candidate：Ranking 拥有榜单和分数，Comment 只可以读取热门候选作为缓存或物化输入。
- Sentinel / 限流配置：运行时保护机制，属于 infrastructure 或 middleware。

### 值对象

Comment 领域层优先用值对象表达有业务含义的基础值，避免业务规则里到处传裸 `int64` / `string`。

| 值对象 | 含义 |
| --- | --- |
| `CommentID` | 评论内部标识 |
| `CommentFloor` | 文章内楼层号，用于外部定位和楼层展示 |
| `PostID` | Content 拥有的文章引用 |
| `UserID`、`AuthorID` | User 拥有的用户引用 |
| `RootCommentID` | 根评论标识 |
| `ParentCommentID` | 直接被回复的评论标识 |
| `CommentContent` | 评论文本，封装非空、长度和空白规则 |
| `ImageFileIDs` | 评论图片文件引用集合，最多 9 张 |
| `VoiceFileID`、`VoiceDuration` | 评论语音引用和时长 |
| `CommentStatus` | 评论生命周期状态 |
| `DeletionMetadata` | 删除操作者、删除原因和删除时间；完整审计仍归 Admin |
| `CommentStatsValue` | 点赞数和回复数的非负计数值 |
| `CommentSortType` | `TIME`、`HOT` 等排序方式 |
| `CursorToken` | 不透明游标，隐藏内部排序字段 |
| `AuthorSnapshot` | 评论展示使用的作者摘要快照或查询 DTO，不是 User 源数据 |

核心约束：

| 值对象 | 约束 |
| --- | --- |
| `CommentContent` | 评论整体必须至少包含文本、图片或语音；文本最多 2000 字 |
| `CommentFloor` | 在同一 `post_id` 下唯一，根评论和回复共享同一序列；删除后不复用、不重排；不是根评论下的回复序号 |
| `ImageFileIDs` | 最多 9 张；只保存文件 ID，不保存文件事实 |
| `VoiceFileID` | 不能与图片同时出现 |
| `VoiceDuration` | 必须为正数；最大值按 Java 上传配置和字段级 contract 核对 |
| `CursorToken` | 对外不透明，不能暴露内部 SQL 排序列语义 |

游标第一阶段采用简单不透明 token：

- `TIME` 游标使用 URL-safe Base64 无 padding 编码，原文为 `ISO-8601 OffsetDateTime + "_" + internalCommentId`。
- `HOT` 游标使用 URL-safe Base64 无 padding 编码，原文为 `likeCount + "_" + internalCommentId`。
- 游标不加密、不签名，只隐藏内部实现细节，不提供安全防篡改能力；解码失败按参数错误处理。
- 如果后续要给游标增加签名或版本字段，必须作为 API 演进处理，旧游标在一个过渡期内仍应可解析。

### 领域服务

领域服务只承载纯业务规则，不依赖数据库、Redis、HTTP client、MQ 或外部 SDK。

| 领域服务 | 职责 |
| --- | --- |
| `CommentContentPolicy` | 校验文本、图片、语音组合规则和长度限制 |
| `CommentTreePolicy` | 判断根评论、回复、被回复评论之间的树结构是否合法 |
| `CommentPermissionPolicy` | 判断普通作者或管理员是否可编辑、删除评论；只基于已加载聚合和操作者上下文 |
| `CommentInteractionPolicy` | 判断评论状态是否允许点赞、取消点赞或回复 |
| `CommentSortPolicy` | 解释 `TIME` / `HOT` 等排序语义，但不生成 SQL |

基础设施检查由 application 编排：

- 文章存在性、可见性和是否允许评论：通过 Content contract。
- 用户存在性、状态、拉黑关系和互动权限：通过 User contract。
- 文件是否存在、URL 解析、上传和删除：通过 Upload/File Service adapter。
- 游标解析、SQL 查询、缓存读写：通过 ports 和 infrastructure。

### 工厂

工厂负责封装复杂聚合创建逻辑，确保创建时满足基础不变量。

**`CommentFactory`**：

- `CreateTopLevel(postID, floor, authorID, content, media)`：创建根评论，`RootID` 和 `ParentID` 均为空。
- `CreateReply(postID, floor, authorID, rootComment, parentComment, content, media)`：创建回复，验证根评论和直接父评论存在且未删除，根评论确实是顶级评论，父评论属于同一根评论树，三者属于同一文章，并设置 `RootID` 和 `ParentID`。

Go 第一阶段不强制让工厂生成 ID。内部主键由数据库生成时，application 可以先创建无 ID 的领域对象，再由 repository 保存后返回 `CommentID`，并在同一事务中写统计和 outbox。

### Application 用例

Comment application 层按命令、查询分层组织 use case。application 拥有事务边界、权限上下文、幂等、端口调用、缓存失效和错误映射。

**命令用例（Commands）**：

- `CreateComment`：校验文章和作者事实，分配文章内 `floor`，创建根评论或回复，写 `comments`、初始化 `comment_stats`、更新根评论回复数，写 `comment.created` outbox。创建回复时，application/repository 必须在同一事务内校验根评论是顶级评论、父评论属于同一文章、父评论未删除且属于同一根评论树；这些跨行和带状态的规则不依赖数据库 `CHECK` 约束表达。
- `UpdateComment`：校验作者权限和状态，更新评论内容，事务后删除详情和列表缓存。
- `DeleteComment`：普通用户删除自己的评论；顶级评论删除时由 use case 在同一事务内编排批量软删除回复，写 `comment.deleted` outbox，事务后清理缓存。
- `AdminDeleteComment`：管理员删除评论，由 Admin facade 或 Comment admin API 委托到 Comment mutation，必须携带 `deletedBy`，可携带 `deleteReason`。Comment 保存最小删除元数据，Admin 继续记录完整审核审计。
- `LikeComment` / `UnlikeComment`：维护点赞关系、`CommentStats` 和 `comment.liked` / `comment.unliked` outbox。
- `UploadCommentImage` / `UploadCommentVoice`：作为 Upload/File Service adapter 的 HTTP facade。该 API 不表示 Comment 拥有文件事实；新前端或新 API 可以直接调用 Upload 服务，是否保留该 facade 由目标 HTTP contract 单独决定。
- `RetryDeadOutboxEvents`：管理端把 DEAD 事件重置为 PENDING，记录操作者和审计字段。
- `SyncRankingHotCandidates`：从 Ranking contract 同步热门文章候选到 Comment 本地缓存或物化输入。

**查询用例（Queries）**：

- `GetCommentDetail`：返回当前评论的 `floor`；如果是回复，同时返回 `rootFloor` 和 `parentFloor` 供前端展开根评论、定位父评论和高亮目标回复。`rootFloor` / `parentFloor` 是查询 DTO 字段，不是 `comments` 表冗余列。
- `ListTopLevelCommentsByPage`：传统 page/size 分页，支持 `TIME` / `HOT` 排序。
- `ListTopLevelCommentsByCursor`：游标分页，支持 `TIME` / `HOT` 排序。
- `ListRepliesByPage`。
- `ListRepliesByCursor`。
- `ListTopLevelCommentsIncremental`：按 `afterCreatedAt`、`afterId` 做增量补拉。
- `ListRepliesIncremental`。
- `GetLikeStatus`、`BatchGetLikeStatus`、`GetLikeCount`。
- `ListAdminComments`：按 keyword、postId、userId 筛选。
- `GetOutboxSummary`。

查询用例可以返回 DTO 或视图模型，不把复杂列表查询塞进 domain。评论列表展示需要作者摘要时，优先批量调用 User contract 或使用本地快照；不得直接读取 User 数据库。

### 一致性与事务边界

**创建根评论事务**：

```text
单个 PostgreSQL 事务：
  comment_post_counters 分配 floor
  + comments 插入根评论
  + comment_stats 初始化
  + outbox_events(comment.created)

事务提交后：
  删除文章评论列表缓存
  best-effort 更新首页评论缓存
```

**创建回复事务**：

```text
单个 PostgreSQL 事务：
  comment_post_counters 分配 floor
  + comments 插入回复
  + comment_stats 初始化回复统计
  + comment_stats(root_comment).reply_count 原子 +1
  + outbox_events(comment.created)

事务提交后：
  删除根评论回复列表缓存
  删除文章评论列表缓存中对应根评论统计
```

`floor` 分配必须使用 `comment_post_counters` 或等价的事务内计数器，不使用 `SELECT max(floor) + 1`。推荐实现：

```sql
UPDATE comment_post_counters
SET next_floor = next_floor + 1
WHERE post_id = ?
RETURNING next_floor - 1 AS floor;
```

首次评论可用 UPSERT 初始化 counter。`floor` 一旦分配即使评论删除也不复用。

**更新评论事务**：

```text
comments 更新内容和更新时间
```

事务提交后删除评论详情、文章评论列表和回复列表缓存。更新评论是否发布跨服务事件取决于实际 consumer；第一阶段如果没有明确 consumer，可以只做内部缓存失效，不发布 `comment.updated` 集成事件。

**删除评论事务**：

```text
comments 标记目标评论 deleted
+ 如果目标是顶级评论，批量标记所有回复 deleted
+ 如果目标是回复，comment_stats(root_comment).reply_count 原子 -1
+ outbox_events(comment.deleted，包含 affectedCount)
```

顶级评论批量删除回复时，第一阶段发布一条 `comment.deleted` 集成事件，并用 `affectedCount` 表示本次软删除影响的评论总数，便于 Content 修正文章评论数。payload 还必须能让 consumer 区分删除的是根评论还是回复，以及影响的 `postId`。是否为每条回复发布独立事件，后续只在 Notification/Ranking 明确需要逐条明细时再扩展。

**点赞/取消点赞事务**：

```text
comment_likes 插入/删除
+ comment_stats.like_count 原子增减
+ outbox_events(comment.liked / comment.unliked)
```

点赞和取消点赞的重复操作按目标 contract 固定：建议已经点赞再次点赞、未点赞取消点赞返回成功且不重复写事件。

**媒体上传**：

上传图片和语音通过 Upload/File Service adapter 完成，不写 Comment 真相源。评论创建时只保存 `file_id` 引用。Comment 媒体上传 API 如保留也只是 facade；新调用方优先使用 Upload 服务。上传成功但评论创建失败时，是否删除临时文件属于事务后补偿策略，不进入 `Comment` 聚合。

**outbox 管理**：

`RetryDeadOutboxEvents` 只改变 Comment 自己的 outbox 状态，不直接发送业务事件。dispatcher 负责 claim pending、发布 RabbitMQ、更新发送状态和 retry/dead 状态。

**outbox envelope 归属**：

`outbox_events` 是 Comment 服务私有可靠消息表，但表中必须保存足够的事件 contract 元数据。推荐 schema 至少包含 `event_id`、`event_type`、`payload_version`、`producer`、`aggregate_type`、`aggregate_id`、`occurred_at`、`payload`、`status`、`retry_count`、`next_retry_at`、`claimed_by`、`claimed_at`、`created_at`、`sent_at` 和 `error_message`。`event_id` 同时就是 RabbitMQ envelope 的 `eventId`，由 application/outbox writer 在业务事务内生成并落库，dispatcher 不生成新的业务事件 ID。

发布到 RabbitMQ 时有两种等价实现：

1. outbox 表直接保存完整 envelope JSON，dispatcher 原样发布。
2. outbox 表分列保存 envelope metadata 和 payload，dispatcher 在发布前组装统一 envelope。

无论选择哪种实现，对外事件 envelope 的语义都由 `libs/contracts/events/comment` 拥有；私有 outbox 表只是 Comment 的可靠投递台账，不是跨服务共享表。

### 查询和缓存策略

PostgreSQL 是评论和统计的真相源。Redis 只保存可重建缓存：

- 评论详情缓存。
- 文章顶级评论列表缓存。
- 根评论回复列表缓存。
- 点赞状态和点赞数缓存。
- 首页评论缓存。
- Ranking 热门候选缓存。

查询规则：

- 传统分页用于 Web 端固定页码场景。Comment 本轮为重做模块，新接口按 `docs/contracts/pagination.md` 默认从 `page=1` 开始；只有字段级 contract 单独声明时才调整。
- 游标分页用于移动端无限滚动，cursor 对外不透明。
- 增量补拉使用 `afterCreatedAt` 和 `afterId` 做稳定排序，避免同一时间戳下漏读或重复读。
- `TIME` 排序以 `created_at` 和 `id` 作为稳定 tie-breaker。
- `HOT` 排序可以使用点赞数、回复数和时间衰减规则，但具体公式必须从 Java 查询/缓存实现提取后固定。
- 缓存 key 由 Comment 本地 resolver 统一生成，不在 handler 或 repository 中散写字符串。
- 展开回复不依赖“回复内序号”。API 先用根评论 `floor` 定位根评论，再按 `root_id` 查询回复列表；通知或深链命中某条回复时，先查该回复详情拿到 `rootFloor`，展开对应根评论后用当前回复的 `floor` 高亮目标。
- 第一阶段不引入 `replyIndex` / `replySequence`。如果产品未来需要展示“某根评论下第几条回复”，再单独设计根评论维度的序号和计数器，不能复用文章内 `floor`。

第一阶段缓存 key 模式建议：

这里的 `commentId` 是 Comment 内部 `comments.id`，只用于服务内缓存定位，不作为对外 HTTP 资源 ID。

| 缓存 | Key 模式 |
| --- | --- |
| 评论详情 | `comment:{commentId}:detail:v2` |
| 评论点赞数 | `comment:{commentId}:stats:likes` |
| 评论回复数 | `comment:{commentId}:stats:replies` |
| 用户是否点赞评论 | `comment:{commentId}:liked:{userId}` |
| 文章评论数 | `comment:post:{postId}:count` |
| 文章顶级评论时间排序 | `comment:post:{postId}:top:time` |
| 文章顶级评论热度排序 | `comment:post:{postId}:top:hot` |
| 根评论回复列表 | `comment:{rootId}:replies` |
| 首页评论快照 | `comment:post:{postId}:homepage:{sort}:{size}:{replyLimit}` |
| Ranking 热门候选 | `comment:ranking:posts:hot:candidates` |
| Ranking 热门候选元信息 | `comment:ranking:posts:hot:candidates:meta` |

事务提交后的缓存失效要求：

| 操作 | 失效或更新 |
| --- | --- |
| 创建根评论 | 删除 `comment:post:{postId}:top:time`、`comment:post:{postId}:top:hot`、`comment:post:{postId}:count`、匹配 `comment:post:{postId}:homepage:*` 的首页快照；必要时更新首页评论缓存 |
| 创建回复 | 删除 `comment:{rootId}:replies`、`comment:{rootId}:stats:replies`、`comment:{rootId}:detail:v2`、`comment:post:{postId}:top:time`、`comment:post:{postId}:top:hot`、匹配 `comment:post:{postId}:homepage:*` 的首页快照 |
| 更新评论 | 删除 `comment:{commentId}:detail:v2`、所属文章列表 key、所属根评论回复 key、匹配 `comment:post:{postId}:homepage:*` 的首页快照 |
| 删除根评论 | 删除目标评论详情、文章顶级评论列表、文章评论数、该根评论回复列表、回复详情缓存、相关点赞状态/计数缓存和首页快照 |
| 删除回复 | 删除回复详情、根评论回复列表、根评论详情、根评论回复数、文章顶级评论列表中根评论统计和首页快照 |
| 点赞/取消点赞 | 更新或删除 `comment:{commentId}:stats:likes`、`comment:{commentId}:liked:{userId}`、目标评论详情、所属列表和首页快照 |

需要通配删除的 key 应由 Comment 本地 cache store 封装实现，application 只表达“失效文章评论列表、根评论回复列表、首页快照”等语义，不直接拼接 Redis key。

### 热门候选同步

Ranking 热门候选第一阶段采用 pull 模式：

- `SyncRankingHotCandidates` 由 Comment 后台 job 定时调用 Ranking contract 获取热门文章候选。
- 多实例部署时使用分布式锁避免重复拉取和重复写 Redis。
- 成功拉取后用 `replaceCandidates` 原子替换 `comment:ranking:posts:hot:candidates` 和 `comment:ranking:posts:hot:candidates:meta`。
- 拉取失败时保留旧值并记录告警，不清空候选集。
- Ranking 仍拥有榜单分数和候选生成规则；Comment 只拥有本地缓存副本和首页评论查询时的使用策略。

### 事件

#### 领域事件与集成事件

领域事件只存在于 Comment domain 和 application 内部，用于表达聚合内发生的业务事实。集成事件是跨服务 contract，必须放到 `libs/contracts/events/comment` 后才能被其他服务稳定依赖。

集成事件中的 `commentId` 是 Comment 内部 `comments.id` 的跨服务 opaque reference，用于 consumer 幂等、通知、统计和热度 ledger；对外 HTTP 仍以 `(postId, floor)` 定位评论。

**Comment 跨服务集成事件**：

| 事件 | 触发用例 | 主要 payload | 当前/目标 consumer | outbox 要求 |
| --- | --- | --- | --- | --- |
| `comment.created` | `CreateComment` | `commentId`、`postId`、`floor`、`rootId`、`parentId`、`authorId`、`createdAt` | Content 更新评论计数；Notification 创建评论/回复通知；Ranking 更新热度 | 关键事件，使用 producer outbox |
| `comment.deleted` | `DeleteComment`、`AdminDeleteComment` | `commentId`、`postId`、`floor`、`rootId`、`authorId`、`deletedBy`、`deleteReason`、`deletedAt`、`isRoot`、`affectedCount` | Content 修正评论计数；Ranking 回滚热度；Notification 可隐藏相关通知 | 关键事件，使用 producer outbox |
| `comment.liked` | `LikeComment` | `commentId`、`postId`、`commentAuthorId`、`likedBy`、`occurredAt` | Notification 创建点赞通知；Ranking 更新热度 | 关键事件，使用 producer outbox |
| `comment.unliked` | `UnlikeComment` | `commentId`、`postId`、`commentAuthorId`、`unlikedBy`、`occurredAt` | Ranking 回滚热度；通知是否回滚按产品语义决定 | 关键事件，使用 producer outbox |

**`comment.created` payload 草案**：

```json
{
  "eventId": "evt_01HY...",
  "eventType": "comment.created",
  "payloadVersion": 1,
  "producer": "zhicore-comment",
  "occurredAt": "2026-06-23T10:30:00Z",
  "aggregateType": "comment",
  "aggregateId": "12345",
  "payload": {
    "commentId": 12345,
    "postId": 67890,
    "floor": 26,
    "rootId": null,
    "parentId": null,
    "authorId": 1001,
    "hasImages": false,
    "hasVoice": false
  }
}
```

事件 payload 只强制携带当前评论的 `floor`。`rootFloor` / `parentFloor` 不作为第一阶段事件必选字段；需要前端跳转展开时，Notification 或前端可以通过 `GetCommentDetail` 回查这些定位字段。实际事件 contract 以后以 `libs/contracts/events/comment` 为准。本文只固定语义方向，不替代字段级 contract。

#### 事件 consumer 归属

- Content：消费 `comment.created` / `comment.deleted` 更新 `post_stats.comment_count`。
- Notification：消费 `comment.created` / `comment.liked` 创建站内通知，是否消费 `comment.deleted` 隐藏通知按产品规则决定。
- Ranking：消费 `comment.created` / `comment.deleted` / `comment.liked` / `comment.unliked` 更新热度 ledger。
- Search：默认不消费 Comment 事件；如果未来搜索评论，再单独定义 Search read model。

### 跨服务依赖

- Content：创建评论前校验文章事实、文章可见性和是否允许评论；评论事件更新文章评论统计。
- User：评论作者资料摘要、用户状态、拉黑关系和互动权限。
- Upload：评论图片和语音上传、文件 URL 解析。
- Ranking：Comment 可以读取热门候选作为首页缓存输入，但不能拥有分数或榜单计算。
- Admin：管理端可以暴露删除评论 facade，但删除命令必须委托 Comment。

### 推荐首个实现切片

Comment 第一轮建议拆成三个小切片推进，避免同时引入评论树、统计聚合、outbox、游标和跨服务缓存导致验证面过大。

**切片 1：创建根评论 + 文章评论传统分页查询**

目标：验证 DDD 分层、数据库 identity 主键、基础事务边界和目标 HTTP 形态。

- Domain：建 `Comment` 聚合、`CommentStats` 聚合、`CommentFactory`、`CommentContentPolicy` 和 `CommentCreated` 领域事件。
- Ports：定义 `CommentCommandRepository`、`CommentQueryRepository`、`CommentStatsRepository`、`ContentPostClient`、`UserProfileClient`、`TransactionRunner`、`Clock`。
- Application：实现 `CreateComment` 和 `ListTopLevelCommentsByPage`，用内存 fake 验证 Content/User 校验、统计初始化和传统分页查询。
- Infrastructure：实现 PostgreSQL `CommentCommandRepository`、`CommentQueryRepository`、`CommentStatsRepository`，以及 Content、User client adapter 的最小能力。
- HTTP：实现 `POST /api/v1/posts/{postId}/comments` 和 `GET /api/v1/posts/{postId}/comments/page`。

**切片 2：删除评论 + 点赞**

目标：验证状态变更、统计聚合、幂等关系表、outbox 和缓存失效。

- 补 `DeleteComment`、`AdminDeleteComment`、`LikeComment`、`UnlikeComment`。
- 补 `CommentLike` 关系实体和 `CommentLikeRepository`。
- 补 `OutboxPublisher`、outbox writer 和 dispatcher。
- 补 `comment.deleted`、`comment.liked`、`comment.unliked` 事件落库和发布。
- 补删除、点赞、取消点赞的 domain/application/infrastructure 测试。
- HTTP 补 `DELETE /api/v1/posts/{postId}/comments/{floor}`、`POST /api/v1/posts/{postId}/comments/{floor}/like`、`DELETE /api/v1/posts/{postId}/comments/{floor}/like`。

**切片 3：回复 + 游标分页 + 增量补拉**

目标：验证扁平化回复树、cursor codec、稳定排序和移动端加载更多兼容。

- 补 `CreateReply`、`ListRepliesByPage`、`ListTopLevelCommentsByCursor`、`ListRepliesByCursor`、`ListTopLevelCommentsIncremental`、`ListRepliesIncremental`。
- 补 `CommentTreePolicy`、`RootCommentID`、`ParentCommentID`。
- 补目标 cursor codec：`TIME` 和 `HOT` URL-safe Base64 无 padding。
- HTTP 补 `GET /api/v1/posts/{postId}/comments/cursor`、`GET /api/v1/posts/{postId}/comments/{floor}/replies/cursor`、对应传统分页和增量补拉接口。

媒体上传、首页缓存、热门候选同步和 outbox admin 不进入前三个最小切片，除非字段级 contract 提取证明旧前端首屏必须依赖。

## 目标 API 范围

Comment 服务明确重做，不保留 Java 以全局 `commentId` 为外部资源 ID 的 API 形态。目标 API 以文章为上级资源，用 `(postId, floor)` 定位评论。这里的路径参数 `{postId}` 表示 Content 对外文章 ID；Comment application 通过 Content contract 解析或校验后，在本地表中保存内部 `post_id BIGINT` 引用。

- `/api/v1/posts/{postId}/comments`：创建评论、文章顶级评论列表。
- `/api/v1/posts/{postId}/comments/page`：文章顶级评论传统分页。
- `/api/v1/posts/{postId}/comments/cursor`：文章顶级评论游标分页。
- `/api/v1/posts/{postId}/comments/incremental`：文章顶级评论增量补拉。
- `/api/v1/posts/{postId}/comments/{floor}`：评论详情、更新、删除。
- `/api/v1/posts/{postId}/comments/{floor}/replies/page`：回复传统分页。
- `/api/v1/posts/{postId}/comments/{floor}/replies/cursor`：回复游标分页。
- `/api/v1/posts/{postId}/comments/{floor}/replies/incremental`：回复增量补拉。
- `/api/v1/posts/{postId}/comments/{floor}/like`：点赞、取消点赞。
- `/api/v1/posts/{postId}/comments/{floor}/liked`、`/api/v1/posts/{postId}/comments/{floor}/like-count`：点赞状态和计数查询。
- `/api/v1/posts/{postId}/comments/batch/liked`：批量点赞状态查询，body 使用 `floor` 列表。
- `/api/v1/comments/media/images`、`/api/v1/comments/media/voice`：可选评论媒体上传 facade；若目标前端直接调用 Upload 服务，可不实现。
- `/api/v1/admin/comments`：管理端评论查询和删除。
- `/api/v1/admin/comment-outbox/summary`、`retry-dead`：outbox summary 和 dead retry。

字段级 request/response、分页起点、排序枚举和错误码需要后续按目标 API 提取到 `services/zhicore-comment/api/http`；Java controller 只作为行为参考，不作为外部路径兼容要求。

回复列表接口中的 `{floor}` 表示根评论的文章内楼层。若调用方传入回复自身的楼层，handler 应先按 `(postId, floor)` 查详情；第一阶段建议返回参数错误并提示必须使用根评论楼层，避免隐式展开带来额外查询和语义混淆。

## 数据归属

Comment 拥有：

- `comments`
- `comment_stats`
- `comment_likes`
- `comment_post_counters`
- Comment 服务自己的 `outbox_events`

评论图片、语音资源只保存 `file_id` 或资源引用；文件事实归 Upload/File Service。

`outbox_events` 是 Comment 服务私有表。仓库允许不同服务数据库中存在同名 outbox 表；表归所在服务数据库拥有，不使用跨服务共享 outbox 表。该表可以保存完整 envelope JSON，也可以分列保存 envelope metadata 和 payload；但 `event_id` 必须在业务事务内生成并作为最终 RabbitMQ envelope 的 `eventId`。

`comments` 目标 schema 草案：

```sql
CREATE TABLE comments (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  post_id BIGINT NOT NULL,
  floor BIGINT NOT NULL,
  author_id BIGINT NOT NULL,
  root_id BIGINT NULL,
  parent_id BIGINT NULL,
  content TEXT NULL,
  image_ids TEXT[] NULL,
  voice_id VARCHAR(64) NULL,
  voice_duration INTEGER NULL,
  status VARCHAR(32) NOT NULL,
  deleted_by BIGINT NULL,
  delete_reason VARCHAR(500) NULL,
  deleted_at TIMESTAMPTZ NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT uq_comments_post_floor UNIQUE (post_id, floor),
  CONSTRAINT ck_comments_floor_positive CHECK (floor > 0),
  CONSTRAINT ck_comments_status CHECK (status IN ('NORMAL', 'DELETED')),
  CONSTRAINT ck_comments_root_parent_pair CHECK (
    (root_id IS NULL AND parent_id IS NULL)
    OR (root_id IS NOT NULL AND parent_id IS NOT NULL)
  ),
  CONSTRAINT ck_comments_no_self_parent CHECK (
    (root_id IS NULL OR root_id <> id)
    AND (parent_id IS NULL OR parent_id <> id)
  ),
  CONSTRAINT ck_comments_body_present CHECK (
    (content IS NOT NULL AND btrim(content) <> '')
    OR COALESCE(cardinality(image_ids), 0) > 0
    OR (voice_id IS NOT NULL AND btrim(voice_id) <> '')
  ),
  CONSTRAINT ck_comments_image_limit CHECK (
    COALESCE(cardinality(image_ids), 0) <= 9
  ),
  CONSTRAINT ck_comments_voice_image_exclusive CHECK (
    voice_id IS NULL OR COALESCE(cardinality(image_ids), 0) = 0
  ),
  CONSTRAINT ck_comments_voice_duration_positive CHECK (
    voice_id IS NULL OR voice_duration > 0
  )
);
```

`root_id` 和 `parent_id` 可以加服务内自引用外键，也可以先只用索引和 repository 校验保护，取决于 migration 阶段对软删除、导入顺序和批量修复的要求。无论是否加外键，application/repository 都必须在创建回复的事务内校验根评论和直接父评论属于同一 `post_id`、父评论未删除、根评论确实是顶级评论，并且父评论属于该根评论树。

`comment_post_counters` 目标 schema 草案：

```sql
CREATE TABLE comment_post_counters (
  post_id BIGINT PRIMARY KEY,
  next_floor BIGINT NOT NULL,
  CONSTRAINT ck_comment_post_counters_next_floor_positive CHECK (next_floor > 0)
);
```

## 迁移风险

- Java 历史中部分评论事件存在直接 MQ 路径，Go 侧关键评论事件必须补齐 producer outbox。
- 评论列表、回复列表、游标分页、增量补拉的排序和 cursor 语义必须逐项保留，否则前端加载更多会出错。
- 回复模型使用 `root_id + parent_id`：`root_id` 用于查询根评论下回复，`parent_id` 表示直接被回复评论。不要再引入 `reply_to_comment_id` 或 `reply_to_user_id` 冗余字段。
- `floor` 是文章内外部定位号，根评论和回复共享同一序列，必须在同一 `post_id` 下唯一、单调递增、删除不复用，不能用动态行号或回复列表内序号替代。
- 顶级评论删除会影响所有回复，必须明确批量删除、统计修正和事件语义。
- 媒体上传入口如果作为 facade 转发 Upload，需要明确文件归属不转移，并处理上传成功但评论创建失败的补偿策略。
- 点赞/取消点赞必须用唯一约束和原子计数保护幂等，不要只依赖 Redis。
- Ranking 热门候选只能作为输入或缓存，不应把榜单分数计算放进 Comment。
- Admin 删除评论必须委托 Comment mutation，不应在 Admin 复制评论状态逻辑。

## DDD 设计总结

本文档按 DDD 战术模式重新设计 Comment 服务，关键点：

1. **Comment 聚合只维护评论树、内容和状态**：点赞数和回复数拆到 `CommentStats`，避免热点聚合。
2. **回复树使用扁平化 `rootId + parentId`**：`rootId` 固定根评论，`parentId` 表示直接被回复评论，不再引入 `reply_to_comment_id` 或 `reply_to_user_id`。
3. **媒体引用不转移文件归属**：Comment 保存 `file_id`，Upload/File Service 仍是文件事实源。
4. **跨服务校验在 application 编排**：文章、用户、关系和文件事实通过 provider contract 获取，不进入 domain。
5. **事件跟随 Comment 数据归属**：评论创建、删除、点赞事件由 Comment outbox 生产，Content、Notification、Ranking 消费。
6. **查询模型不污染领域层**：传统分页、游标分页、增量补拉和管理端查询属于 application query / repository read model。
7. **端口按能力分组**：避免把 repository、cache、client、outbox 混成宽泛 `CommentService`。
8. **分层依赖方向符合 Onion Architecture**：domain 不依赖 HTTP、Redis、PostgreSQL、RabbitMQ、Upload SDK 或 Ranking client。
