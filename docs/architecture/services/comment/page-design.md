# Comment 页面设计

本文记录 Comment 相关页面的前端草稿、页面分区、接口编排、加载状态和降级规则。它是产品 / 前端交互设计事实源，不替代 HTTP 字段级 contract；字段、错误码和 path 仍以 `services/zhicore-comment/api/http/` 为准。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- 评论区依附于文章详情。文章主资源不可用时，不加载评论树、评论点赞状态或回复编辑器。
- 评论列表主资源优先于评论点赞 viewer 状态；点赞状态失败不能让评论文本不可读。
- 创建、回复、编辑和删除评论是写路径，User / Content / File service guard 不可确认时 fail closed。
- 评论图片和语音上传由 File service 完成；Comment 只接收文件 ID。上传失败只影响当前编辑器，不影响已加载评论。
- 删除评论使用软删除展示语义：保留楼层关系，隐藏正文和媒体，必要时展示“该评论已删除”。
- 评论区不使用全屏弹窗作为常规交互；回复、编辑和更多操作优先内联或轻量抽屉。

## 页面范围

本文覆盖：

- 文章详情下的评论区。
- 顶级评论列表。
- 回复列表和回复抽屉 / inline 展开。
- 评论创建 / 回复编辑器。
- 评论点赞状态和点赞动作。
- 删除 / 编辑评论的局部状态。
- 管理端评论审核入口的页面初设计。

本文不覆盖：

- 文章详情主资源和文章 engagement，这些归 Content 页面设计。
- 用户资料和关系状态详情，这些归 User 页面设计。
- 通知收件箱中的评论通知，这些归 Notification 页面设计。

## 评论区入口

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Comments header · count · sort             │
├────────────────────────────────────────────┤
│ Composer                                   │
│ text · media · [Submit]                    │
├────────────────────────────────────────────┤
│ Comment list                               │
│ author · time · text · like · reply        │
│ replies preview · load replies             │
├────────────────────────────────────────────┤
│ Load more / degraded state                 │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. Content 文章详情主资源成功后，评论区才调用 Comment 列表 endpoint。
2. 评论列表加载中显示评论行骨架，不阻塞文章阅读。
3. 列表成功后渲染顶级评论、基础统计、回复预览和删除占位。
4. 登录用户收集当前页评论 ID，调用评论点赞批量状态接口；没有批量接口前不做逐项 N+1。
5. 批量点赞状态返回后只更新仍在当前页的评论行；排序或分页变化后丢弃过期响应。
6. 评论区失败时显示局部错误和重试，不覆盖文章正文。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 文章不可用 | 不请求评论区，不展示评论入口。 |
| 评论列表失败 | 评论区局部错误，文章继续展示。 |
| 评论为空 | 显示空态和评论输入入口。 |
| 点赞状态失败 | 评论文本和计数继续展示，点赞按钮中性降级。 |
| 作者摘要失败 | 使用评论内作者快照或匿名占位，不隐藏评论正文。 |
| File URL 解析失败 | 媒体缩略图显示失败占位，正文继续展示。 |

## 评论创建和回复

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Composer                                   │
├────────────────────────────────────────────┤
│ textarea                                   │
│ media chips · voice attachment             │
├────────────────────────────────────────────┤
│ validation · upload state · [Submit]       │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 未登录用户点击输入框时进入登录引导，不调用创建接口。
2. 登录用户提交前做本地空内容、长度、媒体数量和语音时长校验。
3. 有媒体时先通过 File service 获取文件 ID；文件上传成功后才调用 Comment 创建接口。
4. 创建回复时提交 `parentCommentId`，并在 UI 上明确回复目标。
5. 创建成功后把新评论插入当前可见列表或回复列表，并刷新对应统计。
6. 创建失败时保留输入和已上传文件引用，允许用户重试或移除附件。

### 写路径失败处理

| 场景 | 页面行为 |
| --- | --- |
| Content guard 失败 | 提示文章已不可评论，禁用提交。 |
| User guard 失败 | 提示当前用户状态不可评论或互动。 |
| File 校验失败 | 只标记附件失败，正文仍保留。 |
| 频控 / 限流 | 显示操作太频繁，按钮进入短暂禁用。 |
| Comment 服务降级 | 保留输入，提示稍后重试。 |

## 回复列表

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Replies to comment                         │
├────────────────────────────────────────────┤
│ parent summary                             │
├────────────────────────────────────────────┤
│ reply row · reply row · reply row          │
├────────────────────────────────────────────┤
│ Reply composer · load more                 │
└────────────────────────────────────────────┘
```

回复可以在桌面端使用右侧抽屉，在移动端使用 inline 展开或全宽二级页面。无论形式如何，都必须保持以下加载规则：

- 只有用户点击展开回复或评论行可见时才加载回复列表。
- 父评论已删除时，仍可显示已有回复；是否允许继续回复由后端状态决定。
- 回复加载失败只影响当前父评论的回复区域。
- 新回复成功后更新父评论回复计数和当前回复列表。

## 评论点赞

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ comment actions                            │
│ like icon · count · reply · more           │
└────────────────────────────────────────────┘
```

点赞状态映射：

| API 状态 | UI 状态 | 行为 |
| --- | --- | --- |
| `liked=true` | 选中 | 允许取消点赞。 |
| `liked=false` | 未选中 | 允许点赞。 |
| `liked=null` + `degraded=true` | 中性 unknown | 不显示为未点赞；可提示状态暂不可用。 |

点击规则：

1. 未登录：进入登录引导。
2. 状态 unknown：不做乐观更新，优先重新加载状态或提示暂不可用。
3. 确认未点赞时调用点赞接口；确认已点赞时调用取消点赞接口。
4. 成功后使用响应事实更新行内状态和计数。
5. 失败时回滚乐观状态或保持原状态，显示行内提示。

## 编辑和删除评论

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Comment row                                │
│ text / deleted placeholder                 │
│ actions: edit · delete · report            │
└────────────────────────────────────────────┘
```

规则：

- 只有作者或管理员入口显示编辑 / 删除操作。
- 编辑时复用 composer，但必须展示“正在编辑”的上下文，避免用户误以为在回复。
- 删除顶级评论后，回复结构保留；评论行切换为删除占位。
- 删除请求 pending 时只锁定当前评论行，不锁整页。
- 重复删除成功或已删除状态应表现为幂等完成，不重复扣减本地计数。

## 管理端评论审核入口

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Admin comments                             │
├────────────────────────────────────────────┤
│ Filters: status · post · author · keyword  │
├────────────────────────────────────────────┤
│ Table                                      │
│ comment · post · author · status · action  │
├────────────────────────────────────────────┤
│ Detail panel · delete reason · audit state │
└────────────────────────────────────────────┘
```

管理端评论页面由 Admin facade 暴露入口，删除命令最终委托 Comment。页面必须：

- 显示来源文章、作者摘要、评论正文截断、状态和举报上下文。
- 删除前要求填写原因。
- 删除成功后更新当前行状态并记录审计状态。
- 委托 Comment 失败时不显示为已删除；Admin 可记录失败审计。

## 跨服务页面约定

- Content 详情不可用时，Comment 页面不加载。
- User relation 或 availability 不可确认时，评论创建、回复、点赞等写动作 fail closed。
- File service 只提供文件 ID 和展示 URL；Comment 页面保存草稿时只持有文件 ID，不把 URL 当事实。
- Notification 中点击评论通知进入文章详情后，仍按 Content -> Comment 的顺序加载，不直接打开孤立评论详情。
