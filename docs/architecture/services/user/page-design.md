# User 页面设计

本文记录 User 相关页面的前端草稿、页面分区、接口编排、加载状态和降级规则。它是产品 / 前端交互设计事实源，不替代 HTTP 字段级 contract；字段、错误码和 path 后续以 `services/zhicore-user/api/http/` 为准。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- 用户公开资料是主资源，头像 URL、关注状态、拉黑状态和统计是附加资源；主资源不可用时不展示关系操作。
- 当前登录主体由 Auth 提供，User 页面只展示和修改资料、关系、拉黑和用户设置，不处理凭证和 token。
- 关系按钮不能把未知状态显示成未关注 / 未拉黑；User 降级时写路径 fail closed。
- 头像上传由 Upload 页面 / 组件完成，User 只保存 `avatarFileId` 并在查询时派生 `avatarUrl`。
- 公开主页和设置页都应使用任务型产品 UI：清晰导航、稳定骨架、可恢复表单，不做营销式首屏。

## 页面范围

本文覆盖：

- 我的资料页。
- 编辑资料页。
- 公开用户主页。
- 关注 / 粉丝列表。
- 拉黑列表和拉黑动作。
- 用户隐私与私信设置。
- 签到页初设计。

本文不覆盖：

- 登录、注册和账号安全设置，这些归 Auth 页面设计。
- 作者文章列表的文章加载细节，这些归 Content 页面设计。
- 私信会话页，这些归 Message 页面设计。

## 我的资料页

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Settings nav: profile · privacy · blocks   │
├────────────────────────────────────────────┤
│ Profile summary                            │
│ avatar · nickname · bio · public id        │
├────────────────────────────────────────────┤
│ Stats row                                  │
│ following · followers · posts              │
├────────────────────────────────────────────┤
│ [Edit profile] [View public page]          │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 页面进入时先由 Auth 全局状态确认已登录。
2. 调用 `GET /api/v1/users/me` 或目标 User profile endpoint。
3. 主资源成功后展示资料、头像、统计和设置入口。
4. 头像 URL 解析失败时显示默认头像，不影响资料文本。
5. 资料加载失败且为未登录时跳转登录；服务降级时显示页面级错误和重试。

## 编辑资料页

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Edit profile top bar · save state          │
├────────────────────────────────────────────┤
│ Avatar picker                              │
│ current avatar · upload / remove           │
├────────────────────────────────────────────┤
│ Form                                       │
│ nickname · bio · stranger message toggle   │
├────────────────────────────────────────────┤
│ [Cancel]                         [Save]    │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 先加载当前资料作为表单初始值。
2. 用户选择头像时，先调用 Upload 获取 `fileId` 和临时展示 URL。
3. 用户保存时调用 User 资料更新 endpoint，提交 `nickname`、`bio`、`avatarFileId` 和设置字段。
4. 保存成功后刷新当前资料缓存，并通知依赖作者摘要的页面重新取数。
5. 保存失败时保留本地表单输入和已上传但未保存的头像选择，提示用户可重试。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 昵称冲突 | `nickname` 字段级错误。 |
| Upload 失败 | 只影响头像选择，资料表单仍可编辑。 |
| User 保存失败 | 表单级错误，保留输入。 |
| User 降级 | 禁用保存，显示资料服务暂不可用。 |
| 登录态失效 | 跳转登录，保留路由返回信息。 |

## 公开用户主页

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ User header                                │
│ avatar · nickname · bio                    │
│ follow · message · block                   │
├────────────────────────────────────────────┤
│ Tabs: posts · comments · profile           │
├────────────────────────────────────────────┤
│ Main list                                  │
│ user posts / activity placeholder          │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 路由进入时调用公开资料 endpoint，使用 `publicId` 定位用户。
2. 公开资料成功后，再加载 viewer relation 状态，例如是否关注、是否被拉黑、是否允许私信。
3. 文章 tab 使用 Content 的作者文章列表加载逻辑。
4. relation 状态失败时，公开资料和文章列表继续展示；关注、私信、拉黑按钮显示 unknown 或 disabled。
5. 用户不存在、已删除或不可见时显示页面级 404，不请求文章和 relation 状态。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 公开资料加载失败 | 页面级错误；不展示关注 / 私信按钮。 |
| relation 加载失败 | 关系按钮中性降级；资料仍展示。 |
| 自己访问自己主页 | 不展示关注 / 拉黑，显示编辑资料入口。 |
| 目标用户被拉黑 | 隐藏私信入口，显示解除拉黑入口。 |
| 被目标用户拉黑 | 隐藏互动入口，提示无法互动。 |

## 关注和粉丝列表

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Relation list header                       │
│ tabs: following · followers                │
├────────────────────────────────────────────┤
│ Search / filter                            │
├────────────────────────────────────────────┤
│ User row                                   │
│ avatar · nickname · bio · relation action  │
│ ...                                        │
├────────────────────────────────────────────┤
│ Load more                                  │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 调用关注或粉丝列表 endpoint，使用 cursor 或 page 分页。
2. 列表主资源成功后渲染用户摘要。
3. 如果列表项需要当前 viewer 对每个用户的关注状态，使用批量 relation 状态接口；没有批量接口前不逐项 N+1 请求。
4. 点击关注 / 取关成功后只更新当前行和列表统计；不要整页刷新。
5. 翻页或切换 tab 时丢弃过期 relation 响应。

## 拉黑列表和拉黑动作

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Blocks                                     │
├────────────────────────────────────────────┤
│ Blocked users                              │
│ avatar · nickname · reason state · unblock │
│ ...                                        │
├────────────────────────────────────────────┤
│ Empty / degraded / retry                   │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 设置页进入拉黑 tab 时加载拉黑列表。
2. 在公开用户主页点击拉黑需要二次确认，并说明会影响私信、评论互动或后续可见交互。
3. 拉黑成功后立即更新当前页面 relation 状态，并让 Message / Comment 相关入口重新判断可用性。
4. 解除拉黑成功后当前行移除或切换为已解除状态。

写路径在 User 不可用时必须 fail closed；不能在本地先显示已拉黑但服务端未确认。

## 隐私与私信设置

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Privacy settings                           │
├────────────────────────────────────────────┤
│ Stranger message                           │
│ [toggle] allow messages from strangers     │
├────────────────────────────────────────────┤
│ Visibility hints                           │
│ follow/block impact                        │
├────────────────────────────────────────────┤
│ Save state                                 │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 设置页加载当前 User profile / settings。
2. 切换陌生人私信设置后可立即保存或点击保存，取决于前端整体设置页模式；无论哪种都必须显示保存状态。
3. 保存成功后通知 Message 页面重新计算私信入口可用性。
4. 保存失败时回滚开关或标记未保存，不能静默失败。

## 签到页初设计

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Check-in                                   │
├────────────────────────────────────────────┤
│ Today status · streak · month summary      │
├────────────────────────────────────────────┤
│ Calendar grid                              │
│ day cells · checked / missed / today       │
├────────────────────────────────────────────┤
│ [Check in] · reward / error state          │
└────────────────────────────────────────────┘
```

签到属于 User 完整边界，但不是首批 schema 细化范围。页面初设计只保留未来落点：

- 先加载月度签到状态。
- 今日未签到时展示操作按钮。
- 操作成功后更新今日格子、连续天数和统计。
- User 降级时禁用签到按钮，保留历史日历展示。

## 跨服务页面约定

- Content、Comment、Message、Notification 页面只消费 User 摘要和关系状态，不复制用户资料事实。
- User 资料更新后，依赖作者摘要的页面应按各自缓存策略刷新，而不是等待 User 页面手动传播字段。
- 拉黑和私信设置变化会影响 Message 发送入口、Comment 互动入口和 Notification 触达策略；这些页面必须在进入或动作前重新校验关键关系。
