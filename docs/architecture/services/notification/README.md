# Notification 服务设计

## 事实来源

- Java `zhicore-notification` controller：`NotificationQueryController`、`NotificationCommandController`、`NotificationPreferenceController`、`NotificationDeliveryController`。
- Java `zhicore-notification` domain model：`Notification`、`NotificationGroupState`、`NotificationCampaign`、`NotificationCampaignShard`、`NotificationDelivery`、`UserNotificationPreference`、`UserNotificationDnd`、`UserAuthorSubscription`、通知类型和通道枚举。
- Java `zhicore-notification` application service：通知查询、聚合、已读、偏好、免打扰、作者订阅、campaign、shard、channel delivery、digest 和 delivery retry。
- Java `zhicore-notification` MQ consumer：文章点赞、文章发布、评论创建、用户关注、实时 fanout 和未读数 fanout。
- Java `database/init-all-databases.sql` 中 notification 表族。
- `../zhicore-microservice/docs/architecture/zhicore-notification-platform-design.md`、`../zhicore-microservice/docs/notifications_jsonb_design.md`。
- User、Content、Comment 对通知事件的生产设计。

## 职责边界

`zhicore-notification` 拥有通知收件箱、通知聚合状态、未读数读模型、用户通知偏好、免打扰、作者订阅、广播 campaign、广播分片、通道投递 ledger、全局公告、小助手消息和 Notification 自己的实时 fanout 事件。

Notification 不拥有触发通知的源事实，例如用户、文章、评论、关注关系、私信或榜单分数。它消费源服务事件，生成自己的通知读模型和投递台账。

WebSocket 连接可以由 Gateway 或专门的连接网关在运行时承载，但通知实时投递语义、未读数变化、delivery ledger 和 retry 状态仍归 Notification。Gateway 只能转发、鉴权和路由连接状态，不能拥有 Notification 的收件箱、未读数或通道投递事实。

## DDD 目标设计

Notification 是独立限界上下文。统一语言以“通知、收件箱、通知类型、通知分类、聚合组、未读数、偏好、免打扰、作者订阅、campaign、shard、delivery、摘要、实时 fanout”为核心，不把 User、Content、Comment、Message 或 Push provider 的模型引入 Notification 领域层。

DDD 设计用于指导 Go 目标实现，不表示当前 Go 代码已经完成。Java 侧已有通知平台设计、领域模型、查询服务、Redis 缓存、MQ consumer、campaign 和 delivery 代码是事实来源之一，但 Go 实现按本仓库 `api/http -> application -> domain/ports -> infrastructure` 的依赖方向重新落点。

### 限界上下文与子域

Notification 上下文按职责拆成以下子域：

| 子域 | 职责 | 主要存储 |
| --- | --- | --- |
| Notification Center | 站内通知收件箱、聚合列表、未读数、标记已读、全部已读 | `notifications`、`notification_group_state`、Redis |
| Preference / DND | 类型 x 通道偏好、免打扰、作者订阅 | `notification_user_preference`、`notification_user_dnd`、`notification_author_subscription` |
| Interaction Pipeline | 点赞、评论、回复、关注等点对点通知消费和沉淀 | `notifications`、`notification_group_state`、consumer idempotency |
| Broadcast Pipeline | 作品发布等需要粉丝 fanout 的 campaign、分片和摘要任务规划 | `notification_campaign`、`notification_campaign_shard` |
| Channel Delivery | 站内、WebSocket、Email、SMS 的投递规划、执行、重试和审计 | `notification_delivery` |
| Realtime Fanout | 用户通知、未读数、评论流提示等实时推送事件 | RabbitMQ/WebSocket adapter、Redis |
| Announcement / Assistant | Notification 自己拥有的全局公告、小助手消息和系统辅助内容 | `global_announcements`、`assistant_messages` |

### 聚合

#### `Notification` 聚合

`Notification` 是单条站内通知收件箱记录的聚合根，负责维护通知自身生命周期和已读状态。

- **内部标识**：`NotificationID` 使用 Notification 服务数据库生成的 `BIGINT GENERATED BY DEFAULT AS IDENTITY`，只用于服务内关联、索引和事务，不作为 HTTP 资源 ID。
- **外部标识**：`NotificationPublicID` 存储在 `notifications.public_id`，HTTP path、response 和前端使用的 `notificationId` 都指向该字段。生成方式参考 Content 的短公开 ID：`n` 资源前缀 + 算法版本 + `Base58/Base62(permute64(internal_id, public_id_secret[version]))`，例如 `n1K8x9Q2`；不使用 UUIDv7，也不直接暴露 DB 自增 ID。
- **归属**：`RecipientID` 引用 User 拥有的用户；`ActorID` 可选引用触发者用户；`TargetType` / `TargetID` 是源对象 opaque reference。
- **分类字段**：`Category`、`NotificationType`、`EventCode`、`Importance`。
- **幂等字段**：`SourceEventID` 记录上游事件 envelope 的 `eventId`；`DedupeKey` 表达业务幂等键，例如 `post-liked:{postId}:{actorId}:{recipientId}`，不是资源 ID。
- **聚合字段**：`GroupKey` 用于通知列表聚合，例如 `POST_LIKED:post:{postId}` 或 `campaign:{campaignId}:post:{postId}`，不是资源 ID。
- **展示字段**：`Title`、`Content`、`Payload`。`Payload` 使用 JSONB 保存轻量展示快照和跳转信息，不保存源服务权威事实。
- **状态字段**：`Unread`、`Read`、`Archived`；第一阶段可以只落 `is_read` / `read_at`。
- **行为**：`MarkRead`、`MarkUnreadForRepair`、`Archive`、`AssertRecipient`、`BuildGroupKey`。
- **领域事件**：`NotificationCreated`、`NotificationRead`。

`Notification` 的核心不变量：

- `RecipientID` 必须有效，但用户是否存在、是否被禁用由 application 通过 User contract 或事件上下文校验。
- 不能给触发者自己发送无意义互动通知，例如自己点赞自己的文章、自己关注自己、自己回复自己。
- `NotificationType` 必须映射到稳定 `Category` 和 `EventCode`。
- `Content` 不存长正文，互动内容需要截断；完整文章、评论或用户资料仍归源服务。
- `Payload` 可以包含文章标题、评论摘要、跳转路由、当前评论楼层等展示快照；这些快照不是权威数据。
- `DedupeKey` 一旦确定不可变；重复事件只能命中同一通知或被安全忽略。
- `NotificationPublicID` 一旦生成不可变；对外只接受 public ID，application 解析到内部 `NotificationID` 后继续按 `notification_id + recipient_id` 限定，避免越权读写他人通知。

#### `NotificationGroupState`

`NotificationGroupState` 是聚合通知列表的同步读模型，不是源事实聚合。

- **标识**：`(RecipientID, GroupKey)`。
- **字段**：`NotificationType`、`LatestNotificationID`、`TotalCount`、`UnreadCount`、`LatestActorIDs`、`TargetType`、`TargetID`、`LatestContent`、`LatestTime`。
- **行为**：`ApplyNotificationCreated`、`ApplyNotificationRead`、`ApplyAllRead`、`RebuildFromInbox`。
- **不变量**：`TotalCount >= 0`、`UnreadCount >= 0`、`UnreadCount <= TotalCount`。

它存在的原因：

- 聚合通知列表不能每次都扫描 `notifications` 做 group by。
- 热点用户短时间内收到大量互动时，读模型可以控制读放大。
- Go 目标在创建通知、标记已读、全部已读的同一 PostgreSQL 事务内同步维护它。正常路径下它应与 `notifications` 强一致，但它仍不是唯一真相源；实现必须提供 `RebuildFromInbox` 从 `notifications` 表重建，用于修复历史 bug、手工数据修复或异常中断后的计数问题。

#### `NotificationPreference` / `NotificationDnd` / `AuthorSubscription`

用户通知设置属于 Notification 拥有的偏好聚合族。第一阶段可以按表自然键实现，不需要把三者强行合成一个大聚合。

`NotificationPreference`：

- **标识**：`(UserID, NotificationType, Channel)`。
- **字段**：`Enabled`。
- **行为**：`Enable`、`Disable`。
- **不变量**：第一阶段 `SMS` 默认不允许启用；后续接入短信 provider 后再打开。

`NotificationDnd`：

- **标识**：`UserID`。
- **字段**：`Enabled`、`StartTime`、`EndTime`、`Categories`、`Channels`、`Timezone`。
- **行为**：`IsActiveAt`、`AppliesTo`。
- **不变量**：`StartTime != EndTime`；免打扰只影响主动通道打扰，不删除站内通知事实；`SECURITY` 可以按策略穿透。

`AuthorSubscription`：

- **标识**：`(UserID, AuthorID)`。
- **字段**：`Level`、`InAppEnabled`、`WebsocketEnabled`、`EmailEnabled`、`DigestEnabled`。
- **行为**：`AllowsChannel`、`NormalizeByLevel`。
- **不变量**：`MUTED` 禁止主动通道；`DIGEST_ONLY` 只进入摘要；默认值为普通订阅。

#### `NotificationCampaign`

`NotificationCampaign` 是一次广播型通知的聚合根，负责维护广播任务整体生命周期。

- **内部标识**：`CampaignID`，数据库 identity。
- **触发来源**：`SourceEventID`、`CampaignType`，例如 `POST_PUBLISHED`。
- **源对象引用**：`AuthorID`、`PostID`、`ObjectType`、`ObjectID`。
- **受众估算**：`AudienceEstimate`。
- **展示快照**：`Title`、`Excerpt`、`Payload`、`PublishedAt`。
- **状态**：`Planned`、`Processing`、`Completed`、`Failed`、`Canceled`。
- **行为**：`Plan`、`MarkProcessing`、`MarkCompleted`、`MarkFailed`、`Cancel`。
- **不变量**：`SourceEventID` 唯一；同一上游发文事件只能规划一个 campaign；campaign 不直接保存粉丝列表。

Campaign 适用于高粉作者发布作品和未来需要粉丝 fanout 的运营活动。它不是单条通知，不能把 campaign 状态和用户收件箱状态混在一个表里。全局公告和小助手消息默认属于 Announcement / Assistant；只有当它们需要按粉丝或人群批量投递时，才由 application 额外规划 campaign。

#### `NotificationCampaignShard`

`NotificationCampaignShard` 是广播分片的独立工作单元。

- **内部标识**：`ShardID`，数据库 identity。
- **归属**：`CampaignID`。
- **游标字段**：`StartCursor`、`EndCursor` 或 User follower cursor。
- **统计字段**：`ShardSize`、`ProcessedCount`、`SuccessCount`、`SkippedCount`、`FailedCount`。
- **Claim 字段**：`ClaimedBy`、`ClaimedAt`、`ClaimDeadlineAt`、`NextRetryAt`。
- **状态**：`Pending`、`Processing`、`Completed`、`Failed`。
- **行为**：`Claim`、`MarkCompleted`、`MarkFailed`、`ScheduleRetry`。
- **不变量**：同一 shard 只能被一个 worker claim；claim 必须由 repository 用 `UPDATE ... WHERE status IN (...) AND (next_retry_at IS NULL OR next_retry_at <= now()) RETURNING *` 或 `SELECT ... FOR UPDATE SKIP LOCKED` 保证并发安全；`Processing` 且超过 `claim_deadline_at` 的 shard 可以被重新 claim；`next_retry_at` 只在失败或调度重试时设置，不在普通 claim 时推进业务重试时间。

Shard worker 不应在一个超大事务中处理全部粉丝。Go 目标推荐按“claim shard -> 拉取一页粉丝 -> 小批量事务写 delivery/notification -> 更新 shard 进度”的方式实现，避免 100 万粉丝场景下长事务和锁放大。

#### `NotificationDelivery`

`NotificationDelivery` 是通道投递 ledger 聚合根，负责记录通道规划、执行结果、失败原因和重试。

- **内部标识**：`DeliveryID`，数据库 identity。
- **归属**：`RecipientID`、可选 `CampaignID`、可选 `NotificationID`。
- **通道**：`IN_APP`、`WEBSOCKET`、`EMAIL`、`SMS`。
- **类型**：`NotificationType`。
- **幂等**：`DedupeKey` 全局唯一。
- **状态**：`PLANNED`、`INBOX_CREATED`、`WEBSOCKET_PENDING`、`WEBSOCKET_DISPATCHED`、`DIGEST_PENDING`、`DIGEST_DELIVERED`、`SKIPPED`、`FAILED`。
- **行为**：`BindNotification`、`MarkSent`、`MarkSkipped`、`MarkFailed`、`CanRetry`。
- **不变量**：同一个 `DedupeKey` 只能有一条 delivery；只有可重试状态允许 retry；站内通知存在性不依赖 WebSocket、Email 或 SMS 成功；domain 只判断 delivery 状态是否可重试，操作者是否有权 retry 由 application 基于认证上下文判断。

#### `GlobalAnnouncement` 和 `AssistantMessage`

`GlobalAnnouncement` 和 `AssistantMessage` 是 Notification 拥有的简单聚合：

- `GlobalAnnouncement` 用于全局公告、系统活动、运营消息。
- `AssistantMessage` 用于小助手或系统辅助消息。

它们和 campaign 的区别是：Announcement / Assistant 是 Notification 拥有的内容或消息源，campaign 是批量投递计划和执行台账。第一阶段可以只作为独立查询/写入模型，不纳入交互通知和 campaign 的最小实现切片；需要广播时由 application 把公告或小助手消息引用到 campaign，而不是把两类表合并。

#### 非领域聚合

以下对象不建成领域聚合：

- Redis 未读数、聚合列表缓存和 WebSocket 在线连接：运行时读模型或连接状态。
- RabbitMQ delivery tag、consumer offset、DLQ 消息：消息基础设施状态。
- Push、Email、SMS provider response：通道 adapter 结果，只进入 delivery ledger。
- Sentinel / 限流配置：运行时保护机制。
- 源服务的用户、文章、评论、关注关系：只保存引用或事件快照，不进入 Notification domain。

### 值对象

| 值对象 | 含义 |
| --- | --- |
| `NotificationID` | 通知内部标识 |
| `NotificationPublicID` | 通知外部公开标识，对应 HTTP `notificationId` |
| `RecipientID` | 接收者用户引用 |
| `ActorID` | 触发者用户引用 |
| `NotificationType` | `POST_LIKED`、`POST_COMMENTED`、`COMMENT_REPLIED`、`USER_FOLLOWED`、`POST_PUBLISHED_BY_FOLLOWING`、`POST_PUBLISHED_DIGEST`、`SYSTEM_ANNOUNCEMENT`、`SECURITY_ALERT` |
| `NotificationCategory` | `INTERACTION`、`CONTENT`、`SOCIAL`、`SYSTEM`、`SECURITY` |
| `NotificationChannel` | `IN_APP`、`WEBSOCKET`、`EMAIL`、`SMS` |
| `NotificationImportance` | `NORMAL`、`HIGH`、`CRITICAL` |
| `SourceEventID` | 上游事件 envelope 的 `eventId` |
| `DedupeKey` | 业务幂等键 |
| `GroupKey` | 聚合列表分组键 |
| `NotificationPayload` | JSONB 展示快照和跳转参数 |
| `DndWindow` | 免打扰时间窗和作用范围 |
| `AuthorSubscriptionLevel` | `ALL`、`DIGEST_ONLY`、`MUTED` |
| `AudienceBucket` | `PRIORITY`、`NORMAL`、`DIGEST`、`MUTED` |
| `DeliveryStatus` | 通道投递状态 |
| `CampaignStatus`、`ShardStatus` | 广播任务和分片状态 |

核心约束：

| 值对象 | 约束 |
| --- | --- |
| `NotificationPublicID` | 使用 Notification 本地短公开 ID 规则生成，唯一、稳定、不可复用，不承载权限语义；实现依赖 `permute64`、Base58/Base62 codec 和服务级 secret |
| `DedupeKey` | 必须稳定、可重复计算，不包含随机数 |
| `GroupKey` | 同一用户下用于聚合展示，不能混用不同业务语义 |
| `NotificationPayload` | 只保存展示快照和跳转参数；需要结构化查询的字段优先关系型化 |
| `DndWindow` | 支持跨日时间窗，`start_time != end_time` |
| `NotificationChannel` | `SMS` 第一阶段保留枚举但默认不可启用 |
| `SourceEventID` | 用于消费幂等和追踪，不能当作源对象 ID |

`NotificationType` 是受控扩展枚举，不是随意字符串。新增类型必须同时补齐 `NotificationCategory` 映射、`EventCode`、`NotificationDedupePolicy`、`NotificationGroupingPolicy`、默认展示文案、payload schema 和缓存失效影响；如果新类型需要被其他服务消费，再新增或扩展 `libs/contracts/events/notification`，不要让 consumer 依赖 Notification 服务私有枚举实现。

### 领域服务

领域服务只承载纯业务规则，不依赖 PostgreSQL、Redis、RabbitMQ、WebSocket 或外部 provider。

| 领域服务 | 职责 |
| --- | --- |
| `NotificationDedupePolicy` | 根据事件类型、源对象、触发者和接收者生成稳定 `DedupeKey` |
| `NotificationGroupingPolicy` | 生成 `GroupKey`，决定哪些通知在列表中聚合 |
| `NotificationContentPolicy` | 截断评论摘要、生成默认文案、限制内容长度 |
| `PreferenceDecisionPolicy` | 合并类型偏好、作者订阅、免打扰和安全穿透规则 |
| `ChannelPlanPolicy` | 将偏好决策映射为 `IN_APP`、`WEBSOCKET`、`EMAIL`、`SMS` delivery plan |
| `CampaignShardPolicy` | 计算分片大小、下一分片游标和重试策略 |
| `DeliveryRetryPolicy` | 判断 delivery 是否可重试、计算 `next_retry_at` |
| `UnreadCountPolicy` | 已读、全部已读和重建时的非负计数规则 |

基础设施检查由 application 编排：

- 粉丝列表、用户摘要、作者关系：通过 User contract。
- 文章标题、文章作者、评论定位：优先来自事件 payload；缺失时通过 Content / Comment contract 补齐。
- WebSocket、Push、Email、SMS：通过 channel provider adapter。
- RabbitMQ 消费幂等、DLQ、retry：通过 infrastructure。

### Application 用例

**命令用例（Commands）**：

- `CreateInteractionNotification`：由事件 consumer 调用，按 `eventId` 和 `DedupeKey` 双层幂等创建站内通知、生成 `NotificationPublicID`、同步更新 group state、写 delivery plan，事务后更新未读缓存和实时 fanout。
- `MarkNotificationRead`：接收 HTTP `notificationId` 时先按 `public_id` 解析内部 `NotificationID`，校验接收者，标记单条已读，更新 group unread，事务后递减或删除未读缓存。
- `MarkAllNotificationsRead`：标记用户全部通知已读，重置 group unread，事务后清理未读和聚合缓存。
- `UpdateNotificationPreference`：更新类型 x 通道偏好，第一阶段拒绝启用 `SMS`，事务提交后删除 `notification:{userId}:preferences`。
- `UpdateNotificationDnd`：更新免打扰配置，事务提交后删除 `notification:{userId}:dnd`。
- `UpdateAuthorSubscription`：更新作者级订阅偏好，事务提交后删除 `notification:{userId}:author:{authorId}:subscription`。
- `PlanPostPublishedCampaign`：消费 `content.post.published`，按 `sourceEventId` 幂等创建 campaign 和初始 shard。
- `ExecuteCampaignShard`：claim shard、读取粉丝分片、按偏好规划 delivery、创建站内通知或摘要候选、更新 shard 进度。`claim_timeout` 从配置读取，并可由 `CampaignShardPolicy` 按 shard size 动态收敛到配置允许范围。
- `RetryDelivery`：application 先校验操作者权限，普通用户只能 retry 自己的 delivery，管理员可 retry 任意 delivery；domain 再校验 delivery 状态，重新执行 WebSocket delivery；Email/SMS 第一阶段只记录未配置或跳过。
- `ProcessPendingDigest`：把 `DIGEST_PENDING` delivery 合并为摘要通知，并按偏好尝试推送。
- `RebuildUnreadCount` / `RebuildGroupState`：从 `notifications` 回源重建 Redis 未读数和 `notification_group_state`。触发方式为管理端 API 手动触发或低频定时 job；默认粒度为单个 `recipient_id`，全量 rebuild 必须分批；rebuild 期间该用户聚合列表查询回退到 DB 聚合，避免读取半更新的 group state。
- `CleanupExpiredConsumedEvents`：清理过期 `consumed_events`，由 `application/jobs` 拥有用例，runtime 按配置调度；infrastructure 只提供批量删除实现。

**查询用例（Queries）**：

- `ListAggregatedNotifications`：聚合通知列表，优先读 `notification_group_state`，不一致时回退 DB 聚合。
- `GetUnreadCount`。
- `GetUnreadBreakdown`：按 `INTERACTION`、`CONTENT`、`SYSTEM`、`SECURITY`、`SOCIAL` 统计。
- `GetNotificationPreferences`。
- `GetNotificationDnd`。
- `GetAuthorSubscription`。
- `ListDeliveries`：普通用户只能查自己的 delivery；管理员可按 recipient/campaign/channel/status 筛选。
- `ListGlobalAnnouncements`、`ListAssistantMessages`：后续切片实现。

查询用例返回 DTO 或视图模型，不把复杂分页、聚合和用户摘要补齐放进 domain。聚合列表需要触发者摘要时，优先批量调用 User contract；不得直接读取 User 数据库。

### Ports

Ports 放在 `services/zhicore-notification/internal/notification/ports`，按能力和用例族定义 consumer-side interface。

**核心端口**：

| Port | 职责 |
| --- | --- |
| `NotificationCommandRepository` | 创建通知、按内部 ID 或 public ID 加载、标记已读、全部已读、按 dedupe key 幂等写入 |
| `NotificationQueryRepository` | 聚合列表、未读数、未读 breakdown、管理端查询 |
| `NotificationGroupStateRepository` | upsert group state、单条已读递减、全部已读清零、重建 |
| `NotificationPreferenceRepository` | 偏好、免打扰、作者订阅读写 |
| `NotificationCampaignRepository` | campaign 规划、状态流转、按 `source_event_id` 幂等检查 |
| `NotificationCampaignShardRepository` | shard 创建、claim、进度和重试状态流转 |
| `NotificationDeliveryRepository` | delivery 幂等写入、绑定通知、状态更新、查询和 retry |
| `AnnouncementRepository` | 全局公告读写，可选 |
| `AssistantMessageRepository` | 小助手消息读写，可选 |

**机制端口**：

| Port | 职责 |
| --- | --- |
| `TransactionRunner` | 显式事务边界 |
| `ConsumedEventStore` | RabbitMQ consumer 幂等记录，基于 `consumed_events.event_id` 防重复消费，和业务唯一索引共同实现双层幂等；提供按 `expires_at` 批量清理的方法 |
| `Clock` | 当前时间、业务本地时间和 DND 判断 |
| `NotificationPublicIDCodec` | 根据 Notification 内部 ID 生成和解析短公开 ID；封装 `permute64`、Base58/Base62 codec 和版本前缀，使用 runtime 注入的版本化服务级 secret，不依赖 `zhicore-id-generator`，也不用于 `DedupeKey` / `GroupKey` |

**缓存和外部服务端口**：

| Port | 职责 |
| --- | --- |
| `UnreadCountCacheStore` | 未读数 cache-aside、原子增减、删除和回源修复 |
| `AggregationCacheStore` | 聚合通知列表缓存 |
| `RealtimeFanoutPublisher` | 事务提交后 best-effort 发布用户通知、未读数、评论流提示；使用短超时和熔断，失败只记录日志/metrics，不写 delivery ledger，不承诺逐条可重试 |
| `UserProfileClient` | 批量用户摘要和用户状态 |
| `UserFollowerClient` | 粉丝数量、粉丝游标页和 follower shard |
| `ContentPostClient` | 发文事件缺失字段时补齐文章摘要 |
| `CommentClient` | 评论通知需要跳转定位或回复上下文时补齐评论详情 |
| `WebSocketDeliveryProvider` | 执行 `notification_delivery` 中的 `WEBSOCKET_PENDING` 投递，返回成功、失败和可重试原因，由 application 更新 delivery ledger |
| `EmailDeliveryProvider` | Email 投递，第一阶段可返回 provider unconfigured |
| `SMSDeliveryProvider` | SMS 投递，第一阶段可返回 provider unconfigured |

### 一致性与事务边界

**交互通知创建事务**：

```text
单个 PostgreSQL 事务：
  ConsumedEventStore 先 INSERT consumed_events(event_id, ...)
    - 插入成功：继续处理
    - event_id 冲突：返回 duplicate no-op，事务提交后 ack
  + notifications 生成 public_id 并按 recipient_id + dedupe_key 幂等插入
  + notification_group_state 同步 upsert
  + notification_delivery 可选写 WEBSOCKET_PENDING

事务提交后：
  未读数 Redis HINCRBY 或删除回源
  删除用户聚合列表缓存
  best-effort 发布 user notification / unread fanout
```

如果 `dedupe_key` 已存在，use case 返回 no-op，不重复增加未读数、不重复写 group state、不重复推送。

`consumed_events` 必须和业务写入处于同一事务。事务回滚时幂等记录也回滚，RabbitMQ retry 可以重新处理；事务提交后即使 ack 失败，下一次消费也会被 `event_id` 拦截。业务 `dedupe_key` 仍由 `notifications(recipient_id, dedupe_key)` 或 `notification_delivery(dedupe_key)` 唯一约束兜底，防止不同上游事件重复表达同一通知事实。

`public_id` 生成属于 application + infrastructure 协作：repository 推荐在同一事务中预取 identity 值，再用 `NotificationPublicIDCodec` 生成 `public_id` 后带显式 `id` 插入。事务回滚导致 identity gap 是可接受的，PostgreSQL identity 不保证无 gap，且内部 ID 不对外表达连续性。如果具体数据库或 ORM 不便显式预取 ID，可采用 `INSERT ... RETURNING id` 后同事务 `UPDATE public_id` 的备选方案，代价是多一次写入；无论采用哪种方式，都不能提交一条缺少 `public_id` 的通知。

**标记单条已读事务**：

```text
notifications 按 public_id 解析内部 id，并按 id + recipient_id 标记 read
+ notification_group_state(group_key).unread_count 原子 -1
```

事务提交后递减或删除 `notification:{userId}:unread`，并删除该用户聚合列表缓存。重复标记已读返回成功且不重复扣减。

**全部已读事务**：

```text
notifications 标记用户所有未读为 read
+ notification_group_state 将该用户所有 unread_count 置 0
```

事务提交后删除未读数缓存和聚合列表缓存。返回值可以包含实际影响条数，方便前端或日志。

**发文 campaign 规划事务**：

```text
notification_campaign 按 source_event_id 幂等插入
+ notification_campaign_shard 创建初始 shard
```

粉丝数量通过 User contract 获取。获取失败时不能写半成品 campaign；如果事件字段足够但粉丝估算失败，可按策略写 `PLANNED_WITH_UNKNOWN_AUDIENCE` 后由 job 补偿，第一阶段建议直接失败并由 RabbitMQ retry。

**执行 campaign shard**：

```text
短事务 1：claim pending 或可重试 shard
事务外：调用 UserFollowerClient 拉取一页粉丝
短事务 2..N：批量 upsert delivery / notification / group_state
短事务 final：更新 shard processed/success/skipped/failed 和 next cursor
```

claim 推荐由 repository 使用以下任一模式实现：

```sql
UPDATE notification_campaign_shard
SET status = 'PROCESSING',
    claimed_by = $worker_id,
    claimed_at = now(),
    claim_deadline_at = now() + $claim_timeout::interval,
    updated_at = now()
WHERE id = (
  SELECT id
  FROM notification_campaign_shard
  WHERE campaign_id = $campaign_id
    AND status IN ('PENDING', 'FAILED')
    AND (next_retry_at IS NULL OR next_retry_at <= now())
  ORDER BY id
  FOR UPDATE SKIP LOCKED
  LIMIT 1
)
RETURNING *;
```

`claim_timeout` 不能在 SQL 中硬编码。默认值由配置提供，`CampaignShardPolicy` 可以按 shard size 和批次大小计算推荐值，但必须落在配置的最小/最大范围内。如果 worker crash，`PROCESSING` 且 `claim_deadline_at < now()` 的 shard 允许被 reclaim；失败时由 `MarkFailed` / `ScheduleRetry` 记录失败原因、递增重试次数并设置下一次 `next_retry_at`。不要把读取粉丝、偏好判定、站内写入、WebSocket delivery 全部包进一个长事务。单个批次建议 500 到 2000 用户，具体大小由性能测试决定。

**Delivery retry**：

```text
加载 delivery
校验 retry 权限：
  普通用户：只能 retry recipient_id 等于自己的 delivery
  管理员：可 retry 任意 delivery，权限来自 auth context 或 Admin facade
校验 delivery 状态
加载 notification
重新判断偏好和免打扰
调用 channel provider
更新 delivery 状态
```

WebSocket、Email、SMS 失败不回滚 `notifications`。站内通知是真相源；通道投递是可重试副作用。

WebSocket 有两条不同链路：

- `RealtimeFanoutPublisher`：事务提交后的实时提示，使用短超时，例如本地默认 `100ms` 且生产可配置；RabbitMQ/WebSocket adapter 不可用时快速失败并进入熔断窗口，失败只记录日志和 metrics，不写 delivery ledger，前端可通过列表和未读数查询恢复。
- `WebSocketDeliveryProvider`：处理 `notification_delivery` 中的计划投递，成功标记 `WEBSOCKET_DISPATCHED`；失败标记 `FAILED` 或保持 `WEBSOCKET_PENDING` 并设置 `next_retry_at`，后续由 `RetryDelivery` 或 delivery worker 重试。

### 查询和缓存策略

PostgreSQL 是通知和 delivery 的真相源。Redis 只保存可重建缓存：

| 缓存 | Key 模式 |
| --- | --- |
| 用户未读总数 | `notification:{userId}:unread` |
| 用户未读 breakdown | `notification:{userId}:unread:breakdown` |
| 聚合通知列表 | `notification:{userId}:aggregated:{page}:{size}` |
| 用户偏好快照 | `notification:{userId}:preferences` |
| 用户 DND | `notification:{userId}:dnd` |
| 作者订阅 | `notification:{userId}:author:{authorId}:subscription` |

缓存失效要求：

| 操作 | 失效或更新 |
| --- | --- |
| 创建通知 | 原子增加未读缓存或删除未读缓存；删除 `notification:{userId}:aggregated:*` |
| 标记单条已读 | 原子递减未读缓存或删除未读缓存；删除 `notification:{userId}:aggregated:*` |
| 全部已读 | 删除未读总数、breakdown 和聚合列表缓存 |
| 更新偏好 | 删除 `notification:{userId}:preferences`；不改历史通知 |
| 更新 DND | 删除 `notification:{userId}:dnd`；不改历史通知 |
| 更新作者订阅 | 删除 `notification:{userId}:author:{authorId}:subscription`；不改历史通知 |
| delivery retry | 不改收件箱缓存；只更新 delivery 查询相关缓存，如未来引入 |
| group state rebuild | 删除用户聚合列表缓存 |

未读数必须有回源修正路径。缓存丢失时读 `notifications` 重建；发现 Redis 与 DB 不一致时以 DB 为准。

### 事件

#### 消费事件

Notification 消费的跨服务事件：

| 事件 | 来源 | 目标用例 | 关键字段 | 说明 |
| --- | --- | --- | --- | --- |
| `content.post.liked` | Content | `CreatePostLikedNotification` | `eventId`、`postId`、`authorId`、`likedBy` | 不给作者自己发通知 |
| `comment.created` | Comment | `CreateCommentOrReplyNotification` | `eventId`、`postId`、`commentId`、`floor`、`rootId`、`parentId`、`authorId`、`postAuthorId`、`parentAuthorId` | 评论通知文章作者；回复通知被回复者；缺字段时需回查 Comment/Content |
| `comment.liked` | Comment | `CreateCommentLikedNotification` | `eventId`、`commentId`、`postId`、`commentAuthorId`、`likedBy` | 可在切片 2 后实现 |
| `user.followed` | User | `CreateUserFollowedNotification` | `eventId`、`followerId`、`followingId` | 通知被关注者 |
| `content.post.published` | Content | `PlanPostPublishedCampaign` | `eventId`、`postId`、`authorId`、`title`、`excerpt`、`publishedAt` | 创建 campaign，不同步逐粉丝写库 |
| `comment.deleted` | Comment | `HideOrMarkSourceDeleted` | `eventId`、`commentId`、`postId`、`floor`、`isRoot`、`affectedCount` | 第一阶段可只隐藏跳转或保留通知，不删除收件箱事实 |
| `user.profile.updated` | User | `RefreshNotificationActorSnapshot` | `eventId`、`userId`、`profileVersion`、`nickname`、`avatarFileId` | 可选，用于通知列表快照刷新 |

RabbitMQ consumer 必须以 `eventId` 和业务 `dedupeKey` 双层幂等：

- `eventId` 写入 `consumed_events`，由 `event_id` 唯一约束防重复消费。
- `dedupeKey` 写入 `notifications` 或 `notification_delivery`，由业务唯一索引防不同事件重复表达同一通知事实。
- `consumed_events` 是 Notification 服务私有消费 ledger，不是跨服务事件源。保留期由配置 `consumer.consumed_events_retention` 控制，本地默认 `30d`；清理任务只删除 `expires_at < now()` 且已超过排查窗口的记录。清理任务由 `CleanupExpiredConsumedEvents` 驱动，按 batch size 删除，避免长事务和表锁放大。

#### 生产事件

Notification 生产的跨服务事件第一阶段默认只有实时 fanout，属于 best-effort：

| 事件 | routing key | 用途 | 可靠性 |
| --- | --- | --- | --- |
| 用户通知实时推送 | `notification.realtime.user_notification` | WebSocket/App 订阅用户通知 | best-effort |
| 未读数实时推送 | `notification.realtime.unread_count` | WebSocket/App 更新角标 | best-effort |
| 评论流提示 | `notification.realtime.comment_stream` | 文章详情页提示有新评论 | best-effort |

这些事件不改变业务真相源。丢失后前端可以通过查询通知列表或未读数恢复，因此第一阶段不强制 producer outbox。若未来其他服务依赖 Notification 事件作为权威事实，必须新增正式 `libs/contracts/events/notification` contract，并使用 Notification outbox。

### 跨服务依赖

- User：批量用户摘要、粉丝数量、粉丝分片、关注作者订阅上下文、用户状态。
- Content：发文事件字段缺失时补齐文章摘要；通知跳转仍以 Content 权威文章 ID 为准。
- Comment：评论通知需要回复上下文、楼层、父评论作者或跳转定位时补齐。
- Message：Message 未读数不归 Notification；如果产品要消息中心合并视图，应由 facade 聚合，不把私信未读写进 Notification。
- Admin：全局公告、系统通知或 delivery retry 管理可以由 Admin facade 委托 Notification。
  管理员 retry、公告写入等权限由 Admin facade 或认证上下文提供，Notification domain 不直接依赖 Admin 模型。

### 服务配置要点

Notification 的服务私有配置由 `runtime` 读取和校验，字段名最终以 `services/zhicore-notification/configs/` 模板为准。以下配置必须在实现前固定：

| 配置 | 作用 | 默认与约束 |
| --- | --- | --- |
| `public_id.active_version` | 当前生成 `NotificationPublicID` 使用的算法版本 | 必填；写入新 ID 时只用 active version |
| `public_id.secrets` | 按版本保存短公开 ID 可逆置换 secret | 生产必填且通过 Secret 注入；不允许代码默认值；轮换时新增版本并保留旧版本解码能力 |
| `consumer.consumed_events_retention` | `consumed_events` 保留期 | 本地默认 `30d`；生产可配置 |
| `jobs.cleanup_consumed_events.enabled` | 是否启动 `CleanupExpiredConsumedEvents` | 默认开启；测试可关闭 |
| `jobs.cleanup_consumed_events.interval` | 清理任务触发间隔 | 本地默认 `1h`；不能在每次启动时无界扫描 |
| `jobs.cleanup_consumed_events.batch_size` | 每轮删除最大行数 | 必须有上限，避免长事务 |
| `realtime_fanout.timeout` | best-effort 实时 fanout 超时 | 本地默认 `100ms`；生产可配置 |
| `realtime_fanout.circuit_breaker` | RabbitMQ/WebSocket adapter 持续失败时快速失败 | 第一阶段可用简单失败计数和冷却窗口 |
| `campaign.shard_claim_timeout_min` / `default` / `max` | shard claim lease 时间范围 | `CampaignShardPolicy` 计算值必须落在该范围内 |
| `campaign.batch_size` | 每批处理 follower 数 | 建议 500 到 2000，需压测校准 |
| `group_state.rebuild_lock_ttl` | 单用户 group state rebuild 并发保护 | 防止同一 `recipient_id` 被多个 rebuild 任务同时重建 |

这些配置遵循 `docs/architecture/configuration.md`：敏感项不写代码默认值，启动前校验，日志只输出脱敏摘要。后台任务启动、停止、panic 和 shutdown 语义遵循 `docs/architecture/runtime-operations.md`。

## API 保留范围

Notification 保留当前前端使用的路径和别名，但 Go 目标不要求兼容 Java 旧数据。路径中的 `{notificationId}` 使用 `notifications.public_id` 这类短公开 ID，不使用内部 DB identity；handler 解析 public ID 后再按 `recipient_id` 做权限限定。

- `/api/v1/notifications`：聚合通知列表。
- `/api/v1/notifications/{notificationId}/read`：标记单条已读。
- `/api/v1/notifications/read-all`、`/api/v1/notifications/mark-all-read`：全部已读。
- `/api/v1/notifications/unread/count`、`/api/v1/notifications/unread-count`：未读总数别名。
- `/api/v1/notifications/unread-breakdown`：未读分类统计。
- `/api/v1/notification-preferences`：通知偏好查询和更新。
- `/api/v1/notification-authors/{authorId}/subscription`：作者订阅查询和更新。
- `/api/v1/notification-dnd`：免打扰查询和更新。
- `/api/v1/notifications/deliveries`：delivery 查询。
- `/api/v1/notifications/deliveries/{deliveryId}/retry`：delivery retry；如果该接口对普通用户或管理员公开，`deliveryId` 也应使用 delivery 的短公开 ID，而不是内部 `notification_delivery.id`。

字段级 request/response、分页起点、错误码和权限需要后续按 `docs/contracts/http-schema-template.md` 提取到 `services/zhicore-notification/api/http`。

```sql
  producer VARCHAR(64) NOT NULL,
  dedupe_key VARCHAR(256) NULL,
  consumed_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  expires_at TIMESTAMPTZ NOT NULL,
  trace_id VARCHAR(128) NULL,
  request_id VARCHAR(128) NULL
);

CREATE INDEX idx_consumed_events_expires_at
  ON consumed_events(expires_at);
```

`notification_campaign`、`notification_campaign_shard` 和 `notification_delivery` 沿用 Java 表族方向，但内部 ID 使用数据库 identity；对外暴露的 delivery/campaign/shard ID 如进入 HTTP contract，也使用各自 `public_id`。`notification_delivery.dedupe_key` 必须唯一。`payload` / `latest_actor_ids` 等扩展字段优先使用 JSONB，避免把展示快照塞进不可查询的 TEXT。

## 推荐首个实现切片

**切片 1：站内通知收件箱 + 已读 + 未读数**

目标：先验证 Notification 的核心写入、聚合、已读和缓存边界。

- Domain：`Notification`、`NotificationGroupState`、`NotificationType`、`NotificationCategory`、`NotificationPublicID`、`DedupeKey`、`GroupKey`。
- Application：`CreateInteractionNotification`、`MarkNotificationRead`、`MarkAllNotificationsRead`、`ListAggregatedNotifications`、`GetUnreadCount`、`GetUnreadBreakdown`。
- Ports：`NotificationCommandRepository`、`NotificationQueryRepository`、`NotificationGroupStateRepository`、`UnreadCountCacheStore`、`AggregationCacheStore`、`TransactionRunner`、`Clock`、`NotificationPublicIDCodec`。
- Infrastructure：PostgreSQL repository 和 Redis cache store。
- HTTP 必须范围：
  - `GET /api/v1/notifications`
  - `POST /api/v1/notifications/{notificationId}/read`
  - `POST /api/v1/notifications/read-all`
  - `POST /api/v1/notifications/mark-all-read`
  - `GET /api/v1/notifications/unread/count`
  - `GET /api/v1/notifications/unread-count`
  - `GET /api/v1/notifications/unread-breakdown`
- 切片 1 不包含偏好、DND、作者订阅、delivery retry、公告和小助手 HTTP API。

**切片 2：事件消费 + 交互通知**

目标：接入 User/Content/Comment 事件，验证消费幂等、业务 dedupe 和实时 fanout。

- 消费 `content.post.liked`、`comment.created`、`user.followed`。
- 补 `ConsumedEventStore`、RabbitMQ consumer、DLQ/retry、`RealtimeFanoutPublisher`。
- 补“不通知自己”、重复事件 no-op、缺字段回查策略。

**切片 3：偏好、免打扰和 delivery ledger**

目标：让通知是否进入通道投递有明确规则。

- 补 `NotificationPreference`、`NotificationDnd`、`AuthorSubscription`。
- 补偏好 HTTP API、`ChannelPlanPolicy`、`NotificationDelivery` 和 WebSocket retry。
- HTTP 补 `/api/v1/notification-preferences`、`/api/v1/notification-dnd`、`/api/v1/notification-authors/{authorId}/subscription`、`/api/v1/notifications/deliveries`、`/api/v1/notifications/deliveries/{deliveryId}/retry`。
- Email/SMS provider 第一阶段返回 unconfigured，不影响站内通知。

**切片 4：作品发布 campaign fanout**

目标：验证高粉作者发文场景的异步广播链路。

- 消费 `content.post.published` 创建 campaign。
- 使用 User follower shard contract 拉取粉丝。
- 按偏好生成 `IN_APP`、`WEBSOCKET_PENDING`、`DIGEST_PENDING`、`SKIPPED` delivery。
- 补 shard claim、重试、summary、rebuild 和压测入口。

全局公告、小助手消息、摘要任务表和多通道 provider 不进入前三个最小切片，除非字段级 contract 证明前端首屏必须依赖。公告和小助手归入后续 Announcement / Assistant 切片；如果要做人群广播，再复用 campaign 作为投递计划，不改变公告/小助手自己的数据归属。

## 实现风险

- 高粉作者发布作品不能逐粉丝同步写库，必须使用 campaign + shard + bounded batch。
- 未读数和聚合状态即使同步维护，也必须提供从 `notifications` 回源重建的 use case，防止历史 bug、手工修复或缓存异常导致的计数问题。
- `eventId` 幂等不足以防业务重复，必须同时使用稳定 `dedupeKey`。
- WebSocket、Email、SMS 不能决定站内通知是否存在；通道失败只更新 delivery ledger。
- 偏好和免打扰不能每个 recipient 都回源多次，campaign shard 必须批量读取或缓存。
- `NotificationGroupState` 是读模型，不能作为唯一真相源。
- `comment.created` 事件如果不携带 `postAuthorId`、`parentAuthorId` 或评论跳转定位，Notification 需要回查 Content/Comment；这会影响事件 contract 和批量消费性能。
- Go 目标不规划 Java/Go 运行时共用数据库，也不要求兼容 Java 旧通知数据。Go 写入新的 `NotificationType` / `NotificationCategory` 枚举；如果未来要做一次性历史数据导入，必须单独编写迁移脚本把 Java 旧类型 `LIKE`、`COMMENT`、`FOLLOW`、`REPLY`、`SYSTEM` 等映射到新枚举，不能把兼容分支留在正常 application 路径里。
- Message 私信未读数与 Notification 未读数分开，不能为了消息中心展示把 Message 状态写入 Notification。

## DDD 设计总结

1. **Notification 的源事实是收件箱和投递台账**：触发通知的用户、文章、评论仍归源服务。
2. **交互通知和广播通知分链路**：点对点通知可以直接沉淀，发文通知必须 campaign + shard。
3. **未读数和聚合列表是可重建读模型**：Redis 和 `notification_group_state` 都不能替代 `notifications` 真相源。
4. **偏好和免打扰影响通道规划，不删除历史通知**：站内通知、WebSocket、Email、SMS 的一致性边界分开。
5. **delivery ledger 记录副作用结果**：provider 失败不回滚站内通知。
6. **事件消费需要双层幂等**：`eventId` 防重复消费，`dedupeKey` 防业务重复通知。
7. **分层依赖方向符合 Onion Architecture**：domain 不依赖 PostgreSQL、Redis、RabbitMQ、WebSocket、User client 或 Push provider。
