# Notification 页面设计

本文记录 Notification 相关页面的前端草稿、页面分区、接口编排、加载状态和降级规则。它是产品 / 前端交互设计事实源，不替代 HTTP 字段级 contract；字段、错误码和 path 后续以 `services/zhicore-notification/api/http/` 为准。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- Notification 拥有收件箱、聚合状态、未读数、偏好、免打扰、作者订阅和投递 ledger；页面不从 Content、Comment、User 直接拼装通知事实。
- 通知列表可以保存源服务展示快照，但点击通知后的目标页面仍由源服务重新加载并判断可见性。
- 未读数加载失败不能显示为 `0`；未知状态使用隐藏 badge 或中性占位。
- 标记已读和全部已读可以乐观更新，但失败必须回滚或触发重新拉取。
- WebSocket 实时提示是附加能力。WebSocket 不可用时，站内通知列表和未读数仍通过 HTTP 查询。
- 偏好和免打扰设置是用户安全感相关页面，保存状态必须明确，不可静默失败。

## 页面范围

本文覆盖：

- 通知中心列表。
- 聚合通知详情。
- 未读数和已读动作。
- 通知偏好设置。
- 免打扰设置。
- 作者订阅设置。
- 通道投递 / delivery 状态的管理入口初设计。

本文不覆盖：

- 私信会话详情，这些归 Message 页面设计。
- 文章、评论、用户目标详情，这些归各源服务页面设计。

## 通知中心

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Notification center                        │
├────────────────────────────────────────────┤
│ Tabs: all · interaction · content · system │
│ unread filter · mark all read              │
├────────────────────────────────────────────┤
│ Notification list                          │
│ icon · title · actor · target · unread dot │
│ grouped count · latest time                │
├────────────────────────────────────────────┤
│ Load more / empty / retry                  │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 页面进入时确认 Auth 登录态。
2. 并行调用聚合通知列表和未读数 endpoint。
3. 列表成功后渲染通知快照、聚合数量、未读状态和跳转目标。
4. 需要最新用户头像时批量调用 User 摘要；失败时使用通知 payload 快照或占位。
5. WebSocket 到达新通知事件时，更新未读数并在列表顶部提示“有新通知”，用户点击后刷新列表。
6. 列表分页使用 cursor 或 page；切换 tab 后丢弃旧响应。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 列表加载失败 | 页面主体显示错误和重试。 |
| 列表为空 | 展示当前过滤下暂无通知。 |
| 未读数失败 | 隐藏 badge 或显示 `--`，不显示 `0`。 |
| User 摘要失败 | 使用通知快照，列表仍展示。 |
| WebSocket 断开 | 显示轻量离线状态，HTTP 查询仍可用。 |

## 聚合通知详情

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Notification group                         │
├────────────────────────────────────────────┤
│ group summary · actors · target snapshot   │
├────────────────────────────────────────────┤
│ detail list                                │
│ actor · event · time · read state          │
├────────────────────────────────────────────┤
│ [Open target] [Mark read]                  │
└────────────────────────────────────────────┘
```

规则：

- 聚合列表点击后可以展开组内详情，也可以直接打开目标页；产品上应保持一致。
- 打开目标页时只携带目标路由参数，不把通知 payload 当目标详情事实。
- 目标文章 / 评论 / 用户不可用时，由目标服务页面显示不可用；Notification 不吞掉这个状态。
- 聚合详情加载失败只影响详情面板，不清空列表。

## 已读动作和未读数

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Read actions                               │
│ unread count · mark item · mark all        │
│ pending / failed sync                      │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 标记单条已读时，先本地移除 unread dot 并减少未读数。
2. 调用 mark read endpoint 成功后保持本地状态。
3. 失败时回滚该条 unread 状态，或重新拉取列表和未读数。
4. 全部已读需要二次确认或明确操作反馈；pending 时禁用重复点击。
5. 全部已读成功后清空当前过滤下 unread 状态，并重新拉取未读 breakdown。

未知未读数不能参与算术递减；如果当前 unread count 是 unknown，mark read 成功后应触发重新拉取，而不是本地计算。

## 通知偏好设置

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Notification settings                      │
├────────────────────────────────────────────┤
│ Type x channel matrix                      │
│ interaction · content · social · system    │
│ in-app · websocket · email · sms           │
├────────────────────────────────────────────┤
│ Save state / reset defaults                │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 进入设置页加载当前偏好矩阵。
2. `SMS` 第一阶段默认不可启用，UI 显示禁用态和原因。
3. 用户修改后显示未保存状态。
4. 保存成功后更新本地矩阵并清除未保存标记。
5. 保存失败时保留用户修改并显示可重试状态。

偏好设置只影响未来通知投递规划，不删除已存在站内通知。

## 免打扰和作者订阅

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Quiet hours / subscriptions                │
├────────────────────────────────────────────┤
│ DND toggle · time range · categories       │
├────────────────────────────────────────────┤
│ Author subscriptions                       │
│ author · all / digest / muted              │
├────────────────────────────────────────────┤
│ Save state                                 │
└────────────────────────────────────────────┘
```

规则：

- 免打扰跨日时间窗必须在 UI 上明确，例如 `22:00 - 08:00`。
- 免打扰不删除站内通知，只影响主动打扰通道。
- 作者订阅状态优先从 Notification 加载；作者头像 / 名称可由 User 摘要补齐。
- 作者摘要失败时订阅行仍可展示作者 ID 或旧快照。

## Delivery 管理入口

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Delivery admin                             │
├────────────────────────────────────────────┤
│ Filters: channel · status · recipient      │
├────────────────────────────────────────────┤
│ Table                                      │
│ delivery · channel · status · retry        │
├────────────────────────────────────────────┤
│ Detail · error code · retry state          │
└────────────────────────────────────────────┘
```

Delivery 管理入口属于管理或运维页面，普通用户只应看到自己的可见 delivery 状态。管理员 retry 必须走权限校验和审计；页面不能直接暴露 provider 原始错误、token、手机号、邮箱明文或内部堆栈。

## 跨服务页面约定

- Notification 点击目标后，各源服务按自己的主资源加载逻辑重新确认可见性。
- Content / Comment / User 资料变化不会同步改写历史通知事实；页面可用快照展示历史通知。
- Message 中的私信通知入口如果存在，只能跳到 Message 会话，不能由 Notification 修改 Message 未读状态。
- WebSocket 只负责实时提示；HTTP 列表和未读数仍是页面可恢复基础。
