# Search 页面设计

本文记录 Search 相关页面的前端草稿、页面分区、接口编排、加载状态和降级规则。它是产品 / 前端交互设计事实源，不替代 HTTP 字段级 contract；字段、错误码和 path 后续以 `services/zhicore-search/api/http/` 为准。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- Search 页面以搜索意图和结果扫描效率为核心，避免营销式首屏或过重装饰。
- Search 结果是 Content 派生读模型，不是可见性的最终判断。点击结果后仍由 Content 详情页重新校验文章可见性。
- 搜索建议、热门词和历史是附加资源；主搜索结果失败才显示页面级错误。
- 搜索结果中的作者、封面、摘要和高亮可以来自索引快照；需要权威详情时批量补 Content，补齐失败不能把结果直接改成不存在。
- 搜索历史属于当前用户维度；匿名用户不展示或不持久化服务端历史。

## 页面范围

本文覆盖：

- 全站文章搜索页。
- 搜索建议下拉。
- 热门搜索词。
- 搜索历史。
- 搜索结果过滤和分页。
- 搜索索引降级 / 空态。

本文不覆盖：

- 榜单热度页，这些归 Ranking 页面设计。
- 文章详情页，这些归 Content 页面设计。

## 搜索页

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Search top bar                             │
│ input · clear · submit                     │
├────────────────────────────────────────────┤
│ Suggest / history / hot keywords           │
├────────────────────────────────────────────┤
│ Filters: sort · time · tag · author        │
├────────────────────────────────────────────┤
│ Result list                                │
│ title · highlight · author · stats         │
│ ...                                        │
├────────────────────────────────────────────┤
│ Load more / empty / retry                  │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 页面进入且没有 query 时，加载热门词；登录用户同时加载搜索历史。
2. 用户输入时 debounce 调用建议 endpoint；输入为空时关闭建议或显示历史。
3. 用户提交搜索后调用 `GET /api/v1/search/posts`。
4. 搜索结果成功后立即渲染索引快照字段。
5. 如果结果项需要最新 Content 详情，收集 `postId` 批量补齐；补齐响应过期时丢弃。
6. 分页加载更多时保留已加载结果，在底部显示局部 pending。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 无 query | 展示热门词和历史，不展示空结果。 |
| 搜索结果为空 | 显示空态和可调整过滤条件提示。 |
| 搜索服务失败 | 结果区域页面级错误和重试。 |
| 建议失败 | 隐藏建议或显示轻量提示，不影响提交搜索。 |
| 热门词失败 | 隐藏热门词区域，不阻塞搜索框。 |
| Content 补齐失败 | 保留索引快照，标记详情可能不是最新。 |

## 搜索建议

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Suggest popover                            │
├────────────────────────────────────────────┤
│ keyword suggestion                         │
│ article title suggestion                   │
│ history item                               │
└────────────────────────────────────────────┘
```

规则：

- 建议面板宽度跟随搜索框，不能覆盖输入内容。
- 输入变化后旧建议响应必须丢弃。
- 建议接口失败时不弹全局错误。
- 键盘上下选择和回车提交应保持可访问。

## 热门词和搜索历史

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Explore search                             │
├────────────────────────────────────────────┤
│ Hot keywords                               │
│ keyword · keyword · keyword                │
├────────────────────────────────────────────┤
│ Recent history                             │
│ keyword · time · clear                     │
└────────────────────────────────────────────┘
```

加载逻辑：

1. 热门词匿名可见，进入无 query 搜索页时加载。
2. 搜索历史仅登录用户加载。
3. 清理历史是写动作，pending 时只锁定当前操作。
4. 清理失败时恢复历史项或重新拉取。

热门词失败不应显示“暂无热门”，因为这是未知状态；只有服务确认空列表时才显示空态。

## 结果项点击

点击结果项进入 Content 文章详情页：

- Search 不在点击时自行判断文章是否仍可见。
- Content 详情页如果返回不可用，展示 Content 的页面级不可用状态。
- Search 结果可带 `query` 和 highlight 上下文用于回退导航，但不能把 highlight 写入 Content 正文事实。

## 跨服务页面约定

- Content 可见性事件收敛目标是 Search 索引更新后小于等于 5 秒；页面文案不应承诺实时一致。
- Search 结果不触发 Content engagement 加载；只有进入文章列表 / 详情或专门结果卡需要互动状态时，才按 Content 的二阶段 engagement 规则加载。
- Ranking 热榜和 Search 热门词是不同产品语义，页面不能混用数据源。
