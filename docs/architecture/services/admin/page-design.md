# Admin 页面设计

本文记录 Admin 相关页面的前端草稿、页面分区、接口编排、加载状态和降级规则。它是产品 / 前端交互设计事实源，不替代 HTTP 字段级 contract；字段、错误码和 path 后续以 `services/zhicore-admin/api/http/` 为准。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- Admin 是管理端 facade 和审核编排服务，页面只展示管理流程和审计状态，不复制 User、Content、Comment 的业务规则。
- 管理动作必须先确认管理员身份和权限，危险操作要求原因、确认和可审计结果。
- 委托归属服务失败时，Admin 页面不能显示业务资源已经成功变更；可以显示失败审计或重试入口。
- 管理页面优先密度、筛选、表格、详情面板和可恢复操作，不做营销式 hero。
- 列表主资源和详情 / 审计 / 下游状态分层加载，避免一个附加字段失败导致整表不可用。

## 页面范围

本文覆盖：

- Admin 总览页。
- 用户管理。
- 内容管理。
- 评论管理。
- 举报处理。
- 审计日志。
- 委托操作状态和失败重试。

本文不覆盖：

- Ranking rebuild 具体状态机页面，这些归 Ranking 页面设计，Admin 可作为入口。
- Notification delivery 管理详情，这些归 Notification 页面设计，Admin 可作为入口。
- Ops 对账和修复任务，这些归 Ops 页面设计。

## Admin 总览页

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Admin shell                                │
│ nav: users · posts · comments · reports    │
├────────────────────────────────────────────┤
│ Review queue summary                       │
│ pending reports · failed actions · alerts  │
├────────────────────────────────────────────┤
│ Recent audit logs                          │
│ actor · action · target · result · time    │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 进入 Admin 页面先由 Gateway/Auth 确认管理员身份。
2. 并行加载举报待处理摘要、近期审计日志和关键失败操作。
3. 某个摘要失败时显示局部 degraded，不阻断其他面板。
4. 权限不足时显示 Admin 入口不可用，不进入子页面。

## 用户管理

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Users                                      │
├────────────────────────────────────────────┤
│ Filters: status · role · keyword           │
├────────────────────────────────────────────┤
│ Table                                      │
│ user · account status · profile · actions  │
├────────────────────────────────────────────┤
│ Detail panel · reason · audit result       │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. Admin 查询用户列表，返回账号状态、资料摘要和必要管理字段。
2. 需要详情时打开侧边详情面板，加载 User / Auth 聚合视图或 Admin facade 结果。
3. 禁用 / 启用账号动作由 Admin 委托 Auth；资料修正或资料逻辑删除由 Admin 委托 User。
4. 提交动作前必须填写原因。
5. 成功后更新当前行状态和审计日志摘要。

下游 Auth 或 User 不可用时，列表可以继续展示已有数据，但危险动作必须禁用或返回明确失败。

## 内容和评论管理

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Moderation                                 │
├────────────────────────────────────────────┤
│ Tabs: posts · comments                     │
│ filters: status · author · report state    │
├────────────────────────────────────────────┤
│ Table                                      │
│ target · author · reason · status · action │
├────────────────────────────────────────────┤
│ Preview panel · delete reason · audit      │
└────────────────────────────────────────────┘
```

加载规则：

- 文章管理删除委托 Content。
- 评论管理删除委托 Comment。
- 预览内容使用归属服务摘要或 Admin facade 快照。
- 归属服务返回资源不可用时，当前行标记为已不可用，并允许关闭举报。
- 删除失败时不改变目标状态，只记录失败结果。

## 举报处理

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Reports                                    │
├────────────────────────────────────────────┤
│ Filters: pending · processed · target type │
├────────────────────────────────────────────┤
│ Report list                                │
│ reporter · target · reason · created       │
├────────────────────────────────────────────┤
│ Decision panel                             │
│ evidence · action · note · [Submit]        │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 加载待处理举报列表。
2. 选中举报后加载目标摘要、历史处理记录和相关审计。
3. 管理员选择处理动作：忽略、删除目标、禁用用户、补充备注等。
4. 涉及业务资源变更时，Admin 委托对应服务。
5. 只有委托成功或明确选择忽略后，举报状态才转为已处理。

处理页面必须展示结果来源：本地举报状态、下游业务操作状态和审计日志写入状态。

## 审计日志

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Audit logs                                 │
├────────────────────────────────────────────┤
│ Filters: actor · action · target · result  │
├────────────────────────────────────────────┤
│ Table                                      │
│ time · actor · action · target · result    │
├────────────────────────────────────────────┤
│ Detail · reason · request id · trace id    │
└────────────────────────────────────────────┘
```

规则：

- 审计日志不可编辑。
- 详情中可展示 requestId / traceId 方便排障。
- 不展示敏感 token、密码、完整 Authorization header、完整 cookie 或内部堆栈。
- 审计查询失败时不影响其他 Admin 页面，但危险动作完成后必须能记录或报告审计失败。

## 跨服务页面约定

- Admin 只能作为 facade 调用归属服务，不直接让页面拼接多个业务写接口绕过编排。
- Admin 页面中的资源状态可能滞后；执行危险动作前后都以归属服务响应为准。
- Ops repair、Ranking rebuild、Notification delivery retry 可以从 Admin 进入，但事实和状态机仍归各自服务。
