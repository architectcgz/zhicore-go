# File 页面设计

本文记录文件上传、文件 URL 解析和媒体管理相关页面的前端草稿、接口编排、加载状态和降级规则。它是产品 / 前端交互设计事实源，不替代 HTTP 字段级 contract；字段、错误码和 path 仍以 `services/zhicore-file/api/http/` 为准。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- `zhicore-file` 是统一文件入口和文件生命周期 owner，不拥有头像、封面、评论媒体或文章附件的业务引用关系。
- 上传组件只返回 `fileId`、展示 URL、大小、类型和访问级别；业务页面保存引用时必须调用业务服务。
- 上传成功但业务保存失败时，页面需要明确“文件已上传但未绑定业务资源”的状态，不能让用户误以为资料或文章已保存。
- URL 是展示派生值，不是长期事实；长期引用使用 `fileId`。
- 文件校验错误应就地显示在文件行，不使用全局错误覆盖整个编辑器。
- 批量上传要明确部分失败语义：成功项可继续绑定，失败项保留错误和重试 / 移除操作。

## 页面范围

本文覆盖：

- 图片上传组件。
- 音频上传组件。
- 批量图片上传组件。
- 文件 URL 解析展示。
- 文件删除 / 移除引用的页面语义。
- 媒体选择器和轻量媒体管理入口初设计。

本文不覆盖：

- 业务资源保存，例如 User 保存头像、Content 保存封面、Comment 保存媒体引用。
- 对象存储或 MinIO 管理后台。

## 单文件图片上传

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Image upload                               │
├────────────────────────────────────────────┤
│ Drop zone / choose file                    │
│ preview · file name · size                 │
├────────────────────────────────────────────┤
│ progress · validation · retry / remove     │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 用户选择文件后先做本地 MIME、大小和数量校验。
2. 校验通过后调用 `POST /api/v1/files/image` 或 `POST /api/v1/files/image/with-access`。
3. 上传中显示文件行 progress，禁用重复选择同一文件的提交动作。
4. 上传成功后返回 `fileId` 和展示 URL，交给业务页面表单暂存。
5. 业务页面保存成功后，`fileId` 才成为业务资源引用事实。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 类型不允许 | 文件行错误，允许重新选择。 |
| 文件过大 | 文件行错误，提示限制。 |
| File service 失败 | 文件行错误，保留文件名和重试按钮。 |
| 上传成功但业务保存失败 | 显示业务保存失败，保留 `fileId`，允许重试保存或移除。 |
| URL 解析失败 | 显示缩略图失败占位，仍保留 `fileId`。 |

## 音频上传

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Audio upload                               │
├────────────────────────────────────────────┤
│ choose audio · duration / size             │
│ player preview                             │
├────────────────────────────────────────────┤
│ progress · validation · retry / remove     │
└────────────────────────────────────────────┘
```

规则：

- 音频上传成功后可以使用返回 URL 做本地播放预览。
- 播放 URL 失效或解析失败时，只影响预览，不删除 `fileId`。
- Comment 语音场景需要额外校验时长；该业务规则归 Comment 页面或 Comment 后端 guard。

## 批量图片上传

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Batch images                               │
├────────────────────────────────────────────┤
│ toolbar: access level · add files          │
├────────────────────────────────────────────┤
│ file grid                                  │
│ preview · status · retry · remove          │
│ preview · status · retry · remove          │
├────────────────────────────────────────────┤
│ success count · failed count · continue    │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 本地先校验总数量、单文件大小和类型。
2. 调用 `POST /api/v1/files/images/batch` 或按产品需要逐个调用单文件上传。
3. 当前 Go contract 下批量接口只返回成功项，单个失败项不会带失败详情；前端若需要逐项错误，应优先逐文件上传或等待 contract 扩展。
4. 成功项进入可绑定列表，失败项保留本地错误。
5. 用户确认后，业务页面把成功 `fileId[]` 提交给业务服务。

## 文件 URL 解析展示

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Media preview                              │
│ file id · loading url · preview / fallback │
└────────────────────────────────────────────┘
```

规则：

- 业务服务返回 `fileId` 但没有 URL 时，前端可调用 `GET /api/v1/files/{fileId}/url`。
- URL 解析失败时显示预览失败占位，不删除业务引用。
- `PRIVATE` 文件 URL 的签名和有效期由 `zhicore-file` 决定；前端不能把 URL 长期缓存为事实。

## 删除文件和移除引用

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Remove media                               │
├────────────────────────────────────────────┤
│ unlink from draft / delete file            │
│ warning · pending · result                 │
└────────────────────────────────────────────┘
```

页面语义必须区分：

- **移除引用**：从 User / Content / Comment 表单中移除 `fileId`，由业务服务保存后生效。
- **删除文件**：调用 `zhicore-file` 删除实际文件，可能影响其他引用，首期不应在普通业务编辑器中默认暴露危险操作。

删除文件失败不应回滚业务资源删除，除非业务服务明确把文件删除作为同一操作的强一致前置条件。

## 媒体管理入口初设计

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Media manager                              │
├────────────────────────────────────────────┤
│ Filters: type · access · owner context     │
├────────────────────────────────────────────┤
│ Media grid                                 │
│ preview · file id · size · used by         │
├────────────────────────────────────────────┤
│ url resolve · delete / unlink state        │
└────────────────────────────────────────────┘
```

媒体管理入口依赖 `zhicore-file` 文件元数据和引用状态查询能力。首期可以只使用上传组件和 URL 解析；后续补齐文件列表、引用状态、清理状态和删除审计后，再实现全量媒体管理页。

## 跨服务页面约定

- User 头像、Content 封面 / 正文媒体、Comment 图片 / 语音保存事实归对应业务服务。
- 上传组件不能直接更新业务页面的已保存状态；业务保存成功后才显示最终完成。
- Auth 未登录时，是否允许上传由具体业务页面决定；`zhicore-file` endpoint 当前 contract 不读取身份，但业务页面不应绕过业务权限保存引用。
