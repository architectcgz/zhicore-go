# Message 页面设计

本文记录 Message 相关页面的前端草稿、页面分区、接口编排、加载状态和降级规则。它是产品 / 前端交互设计事实源，不替代 HTTP 字段级 contract；字段、错误码和 path 后续以 `services/zhicore-message/api/http/` 为准。

当前状态：本文固定页面初设计和加载逻辑，不表示前端已经实现。

## 设计原则

- Message 页面是任务型会话界面，重点是会话切换、发送反馈、未读状态和失败可恢复，不做装饰性消息卡片。
- 会话摘要由 Message 拥有，用户资料摘要来自 User；User 摘要失败不能隐藏已有会话。
- 历史消息真相源可能是外部 IM provider。provider 未接入或降级时，页面可以展示空历史或降级提示，但不能伪造历史消息。
- 发送私信前必须确认 User 拉黑、私信设置、关注关系和当前用户状态；这些 guard 不可确认时 fail closed。
- 标记已读失败不阻塞用户阅读，但必须保留未读状态或显示局部同步失败。

## 页面范围

本文覆盖：

- 消息中心入口。
- 会话列表。
- 会话详情 / 聊天线程。
- 新建私信入口。
- 未读数和标记已读。
- 历史消息 provider 降级状态。

本文不覆盖：

- Notification 收件箱和系统通知，这些归 Notification 页面设计。
- 用户资料、关注和拉黑设置，这些归 User 页面设计。
- Auth 登录态恢复，这些归 Auth 页面设计。

## 消息中心入口

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Message center                             │
├────────────────────────────────────────────┤
│ Tabs: private messages · notifications     │
├────────────────────────────────────────────┤
│ Conversation list        │ Thread          │
│ user · snippet · unread  │ messages        │
│ ...                      │ composer        │
└────────────────────────────────────────────┘
```

消息中心可以在产品上聚合私信和通知，但数据归属必须清晰：私信 tab 由 Message 加载，通知 tab 由 Notification 加载。

## 会话列表

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Conversations                              │
├────────────────────────────────────────────┤
│ search · unread filter                     │
├────────────────────────────────────────────┤
│ row: avatar · name · snippet · unread      │
│ row: avatar · name · snippet · time        │
├────────────────────────────────────────────┤
│ empty / load more / retry                  │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 页面进入时确认 Auth 登录态；未登录跳转登录。
2. 调用会话列表 endpoint，加载会话摘要、未读数、最近消息摘要和参与者 ID。
3. 批量调用 User 摘要接口补头像和昵称；User 摘要失败时使用会话内快照或占位。
4. 会话列表成功后默认选中第一条未读会话或上次打开会话。
5. 切换过滤条件时取消或丢弃旧列表响应。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 会话列表失败 | 列表区域显示错误和重试，右侧线程清空。 |
| 会话为空 | 显示空态和可选新建私信入口。 |
| User 摘要失败 | 会话继续展示，头像 / 昵称降级。 |
| 未读数加载失败 | 不显示为 `0`，使用 `--` 或隐藏 badge。 |
| 登录态失效 | 跳转登录，清理当前消息页面状态。 |

## 会话详情 / 聊天线程

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Thread header                              │
│ user summary · status · actions            │
├────────────────────────────────────────────┤
│ Message history                            │
│ date divider · incoming · outgoing         │
│ provider degraded / empty history          │
├────────────────────────────────────────────┤
│ Composer                                   │
│ text input · send state · [Send]           │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 选中会话后先展示会话摘要。
2. 调用历史消息 endpoint 或外部 IM provider adapter 暴露的查询接口。
3. provider 未接入时展示空历史兼容态，保留发送入口是否可用由 guard 决定。
4. 历史加载成功后按时间稳定排序，分页向上加载更早消息。
5. 线程进入可见状态后调用标记会话已读；失败时保留未读 badge 或显示同步失败。

### 状态处理

| 场景 | 页面行为 |
| --- | --- |
| 历史 provider 未接入 | 显示空历史或“历史暂不可用”，不伪造消息。 |
| 历史加载失败 | 线程局部错误，composer 按 guard 决定是否可用。 |
| mark read 失败 | 保留未读状态，显示轻量同步失败。 |
| 对方不可用 | 禁用 composer，显示用户不可私信。 |
| 被拉黑 / 已拉黑 | 禁用发送，显示关系状态。 |

## 新建私信和发送

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ New message                                │
├────────────────────────────────────────────┤
│ recipient search / selected user           │
├────────────────────────────────────────────┤
│ Composer                                   │
│ text input                                 │
├────────────────────────────────────────────┤
│ guard state · [Send]                       │
└────────────────────────────────────────────┘
```

### 加载逻辑

1. 从用户主页点击私信时携带目标 `publicId`，先加载 User 摘要和 Message guard。
2. 从消息中心新建时先搜索或选择目标用户，再加载 guard。
3. guard 成功且允许发送时启用 composer。
4. 点击发送后调用 Message 发送接口。
5. 发送成功后创建或更新会话摘要，并把消息加入当前线程的 pending -> sent 状态。
6. 发送失败时消息保留在本地失败态，可重试或删除。

写路径不能只依赖前端已有关系状态；发送前或发送时仍由后端校验 User guard。

## 未读数

### 页面草稿

```text
┌────────────────────────────────────────────┐
│ Unread summary                             │
│ all unread · conversation badges           │
└────────────────────────────────────────────┘
```

规则：

- 顶部导航的消息未读数来自 Message 未读摘要，不从会话列表行数推导。
- 未读数加载失败时隐藏 badge 或显示中性占位，不能显示为 `0`。
- 标记已读成功后本地乐观减少未读数；失败时回滚或触发重新拉取。
- Notification 未读数由 Notification 页面设计负责，聚合展示时应标明来源。

## 跨服务页面约定

- User 拉黑、关注和陌生人私信设置变化后，Message 页面进入线程或发送前必须重新确认 guard。
- Notification 的消息中心聚合只能读取 Message 摘要，不能修改 Message 的会话未读事实。
- 外部 IM provider 降级时，Message 页面必须明确历史不可用或发送不可用的边界，避免把 provider 空响应误读成真实无历史。
