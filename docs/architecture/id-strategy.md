# ID 策略

本文件记录 `zhicore-go` 的内部主键、外部公开 ID 和业务编号策略。

## 结论

- 内部主键默认使用各服务数据库自己的 `BIGINT` sequence / identity。
- 不把 `zhicore-id-generator` 作为默认核心依赖，不为每个普通业务实体引入 Snowflake 或 segment 发号。
- 对外暴露的资源标识和内部主键解耦，使用独立字段，例如 `public_id`、`public_no`、`order_no`。
- 普通公开内容可以参考 Bilibili `BV` 的思路：短、稳定、带版本的外部 ID，但不要把它当作强加密。
- 敏感资源不直接编码内部主键，应使用独立业务编号或随机公开 ID。

## 内部主键

各服务拥有自己的数据库和表时，内部实体 ID 默认使用本服务数据库生成：

```sql
id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY
```

内部 ID 的定位：

- 只作为服务内部关联、事务、索引和分表依据。
- 可以被其他服务保存为 opaque reference，例如 `author_id`、`post_id`、`file_id`。
- 其他服务不得依赖该 ID 的生成方式、连续性或可读含义。
- 不要求全局可读，也不承担对外展示职责。

当前没有旧数据需要兼容，因此优先采用简单、可运维、易迁移的数据库 sequence。只有未来出现跨数据库离线发号、多主写入、强依赖全局趋势 ID 或数据库 sequence 成为明确瓶颈时，才重新评估 Snowflake / segment / 独立发号集群。

## 外部公开 ID

对外 URL、API 返回值、前端路由和用户可见编号不直接使用内部 `id`。每类资源按风险选择独立外部字段：

| 场景 | 字段建议 | 生成方式 | 说明 |
| --- | --- | --- | --- |
| 文章、帖子、话题 | `public_id` | `prefix + version + Base58/Base62(permute64(id, secret))` | 类 Bilibili 风格短 ID；短、可逆、弱暴露顺序 |
| 评论楼层 | `floor` / `post_number` | topic/post 内局部递增 | 只在所属主题内唯一，不作为全局实体 ID |
| 订单、支付、风控资源 | `order_no` / `public_no` | 业务前缀、日期、渠道、序列、随机段、校验位组合 | 不直接编码内部主键 |
| 文件、事件、幂等键 | `public_id` / `event_id` / `idempotency_key` | UUIDv7 或足够长度的随机 ID | 优先保证全局唯一和低耦合 |
| 临时授权 token | token 字段 | AEAD 加密后 Base64URL 编码 | 用于授权，不作为资源编号 |

`Base36`、`Base58`、`Base62` 都只是编码，不是加密。它们负责把数字变短、变得适合 URL；是否安全取决于编码前的生成或混淆策略。

## 类 Bilibili 短 ID

普通公开内容可以采用类似 `BV` 的外部 ID 形态：

```text
内部 id -> 64-bit 可逆置换 -> Base58/Base62 编码 -> prefix + version + code
```

示例：

```text
id: 2447119
public_id: p1K8x9Q2
URL: /t/some-title/p1K8x9Q2/26
```

约定：

- `prefix` 表示资源类型，例如 `p` 表示 post/topic。
- `version` 表示算法版本，例如 `1`。
- `code` 使用 Base58 或 Base62 编码后的短字符串。
- 可逆置换必须带服务级密钥或盐，避免简单 `Base62(id)` 直接暴露连续数量。
- 数据库仍保存 `public_id` 并建立唯一索引，外部查询优先 `WHERE public_id = ?`。

这种方案解决的是“短、稳定、不直接暴露连续数字”，不是强安全加密。不要用于订单、支付、私信、安全操作等敏感资源。

## 加密和可逆性

如果业务确实需要把内部 ID 加密成可解密 token，应使用 AEAD：

- AES-GCM
- XChaCha20-Poly1305
- AES-GCM-SIV

流程是：

```text
id -> AEAD 加密 -> Base64URL 编码
```

这种 token 具备保密和防篡改能力，但长度通常会明显变长，不适合作为短 URL 资源 ID 的默认方案。

当前 `zhicore` 的默认选择是：资源外部 ID 独立存储，内部极少需要解密时直接通过 `public_id` 查询映射，不依赖可解密 token。

## 表结构建议

普通公开资源：

```sql
CREATE TABLE posts (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  public_id VARCHAR(32) NOT NULL UNIQUE,
  ...
);
```

订单类资源：

```sql
CREATE TABLE orders (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  order_no VARCHAR(64) NOT NULL UNIQUE,
  ...
);
```

要求：

- `public_id`、`order_no` 这类外部字段必须有唯一索引。
- 内部服务间关联优先使用内部 `BIGINT id`；对外 API 返回是否包含内部 ID 按兼容规则和资源风险单独决定。
- 对外字段一旦发布，就按 contract 管理，不随内部主键策略重构而破坏。

## `zhicore-id-generator` 定位

`zhicore-id-generator` 不作为当前默认发号基础设施。

保留该服务目录的原因：

- 保留旧服务映射中的集中发号能力入口，避免 Go 设计评估时丢失模块。
- 作为未来 Snowflake、segment 或独立发号集群需求的可选落点。
- 作为现有 Java 行为的参考，不要求早期服务依赖它。

在重新启用集中发号前，必须先明确：

- 哪些业务场景不能接受数据库 sequence。
- 是否真的需要跨数据库、跨实例、离线或多主写入发号。
- 发号服务的高可用、时钟回拨、号段耗尽、监控和灾备策略。
- 对外编号是否会暴露业务数量，以及是否需要独立公开编号。
