# 文档与专题规则归属

- 全局 prompt 只保留始终生效、漏了会反复出事故的硬规则。
- 领域细则默认放进对应 skill，而不是继续扩大全局 prompt。
- 代码、配置、接口、部署、架构边界或用户可见行为变化时，必须在实现前按项目文档规范判断文档 owner、事实源和需要更新的路径；实现完成时只核对计划中的文档是否已随改动同步。
- 当用户要求”记录错误””记住这个问题””别再犯”时，必须先判断该问题是否属于现有 agent policy 或 skill 分类；能归入现有分类时优先补到对应分类，只有明显不适配时才新建分类或只记 incident log。
- 构建前端 UI 时，界面上只允许出现真实面向终端用户的产品文案。
- 实现说明、结构说明、设计意图、布局介绍、占位提示、字段注释、待办标记等内容，一律不得渲染到可见 UI 中。
- 如需解释设计或实现原因，只能写在 assistant 回复或代码注释里，不能写进页面。
- 除非用户明确指定”这句话要显示在页面上”，否则默认所有说明性文字都不是 UI 文案。

## 文档范围说明格式（架构文档、专题文档正文）

适用：架构文档、模块设计文档、专题文档等需要明确边界的技术文档正文。

推荐在文档”定位”或”本文档范围”章节使用双栏表格格式，左列列出”本文档负责”的主题，右列列出”本文档不负责（见其他文档）”的相关主题并指向对应文档。

**标准格式：**

```markdown
## 本文档范围

| 本文档负责 | 本文档不负责（见其他文档） |
|-----------|-------------------------|
| Contest 状态机与转换规则 | 事件发布策略 → `事件发布与降级策略.md` |
| 乐观锁 + 分布式锁并发控制 | 容器调度细节 → `容器调度与并发控制.md` |
| 事务边界与回滚策略 | 数据库事务模式 → `数据库事务管理模式.md` |
```

**格式要求：**

- 左列：本文档实际覆盖的主题和职责，使用陈述句或名词短语
- 右列：相关但不在本文档范围内的主题，使用 `主题名 → 文档路径` 格式
- 右列为空时填 `-`，表示该主题无其他文档补充
- 箭头格式统一为 `→`，后接相对路径或文件名（如 `../backend/modules/contest.md` 或 `事件发布与降级策略.md`）
- 避免使用”负责 xxx，不负责 xxx”的散文式表述；表格格式可读性更高，定位更快

**优点：**

- 读者可以快速判断”这份文档是否覆盖我要找的内容”
- 明确边界，避免文档职责重叠或遗漏
- 便于文档间交叉引用，帮助读者找到相关文档
- 表格格式比散文式”负责/不负责”列表更清晰、更易维护

**项目实施：**

- 各项目可在自己的 `docs/文档规范.md` 中引用本格式作为架构文档标准
- 新增架构文档时默认使用此格式
- 存量架构文档可在修订时逐步迁移到此格式

**完成检查约束：**

修改文档后，必须运行项目配置的文档一致性检查脚本验证格式合规性。

- CTF 项目：`python3 scripts/check-docs-consistency.py`
- 其他项目：查看项目 `docs/文档规范.md` 或 `AGENTS.md` 中的”文档验证”章节

**禁止行为：**

- 不得在声称”完成”前跳过自动化检查
- 不得用”应该通过””理论上没问题”替代实际执行证据
- 不得在检查失败时直接报告完成，必须先修复再重新检查
- 不得使用 `**本设计负责**：` + bullet list 的旧格式替代表格格式
- 不得在”组件边界”或”本文档范围”章节使用散文段落替代表格

**格式检查要点：**

当文档包含”组件边界”或”本文档范围”章节时：
1. 必须使用双栏表格，不能用 bullet list 或散文段落
2. 表头必须为 `| 本文档负责 | 本文档不负责（见其他文档） |` 或 `| 本设计负责 | 本设计不负责（见其他文档） |`
3. 每行左列陈述职责，右列要么指向其他文档（`主题 → 路径`），要么填 `-`
4. 修改完成后必须立即运行项目文档检查脚本

## 结构性改动流程

- 对跨多个文件的 service / repository / handler / 页面职责拆分，已有 `docs/plan/impl-plan/*` 方案文档的改动，或会影响模块边界、调用路径、依赖装配、测试组织、review 成本的重排类改动，默认视为结构性改动。
- 结构性改动不能直接进入实现；必须先确认或补齐方案文档，明确当前阶段，按方案实现最小可审查改动，运行最小充分验证，再切换到独立 review 心智检查 correctness、回归风险、测试缺口和结构债。
- 对结构性改动或其他 `非琐碎任务`，独立 review 不是可选增强，而是默认完成条件；如果需要 reviewer subagent 来满足这道 gate，默认直接使用，不再把“是否允许派 reviewer”当成单独确认项。
- 如果实现触达 audit、review、architecture、improvement backlog 中已经记录的结构性技术债，本次必须在 touched surface 内收口该债务；不能把同一位置的已知结构债写成 `residual risk`、`follow-up` 或“后续再拆”后直接交付。
- 如果发现收口范围会超出当前任务，应在编码前回到方案阶段重新拆任务；最终 review 若发现 touched surface 上的已知结构债仍然存在，应视为 blocker。
